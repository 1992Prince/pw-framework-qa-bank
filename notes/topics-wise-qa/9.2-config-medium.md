- However, running `npx playwright test` always picks up `playwright.config.js` by default.

npx playwright test --config new_config_file_name.js

**11. How do you configure retries for failed tests in Playwright?**

- Set the `retries` property at the root level of the config:
```js
export default defineConfig({
  retries: 2,
});
```
- This means a failed test will be re-run up to 2 more times before being marked as a final failure.
- Common pattern: `retries: process.env.CI ? 2 : 0` â€” retry on CI (flaky environments, network variance) but fail fast locally so you notice real issues immediately.
- Retries can also be overridden per-project if some browsers/suites are flakier than others.
- You can also override it from the CLI without touching the config:
```
npx playwright test --retries=3
```

**12. How do you configure screenshot, video, and trace capture in the config file?**

```js
use: {
  screenshot: 'only-on-failure',
  video: 'retain-on-failure',
  trace: 'on-first-retry',

```
- Options include: `on`, `off`, `on-first-retry`, `retain-on-failure`

**13. How do you configure multiple `projects` in Playwright for cross-browser / cross-device testing? Give the syntax.**

- Use the `projects` array at the root level of the config. Each object represents an independent test configuration (browser, device, or environment) that the whole suite will run against.
```js
export default defineConfig({
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox', use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit', use: { ...devices['Desktop Safari'] } },
  ],
});
```
- When you run `npx playwright test`, your entire test suite runs once per project â€” so 3 projects means each test runs 3 times, once per browser.

**14. How would you set up projects to run tests in Chrome, Chromium, Edge, WebKit, Pixel 5, and iPhone 12, along with a custom viewport?**

- Example for Chrome/Edge (both use the Chromium engine, differentiated by `channel`):
```js
{
  name: 'chromium',
  use: {
    browserName: 'chromium',
    headless: false,
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
    channel: 'chrome', // use 'chrome' for Google Chrome
  },

{
  name: 'edge',
  use: {
    browserName: 'chromium',
    headless: false,
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',

  },
},
```

```js
{ name: 'Microsoft Edge', use: { ...devices['Desktop Edge'], channel: 'msedge' } },

```
- To run a specific project: `npx playwright test --project "Mobile Safari"`
- To run multiple projects: `npx playwright test --project "Mobile Safari" "edge"`
- Running plain `npx playwright test` runs all tests in every enabled project â€” e.g., 10 tests Ă— 4 projects (chromium, edge, Mobile Chrome, Mobile Safari) = 40 test runs.

**15. How do you run tests serially or in parallel in Playwright? Explain `workers` and `fullyParallel`.**

- Two root-level properties control this: `workers` and `fullyParallel`.
- `workers: process.env.CI ? 1 : undefined`
  - `undefined` locally means Playwright picks half of the available CPU cores.
  - E.g., an 8-core machine â†’ 4 workers.
- With `workers: undefined` and no `fullyParallel`:
  - If you have 10 spec files, each with multiple tests, Playwright distributes spec files across workers (e.g., 4 spec files running in parallel across 4 workers).
  - Tests within a single spec file still run sequentially (serially); once one spec file finishes, another is picked up.
- With `fullyParallel: true`:
  - Tests are no longer confined to running in file order â€” individual tests from any spec file can be picked up and run in parallel across workers.
  - This gives true test-level parallelism instead of just file-level parallelism.
- In CI, it's common to keep `workers: 1` for stability/sequential execution; with strong infra, more workers can be used for speed.

**16. How do you enable multiple reporters in a Playwright framework? What reporters does Playwright support, and what's the syntax?**

- Set the `reporter` property at the root level, passing an array of `[name, options]` pairs to combine reporters:
```js
export default defineConfig({
  reporter: [
    ['list'],
    ['html', { open: 'never' }],
    ['json', { outputFile: 'results.json' }],
    ['junit', { outputFile: 'results.xml' }],
  ],
});
```
- Built-in reporters Playwright supports: `list` (console output), `dot` (compact CI-friendly output), `html` (interactive report), `json`, `junit` (for CI tools like Jenkins), and `line`.
- You can also write a **custom reporter** and reference it by file path instead of a built-in name.
- Multiple reporters run together in the same test run â€” e.g., `list` for live console feedback while `html` and `junit` generate files for later review/CI integration.

**17. How do you set up/configure the test directory (`testDir`) for a suite in the config file?**

- Set the `testDir` property at the root level to point to the folder containing your spec files:
```js
export default defineConfig({
  testDir: './tests',
});
```
- Playwright will recursively pick up test files inside that folder (by default, files matching `*.spec.ts` / `*.spec.js`).
- `testDir` can also be overridden **per-project** â€” useful when different suites (e.g., regression vs. smoke) live in different folders:
```js
projects: [
  { name: 'regression', testDir: './tests/regression' },
  { name: 'smoke', testDir: './tests/smoke' },
],
```
- You can further narrow which files run using `testMatch` / `testIgnore` patterns alongside `testDir`.

**18. What is the purpose of `globalSetup` and `globalTeardown` in Playwright?**

- `globalSetup` is a script that runs **once, before the entire test suite starts** â€” regardless of how many workers, projects, or spec files exist.
- `globalTeardown` is a script that runs **once, after the entire suite finishes**.
- Common uses:
  - `globalSetup`: logging in and saving a `storageState` session, seeding test data, starting a mock server, setting environment variables.
  - `globalTeardown`: cleaning up test data, tearing down a server, closing external connections.
- Configured at the root level:
```js
export default defineConfig({
  globalSetup: require.resolve('./global-setup.ts'),
  globalTeardown: require.resolve('./global-teardown.ts'),
});
```
- Key distinction from a per-project `setup` project: `globalSetup`/`globalTeardown` run **once for the whole run**, not once per project/worker â€” so they're ideal for expensive one-time operations rather than per-browser auth flows.

**19. How do you set global suite timeout, test timeout, action timeout, and navigation timeout at the suite level?**

- All four are configured in `playwright.config.ts`, but at different levels:
```js
export default defineConfig({
  globalTimeout: 60 * 60 * 1000,   // max time for the ENTIRE test run (1 hour)
  timeout: 30 * 1000,               // max time for EACH individual test (30s)
  expect: {
    timeout: 5000,                  // max time for expect() assertions
  },
  use: {
    actionTimeout: 10 * 1000,       // max time for actions like click(), fill()
    navigationTimeout: 15 * 1000,   // max time for page.goto(), waitForNavigation()
  },
});
```
- `globalTimeout` â€” caps the whole run; if exceeded, Playwright stops and marks remaining tests as skipped/interrupted.
- `timeout` â€” per-test ceiling; if a single test runs longer, it's marked as timed out.
- `actionTimeout` / `navigationTimeout` â€” finer-grained, catch a single slow action or slow page load without necessarily failing the whole test if the rest completes reasonably fast.
- All of these can also be overridden at the project level or even per-test using `test.setTimeout()`.

**20. How do you manage environment-specific configurations, like different URLs and credentials, across environments?**

- Most common approach: use environment variables (`.env` files with a package like `dotenv`) and reference them in the config:
```js
export default defineConfig({
  use: {
    baseURL: process.env.BASE_URL || 'https://qa.example.com',
  },
});
```
- Then switch environments via CLI:
```
BASE_URL=https://staging.example.com npx playwright test
```
- Alternative approach: define separate **projects** per environment (e.g., `qa`, `staging`, `prod`), each with its own `baseURL` and credentials, and run the relevant one with `--project`.
- Credentials (usernames/passwords/API keys) are typically kept out of the repo entirely â€” pulled from `.env` files (gitignored) or CI secret managers â€” and referenced via `process.env` in config or fixtures.

**21. Where do you set test data / test IDs for an application, and when would you typically set this up?**

- `testIdAttribute` is configured inside the `use` block â€” it tells Playwright which HTML attribute to treat as the test ID selector (default is `data-testid`):
```js
use: {
  testIdAttribute: 'data-test', // if your app uses data-test instead of data-testid
}
```
- Once set, you can locate elements with `page.getByTestId('submit-button')` regardless of which attribute name your app actually uses.
- You'd typically set this up **once, at framework initialization**, based on whatever convention the application's developers already use in the DOM (some teams use `data-testid`, others `data-qa`, `data-test`, etc.) â€” it's a coordination point between dev and QA early in the project.
- Actual test **data** (like test users, sample payloads) is usually managed separately â€” either in fixture files, JSON/CSV data files, or generated via API calls in `globalSetup` â€” not inside `playwright.config.ts` itself.
