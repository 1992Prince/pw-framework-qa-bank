#### **25. Suppose you have 10 spec files (each with multiple tests) running on an 8-core machine with `workers: undefined` — walk through how Playwright would execute them, and how enabling `fullyParallel: true` changes that behavior.**

*(Same underlying concept as Q15 — repeated here as a scenario walkthrough.)*

- With an 8-core machine and `workers: undefined`, Playwright picks 4 workers (half the cores).
- Without `fullyParallel`: the 10 spec files get distributed across those 4 workers — so 4 files run at a time, one per worker. Tests **within** each file still run one after another, in sequence. As soon as a worker finishes its file, it picks up the next one from the queue, until all 10 are done.
- With `fullyParallel: true`: Playwright no longer treats each file as one sequential unit per worker. Instead, individual tests — from any of the 10 files — get spread across the 4 workers and run in parallel, regardless of which file they came from.
- Net effect: `fullyParallel: true` gives finer-grained parallelism and generally faster runs, but tests must be fully independent of each other (no shared state or execution-order assumptions) for this to be safe.

#### **26. How would you implement retry logic for flaky tests without masking genuine, real failures?**

- Set a modest `retries` value (e.g., 1–2) rather than a high number — enough to smooth over genuine flakiness (network blips, timing issues) without hiding a real bug that fails consistently.
- Use Playwright's built-in retry reporting: a test that passes only after a retry is still flagged as **"flaky"** in the report (not a clean pass), so it stays visible even though the run technically succeeded.
- Pair retries with tracing/screenshots/video **on-first-retry** — this way, if a test does need a retry, you capture debug evidence to investigate *why*, instead of just silently re-running and moving on.
- Avoid using retries as a substitute for fixing bad waits/selectors — retries should catch genuine environmental flakiness, not compensate for a test that's fundamentally racy (e.g., missing a proper `await` or wait condition).
- Track flaky-test trends over time (via CI reports) so a test that "needs a retry" every single run gets flagged for a real fix rather than being retried indefinitely.

#### **27. How do you enable `storageState` (session reuse) in a framework using project dependencies? Walk through the setup.**

- A dedicated **auth setup** test file performs the login and saves the resulting session:

```js
// auth.setup.ts
import { test as setup } from '@playwright/test';

setup('authenticate', async ({ page }) => {
  await page.goto('/login');
  await page.fill('#username', 'testuser');
  await page.fill('#password', 'password');
  await page.click('#login-button');
  await page.context().storageState({ path: 'playwright/.auth/user.json' });
});
```

- One project is configured to point to this setup file, and also has a corresponding teardown project:

```js
projects: [
  { name: 'setup', testMatch: /auth\.setup\.ts/ },
  { name: 'teardown', testMatch: /auth\.teardown\.ts/ },
  {
    name: 'chromium',
    use: { ...devices['Desktop Chrome'], storageState: 'playwright/.auth/user.json' },
    dependencies: ['setup'],
    teardown: 'teardown',
  },
],
```

- Other projects declare a dependency on the setup project via `dependencies: ['setup']`.
- Result: login happens only once via the saved storage session, and all subsequent tests across all projects reuse that session instead of logging in again for every test.

#### **28. What information gets stored in a `storageState` (session) file?**

- **Cookies** — including session cookies, auth tokens, and any cookie-based flags the app relies on.
- **Local storage** entries, per origin — key/value pairs the app has written to `localStorage`.
- **Origins** — the file is structured per-origin, so it can hold state for multiple domains if the app spans more than one (e.g., an auth subdomain plus the main app).
- It does **not** store session storage (`sessionStorage`) by default, since that's meant to be tab/session-scoped and not persisted.
- Example shape:

```json
{
  "cookies": [ { "name": "session_id", "value": "...", "domain": "...", ... } ],
  "origins": [
    {
      "origin": "https://example.com",
      "localStorage": [ { "name": "authToken", "value": "..." } ]
    }
  ]
}
```

#### **30. We have lots of properties at root level like `retries`, `workers`, etc., with values like `process.env.CI ? 2 : 0`, and `baseURL` in the `use` block — but how do you actually pass these CI values from a GitHub Actions workflow?**

Let's break this into small pieces.

Step 1: How does `process.env.CI` get read?

`process.env` is just Node.js reading environment variables from the machine it's running on. `CI` is one specific variable name.

```js
retries: process.env.CI ? 2 : 0,
```

This just means: "if an env var named `CI` exists (and is truthy), use 2 retries, else use 0."

Step 2: Does GitHub Actions set `CI` automatically?

Yes. **Every GitHub Actions runner automatically sets `CI=true`** as a default environment variable — you don't need to do anything for this specific one. So the moment your test runs inside a GitHub Actions job (instead of your local machine), `process.env.CI` becomes `"true"`, and `retries` automatically becomes `2`. Locally on your laptop, `CI` is undefined, so it falls back to `0`.

That's it for `CI` specifically — it's free, built into every GitHub Actions runner.

Step 3: How do you set your OWN custom env vars (like `BASE_URL`) in the workflow?

You define them under `env:` in the YAML file, at the workflow, job, or step level:

```yaml
jobs:
  test:
    runs-on: ubuntu-latest
    env:
      BASE_URL: https://staging.example.com   # <-- your custom variable
    steps:
      - run: npx playwright test
```

When the `npx playwright test` step runs, GitHub Actions injects `BASE_URL` into the process environment for that step — so inside Node.js, `process.env.BASE_URL` will equal `"https://staging.example.com"`.

Step 4: How does that reach `baseURL` in the config file?

You just read it in your config the same way, with a local fallback for when you're not in CI:

```js
// playwright.config.ts
export default defineConfig({
  use: {
    baseURL: process.env.BASE_URL || 'https://qa.example.com',
  },
});
```

- In CI → `process.env.BASE_URL` is `"https://staging.example.com"` (set by the workflow) → that's used.
- Locally → `process.env.BASE_URL` is undefined → falls back to `'https://qa.example.com'`.

Putting it together — full flow

```
GitHub Actions workflow (YAML)
   env: BASE_URL: https://staging.example.com
        ↓ (injected into the runner's process environment)
playwright.config.ts
   baseURL: process.env.BASE_URL || 'fallback'
        ↓ (config is loaded when `npx playwright test` runs)
Your tests
   page.goto('/login')  →  resolves to https://staging.example.com/login
```

So there are really just two mechanisms:
1. **`CI`** — already set for you by GitHub Actions, no config needed on your end.
2. **Anything custom (`BASE_URL`, `API_KEY`, etc.)** — you explicitly declare it under `env:` in the workflow YAML, and your config reads it via `process.env.YOUR_VAR_NAME`.

#### **31. How do you pass `browserName`, `channel`, `baseURL`, `retries`, `workers` for a specific project from GitHub Actions?**

- Two common approaches:

**Approach 1 — CLI flags in the workflow step**, overriding config values without touching the file:

```yaml
- run: npx playwright test --project="chromium" --workers=4 --retries=2
  env:
    BASE_URL: https://staging.example.com
```

- Playwright's CLI flags (`--project`, `--workers`, `--retries`) take precedence over whatever is set in `playwright.config.ts`, so you can run different projects/settings per job without maintaining separate config files.

**Approach 2 — env vars read inside the config**, so `browserName`, `channel`, etc. are dynamic per project:

```js
// playwright.config.ts
export default defineConfig({
  workers: process.env.WORKERS ? Number(process.env.WORKERS) : undefined,
  retries: process.env.CI ? 2 : 0,
  projects: [
    {
      name: process.env.BROWSER || 'chromium',
      use: {
        browserName: (process.env.BROWSER as any) || 'chromium',
        channel: process.env.CHANNEL || undefined,
        baseURL: process.env.BASE_URL,
      },
    },
  ],
});
```

```yaml
# workflow — run the same pipeline against different browsers via a matrix
jobs:
  test:
    strategy:
      matrix:
        browser: [chromium, firefox, webkit]
    env:
      BROWSER: ${{ matrix.browser }}
      BASE_URL: https://staging.example.com
    steps:
      - run: npx playwright test
```

- The **matrix strategy** is the key piece here — it lets GitHub Actions spin up parallel jobs, each with a different `BROWSER`/`CHANNEL`/`BASE_URL` value, all pointing at the same config file and same test suite.
- In practice, teams often combine both: CLI flags for simple overrides (`--project`, `--workers`), and env vars + matrix for anything that needs to vary per job (browser, environment URL, channel).
