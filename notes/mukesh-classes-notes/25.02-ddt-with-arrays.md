# Data-Driven Testing in Playwright

## What is it?

Data-driven testing is an approach where the same test script runs multiple times with different sets of input data.  
Instead of writing separate tests for each data combination, you parameterize the test and feed it data from an external or inline source.

---

## When to use it?

- Validating a login form with invalid credentials, blank values, special characters, lengthy inputs, etc., and verifying the expected error message for each.
- Creating multiple records and verifying each.
- Any scenario where the test logic is the same but data changes.

---

## Data-Driven via Array — Login Validations & Error Validations

Below is an array of objects. Each object can be called a **row**, and within the object we keep **key-value pairs** — the key is the data name and the value is its value. e.g., `username: 'admin'`

> In Excel, one row is one record and column name/value maps to key/value.  
> In DDT, each row is one record and each record has key-value parameters.

```js
let loginTestData = [
  { username: 'admin1',  password: 'admin1', error: 'invalid credentials' },
  { username: 'adm@',   password: 'admi',   error: 'invalid values in username field' },
  { username: '',        password: 'admin1', error: "username field can't be blank" }
]
```

---

## Data Sources

Data can come from **arrays** (inline), **CSV files**, **JSON files**, **Excel sheets**, or **databases**.

---

## How it Works

- Load the data first, then iterate and trigger the test for each record.
- Each record (row) is equivalent to one test execution.
- If there are 10 records, the test runs 10 times.
- Each iteration is **independent** — if record 2 fails, the others still run.
- In the report, each iteration shows as a **separate test** with its own pass/fail status.

---

## Example — Using Arrays

```js
import { test, expect } from "@playwright/test";

// array of objects
const testData = [
  { id: "TC001", name: "Sarika", email: "sarika@example.com", password: "password123" },
  { id: "TC002", name: "John",   email: "john@example.com",   password: "password456" },
  { id: "TC003", name: "Alice",  email: "alice@example.com",  password: "password789" },
];

// iterate array using for...of loop
// test name must be unique — use a string template with id to ensure that

for (const data of testData) {
  test(`Test ${data.id} - ${data.name}`, async ({ page }) => {
    await page.goto("https://example.com/practice");
    await page.getByTestId("form-email").fill(data.email);
    await page.getByTestId("form-password").fill(data.password);
    await page.getByTestId("form-confirm-password").fill(data.name);

    if(data.id === 'TC001'){
      // this assertion will fail for TC001 as the expected value is "123" but the actual value is "TC001"
      await expect(data.id).toBe("123"); 
    }

  });
}
```

---

## Conditional Assertions per Record

You can add assertions conditionally using `if` / `else if` / `else` blocks based on the test data values.  
This lets you validate different expected outcomes for different records within the same loop.

```js
for (const data of testData) {
  test(`Test ${data.id} - ${data.name}`, async ({ page }) => {
    await page.goto("https://example.com/practice");
    await page.getByTestId("form-email").fill(data.email);
    await page.getByTestId("form-password").fill(data.password);
    await page.getByTestId("form-confirm-password").fill(data.name);

    if (data.id === 'TC001') {
      // assertion specific to TC001
      await expect(page.getByTestId("success-msg")).toHaveText("Welcome, Sarika!");
    } else if (data.id === 'TC002') {
      // assertion specific to TC002
      await expect(page.getByTestId("success-msg")).toHaveText("Welcome, John!");
    } else {
      // assertion for all other records
      await expect(page.getByTestId("success-msg")).toBeVisible();
    }
  });
}
```

---

## Assertion Failure Does Not Stop Other Iterations

Each record in the array runs as its **own independent test**.  
If an assertion fails for TC001, Playwright marks that test as **failed** and immediately moves on to TC002 — it does **not** abort the remaining iterations.

- TC001 fails → marked ❌ in report → TC002 still runs
- TC002 passes → marked ✅ → TC003 still runs
- Each test has its own isolated pass/fail status

> This is the core power of DDT — one bad data record does not block the entire suite.  
> All records get exercised and the report clearly shows which ones passed and which ones failed.

---

## Why Must Test Names Be Unique?

Playwright requires each test to have a unique title.  
If two tests share the same name, Playwright throws an error because it can't distinguish them in reporting and execution.

---

## Interview Note

Arrays are fine for demos, but in real projects data is typically stored in **JSON** or **CSV files** to keep test code and data separate *(separation of concerns)*.
