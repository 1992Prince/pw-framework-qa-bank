# Playwright Prerequisites — Node.js, npm & npx

---

## 1. Prerequisite — Node.js (Must Install First)

Playwright is built on top of Node.js — it's the engine that runs your test scripts.  
Without Node.js, nothing works — no npm, no npx, no Playwright.  
So install Node.js **first** before anything else.

### Version

- Node.js **v18 or above** is mandatory — below v18 Playwright will throw errors or refuse to install.
- Always download the **LTS version** from https://nodejs.org.
- The installer gives you both Node.js + npm together automatically.

### Verify after install

```bash
node -v    # should show v18.x.x or above
npm -v     # shows npm version (bundled with Node)
```

If Node.js is not installed, `npm` and `npx` commands won't be recognized ("command not found"), you can't install any library, and your test scripts won't execute at all.

> **Node.js is the very first thing you install. Everything else depends on it.**

---

## 2. What is Node.js?

Node.js is an open-source, cross-platform JavaScript runtime environment that lets you run JavaScript code **outside the browser** — directly on your computer/server.

Normally JavaScript only runs inside a browser (Chrome, Firefox, etc.).  
Node.js took Chrome's **V8 JavaScript engine**, wrapped it in a standalone program, and made it possible to run JavaScript anywhere — on your terminal, on a server, on your machine — without needing a browser at all.

### Why does Playwright need Node.js?

- Playwright tests are written in JavaScript/TypeScript.
- These test files are just `.js` or `.ts` files sitting on your computer — they're not running inside a browser page.
- Node.js is the engine that reads and executes those test scripts on your machine.
- Without it, your computer doesn't know how to run a `.js` file.
- Also, `npm` and `npx` (which you use to install and run Playwright) are Node.js tools — they don't exist without Node.js.

### What is npm?

- **npm = Node Package Manager** — it's a tool that installs, manages, and updates libraries in your project.
- Think of it like the Play Store for Node.js projects — it fetches packages from the npm registry (a huge online store).
- npm comes **bundled with Node.js** — no separate install needed.

```bash
node -v    # Node.js version
npm -v     # npm version
```

---

## 3. What is npx?

- **npx = Node Package Executor** — it runs/executes commands from packages.
- It comes bundled with Node.js (from v8.2+).
- When you run an `npx` command, it first checks if the package is installed locally in `node_modules` — if yes, it runs from there. If not, it temporarily downloads it, executes, and discards it.

```bash
npx playwright install   # looks for playwright in node_modules, finds it, runs the "install" command
npx playwright test      # locates playwright in node_modules and runs your tests
npx -v                   # check npx version (bundled with npm, no separate install)
```

---

## 4. npm vs npx — Clear Difference

In simple terms:
- **npm** downloads and saves libraries into your project.
- **npx** runs commands from those libraries.

> npm is the **installer**, npx is the **executor**. You need both — npm brings the library in, npx uses it.

---

### Q — How `npm init playwright@latest` Works (Step by Step)

When you run:

```bash
npm init playwright@latest
```

1. npm goes to the npm registry (online store at `registry.npmjs.org`).
2. Searches for the package `@playwright/test` and picks the latest version.
3. Downloads it and saves it inside your project's `node_modules/` folder.
4. Adds it to `devDependencies` in `package.json`.
5. Runs the Playwright setup wizard (asks TS/JS, folder name, browsers, CI).
6. npm's job is done — it installed the library. It doesn't run any tests.

---

### Q — How `npx playwright test` Works (Step by Step)

When you run:

```bash
npx playwright test
```

1. npx looks inside `node_modules/.bin/` for a `playwright` executable.
2. Finds it (because npm already installed it earlier).
3. Passes the `test` argument to it.
4. Playwright test runner picks up your config, finds spec files, launches browsers, and executes your tests.
5. npx didn't install anything — it just ran what was already there.

---

### Q — Can We Interchange npm and npx?

**No.** They do completely different jobs:

- `npm playwright test` → ❌ won't work — npm only installs, it doesn't know how to "run" tests.
- `npx install -D @playwright/test` → ❌ won't work — npx only runs commands, it doesn't install packages as dependencies.

They are **partners, not substitutes**.

---

### Q — Why do we run Playwright commands with npx and not npm?

Because Playwright provides CLI commands (`playwright test`, `playwright install`, `playwright codegen`) that need to be **executed**, not installed again.  
They already sit inside `node_modules` — npx just finds and runs them from there.  
Any command or executable present in `node_modules/.bin/` can be run via `npx`.

---

## 5. Step-by-Step: Create a Fresh Playwright Project

### Step 1 — Create Project Folder and Initialize

```bash
mkdir my-playwright-project
cd my-playwright-project
npm init
```

### What is `npm init` and Why Do We Need It?

`npm init` creates a `package.json` file in your folder — this file is the **identity card** of your project.  
Without it, your folder is just a random folder — npm won't recognize it as a project, you can't install libraries, and you can't share it properly with teammates.  
Running `npm init` converts a regular folder into a proper Node.js project.

It asks a few questions interactively — project name, version, description, entry point, test command, author, license, etc.  
You can press Enter to accept defaults for all. Or use `npm init -y` to skip all questions and create `package.json` instantly with defaults.

### Q — What Does `package.json` Look Like After `npm init -y`?

At this point (before installing anything):

```json
{
  "name": "my-playwright-project",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "echo \"Error: no test specified\" && exit 1"
  },
  "keywords": [],
  "author": "",
  "license": "ISC"
}
```

> Notice there are no `dependencies` or `devDependencies` yet — those sections appear only after you install packages.

### Q — When Do We Use `npm init`?

Anytime you're starting a new Node.js-based project — whether it's Playwright, React, Express, or any other framework.  
Every project needs a `package.json`, and `npm init` is how you create it.

---

### Step 2 — Install Playwright

```bash
npm init playwright@latest
```

**What this does internally:**

1. Goes to the npm registry, fetches the latest version of `@playwright/test`, and installs it.
2. Runs an interactive setup wizard asking:
   - TypeScript or JavaScript?
   - Test folder name?
   - Add GitHub Actions workflow?
   - Install browsers?
3. If you say Yes to browsers, it automatically runs `npx playwright install` which downloads all 3 patched browser binaries to a global cache:
   - **Windows:** `C:\Users\<user>\AppData\Local\ms-playwright\`
   - **Mac:** `~/Library/Caches/ms-playwright/`
   - **Linux:** `~/.cache/ms-playwright/`

> `@latest` = download the newest available version from the npm registry.

**Want a Specific Version Instead?**

```bash
npm init playwright@1.40.0                  # full setup with specific version
npm install -D @playwright/test@1.40.0      # library only with specific version
```

## 6. Folder Structure Generated

After setup, your project looks like this:

```
project-root/
├── node_modules/              ← All installed libraries live here
├── tests/
│   └── example.spec.ts        ← Sample test file
├── tests-examples/
│   └── demo-todo-app.spec.ts  ← Demo tests for learning
├── package.json               ← Project's ID card (name, version, dependencies)
├── package-lock.json          ← Locks exact versions of all packages
└── playwright.config.ts       ← Main Playwright config (browsers, timeouts, retries, etc.)
```

If you opted for GitHub Actions:

```
.github/
  └── workflows/
      └── playwright.yml       ← CI pipeline — runs tests on every push/PR
```

**Key points about each:**

| File/Folder | Notes |
|---|---|
| `node_modules/` | Where Playwright and all dependencies physically live. Never edit manually. Can be regenerated with `npm install`. Never push to Git. |
| `tests/` | Your test files go here. Comes with a sample spec file. |
| `package.json` | Project name, version, scripts, dependencies. You can edit this. |
| `package-lock.json` | Locks exact versions so every teammate gets the same setup. Never edit manually. |
| `playwright.config.ts` | Configure browsers, base URL, timeouts, retries, reporters, parallel execution, etc. |

## Add Playwright to an Existing Project

Suppose you already have an existing project and just want to add the `@playwright/test` library:

```bash
npm install -D @playwright/test
```

**Use when:**

- You already have a Node.js project.
- Playwright is missing / not working.
- You want to update Playwright.
- You only want the Playwright test library, not the full setup wizard.

### What is `-D`?

- `-D` is short for `--save-dev`.
- It installs the package as a **devDependency** (needed only for development/testing, not in production).
- It gets added under the `devDependencies` section in `package.json`.

> **Why Playwright uses `-D`?** Because Playwright is a testing tool — you don't need it in the final production build of the app.

---

## 7. `package.json` — What It Contains

After installing Playwright, `package.json` looks like this:

```json
{
  "name": "my-playwright-project",
  "version": "1.0.0",
  "description": "",
  "main": "index.js",
  "scripts": {
    "test": "playwright test"
  },
  "keywords": [],
  "author": "",
  "license": "ISC",
  "devDependencies": {
    "@playwright/test": "^1.48.0",
    "@types/node": "^22.7.5"
  }
}
```

**Important components:**

- `name` — your project name.
- `version` — follows semantic versioning (`major.minor.patch`).
- `scripts` — shortcuts you run via `npm run <name>` (e.g., `npm run test`).
- `dependencies` — libraries needed in production.
- `devDependencies` — libraries needed only during development/testing — Playwright goes here because it's a testing tool, not part of the final app.

### Q — Where to see Playwright version?

Either check `devDependencies` in `package.json`, or run `npx playwright --version`.

---

## 8. `package.json` vs `package-lock.json`

### `package.json` — What It Is and How It Works

`package.json` is the **ID card** of your project.  
It holds your project name, version, scripts, and most importantly — the list of all libraries (dependencies) your project needs.

---

### How Do Libraries Get Added to `package.json`?

There are two ways:

#### Way 1 — Via `npm install` command *(most common)*

Every time you install a library using npm, it automatically gets added to `package.json` with its version.

```bash
npm install -D @playwright/test          # Playwright testing library
npm install -D @faker-js/faker           # Faker library for generating test data
npm install -D allure-playwright         # Allure reporting library
npm install -D dotenv                    # Environment variables library
```

After running these, your `package.json` automatically updates:

```json
"devDependencies": {
  "@playwright/test": "^1.48.0",
  "@faker-js/faker": "^9.2.0",
  "allure-playwright": "^3.0.4",
  "dotenv": "^16.4.5"
}
```

> You didn't type any of this manually — npm added each library here the moment you installed it.

#### Way 2 — Edit manually

You can also directly open `package.json`, type a library name and version under `devDependencies`, save the file, and run `npm install` — npm will read it and install whatever is listed.  
This is useful when you know exactly which version you want.

---

### How to Update a Library Version?

**Option 1 — Via command:**

```bash
npm install -D @playwright/test@1.52.0    # installs specific new version
npm install -D @playwright/test@latest    # installs latest available version
```

**Option 2 — Edit `package.json` manually:**

Change the version number in `package.json` from `"^1.48.0"` to `"^1.52.0"`, then run `npm install`.  
npm reads the updated version and downloads it.

---

### How Does a New Teammate Set Up the Same Project?

This is where `package.json` shines. When a new team member joins:

```bash
git clone <repo-url>
npm install
```

`npm install` reads `package.json`, sees all the listed libraries (Playwright, Faker, Allure, dotenv, etc.), and installs all of them automatically on their machine with the specified versions.  
They don't need to install each library one by one — `package.json` acts as the **master list**.

---

### `dependencies` vs `devDependencies` — What's the Difference?

You've seen the `-D` flag (or `--save-dev`) in install commands. Here's what it does:

```bash
npm install -D @playwright/test      # goes under devDependencies
npm install express                  # goes under dependencies (no -D flag)
```

```json
{
  "dependencies": {
    "express": "^4.18.2"
  },
  "devDependencies": {
    "@playwright/test": "^1.48.0"
  }
}
```

- **`dependencies`** (without `-D`) — libraries needed to run the app **in production**. When your app is deployed and serving real users, these libraries must be present. Example: Express (web server), React (UI framework), database drivers.

- **`devDependencies`** (with `-D`) — libraries needed only during **development and testing**. When the app goes to production, these are not needed. Example: Playwright (testing tool), Faker (test data generator), Allure (test reports), ESLint (code linter).

**When do we use `-D`?** Ask yourself — *"Does the end user or production server need this library to run the app?"*

- Yes → `npm install <library>` (goes to `dependencies`)
- No, only developers/testers need it → `npm install -D <library>` (goes to `devDependencies`)

Playwright is a testing tool — real users never interact with it, the production server doesn't need it. So it always goes under `devDependencies` with `-D`.

> For pure automation/testing projects (like a Playwright framework), technically everything is dev-related so everything goes under `devDependencies`. The distinction matters more in full-stack projects where the same codebase has both application code and test code.

---

### `package-lock.json` — Why We Need It

Now you know `package.json` lists libraries with flexible version ranges like `"^1.48.0"`.  
That `^` means *"1.48.0 or any newer compatible version."*  
So when someone runs `npm install`, npm picks the newest version that fits the range at that moment.

**The problem:** You run `npm install` today and get `1.48.3`. Your teammate runs it next month and gets `1.48.9`. Your CI server runs it next week and gets `1.48.6`. Everyone has different versions even though `package.json` is the same. Different versions can have different bugs, different behaviors — and suddenly tests pass on your machine but fail on theirs.

#### What `package-lock.json` Does

`package-lock.json` locks down the **exact version** of every single package that was installed — not just the main libraries, but also all their internal sub-dependencies.

```
package.json says:        "@playwright/test": "^1.48.0"     ← flexible range
package-lock.json says:   "@playwright/test": "1.48.3"      ← exact version, locked
```

When `package-lock.json` exists and someone runs `npm install`, npm ignores the flexible range in `package.json` and installs the exact locked version from `package-lock.json`.  
This guarantees every developer, every CI server, every environment gets the **same versions**.

#### Rules for `package-lock.json`

- **Never edit it manually** — npm manages it automatically.
- **Always push it to Git** — never add it to `.gitignore`.
- It gets auto-updated whenever you install, update, or remove a package.

## Summary

- Node.js **v18+** must be installed first — Playwright is built on Node.js. Without it, npm, npx, and Playwright won't work at all.
- Installing Node.js automatically gives you **npm** (package manager) and **npx** (package executor). Verify with `node -v`, `npm -v`, `npx -v`.
- **npm installs libraries, npx runs/executes commands** from installed libraries — they cannot be interchanged.
- `npm init` creates `package.json` — without this file, npm doesn't recognize your folder as a project. Use `npm init -y` to skip questions.
- `npm init playwright@latest` = full fresh project setup with wizard (folders, config, sample tests, browsers).
- `npm install -D @playwright/test` = library-only install for existing projects (no wizard, no folders, no config).
- `-D` is short for `--save-dev` — adding `-D` installs the library under `devDependencies` (needed only for development/testing), not under `dependencies` (needed in production).
- For pure testing/automation projects like Playwright frameworks, **all libraries go under `devDependencies`** with the `-D` flag.
- Libraries get added to `package.json` automatically when you install via npm command. You can also edit `package.json` manually and run `npm install`.
- To update a library, either run `npm install -D <library>@<new-version>` or change the version in `package.json` and run `npm install`.
- When a new teammate clones the repo and runs `npm install`, **all libraries listed in `package.json` get installed automatically** on their machine.
- `package.json` stores flexible version ranges (e.g., `^1.48.0`). `package-lock.json` locks exact versions (e.g., `1.48.3`) so everyone gets the same setup.
- `package.json` and `package-lock.json` are pushed to GitHub, but `node_modules/` is **never pushed** — add it to `.gitignore`. It can be regenerated anytime with `npm install`.
- Browser binaries are **not stored in `node_modules/`** — they live in a global cache (`ms-playwright/` folder). Teammates must run `npx playwright install` separately after cloning.
- `npx playwright install` = download browsers. `npx playwright install --with-deps` = download browsers + OS-level system libraries (required on Linux CI servers).
- `@latest` in commands means fetch the newest version from npm registry. To install a specific version, use `@<version>` (e.g., `@playwright/test@1.52.0`).
- Playwright version can be checked in `package.json` under `devDependencies` or by running `npx playwright --version`.
