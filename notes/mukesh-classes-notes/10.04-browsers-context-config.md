## 6. PLAYWRIGHT KE PATCHED BROWSERS — INTERNAL DETAIL ⭐

Yeh ek bahut important concept hai jo mostly miss ho jaata hai.

### System Browser vs Playwright Browser

- When you run `npx playwright install` — it does not download your system's Chrome or Firefox application.
- It downloads Playwright's own specially patched browser binaries.
- These patched binaries are stored at: C:\Users\<YourUsername>\AppData\Local\ms-playwright

### System Browser vs Playwright Browser |

- `System Browser` — installed at the OS level, gets auto-updated, and has limited CDP support since it is a full end-user application not meant for automation.
- `Playwright Browser` — stored in node_modules/.cache/ms-playwright/ (local to project), its version is binded with the Playwright version and does not get auto-updated. CDP support is strong because Playwright takes the open-source browser projects (Chromium, Firefox, WebKit) and injects its own CDP instrumentation into them — giving it full control over the browser for automation.
- Chrome and Firefox as applications are OS-installed programs — Playwright never touches or installs those. - - What Playwright installs are browser engines, not applications.

### What is Patching in Playwright?

- Patching means Playwright takes the open-source source code of each browser engine, injects its own automation hooks and CDP instrumentation into it, and builds its own custom browser binary from that modified source.
- This gives Playwright deep, reliable control over the browser that a normal browser install cannot provide.

### How is it different from Selenium?

- Selenium drives the actual installed browser on your system (Chrome, Firefox) via browser drivers like chromedriver — it does not patch or modify anything.
- Playwright ships its own patched browser binaries and does not rely on whatever browser version is installed on the machine, making it more stable and version-consistent.

### Chromium Patching

- Playwright uses open-source Chromium — not Google Chrome.
- Chromium is the public base project maintained by Google. Google Chrome is Chromium plus proprietary additions like codecs, crash reporting, and Google services.
- Playwright injects its own CDP code into Chromium's source to add extra automation capabilities that normal Chromium does not have — which is why Playwright's Chromium binary is slightly larger in size than standard Chromium

### Firefox Patching

- Mozilla's Firefox does not natively support CDP — it was never built for automation via CDP.
- Playwright has modified Firefox's source code to make it support a CDP-like protocol, maintaining its own patched Firefox fork to enable this.

### WebKit Patching

- WebKit is Safari's rendering engine — and Safari itself is only available on macOS and iOS.
- Playwright has added automation hooks into WebKit's source, which allows its patched WebKit binary to run on Windows and Linux as well — something the real Safari cannot do.

### Why Patched Browsers are necessary

- **Version control** — every Playwright version ships with an exactly matching browser version — no version mismatch surprises.
- **Enhanced CDP** — extra automation APIs and hooks are available that normal browser installations do not provide.
- **Cross-platform WebKit** — you can test Safari behavior on Windows and Linux as well — not just macOS.
- **Predictable behavior** — CI and local machine both run the exact same browser version — eliminates the classic "works on my machine" problem.

## 7. `playwright.config.ts` — DEEP DIVE

***playwright.config.ts*** is the **single source of truth** for the entire Playwright framework — all **global settings** like `testDir`, `timeout`, `retries`, `workers`, `parallelism`, `reporter`, and **browser-level settings** like `baseURL`, `actionTimeout`, `navigationTimeout`, `headless`, `screenshot`, `video`, and `trace` are configured here in one place — so every test in the framework picks up these settings automatically without repeating them in individual test files.

Ek jagah change karo — poore test suite pe apply hota hai.

```typescript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  // ── Test-level settings ──────────────────────────────────────
  testDir: './tests',           // test files kahan dhundhe
  timeout: 30000,               // ek test ka max time — 30 seconds default
  retries: 2,                   // fail pe retry — 0 local, 2 CI
  workers: 4,                   // parallel processes kitne
  fullyParallel: true,          // file ke andar bhi parallel
  forbidOnly: !!process.env.CI, // CI mein test.only() block
  reporter: 'html',             // report format

  // ── Per-action settings ──────────────────────────────────────
  use: {
    baseURL: 'http://localhost:3000',     // relative goto('/login') possible
    actionTimeout: 15000,                 // har action ka max wait — 15s
    navigationTimeout: 30000,            // page.goto() ka max wait
    headless: true,                       // browser visible nahi hoga
    screenshot: 'only-on-failure',        // fail pe screenshot
    video: 'retain-on-failure',           // fail pe video
    trace: 'on-first-retry',              // retry pe trace — CI standard
  },

  // ── Multi-browser setup ──────────────────────────────────────
  projects: [
    { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
    { name: 'firefox',  use: { ...devices['Desktop Firefox'] } },
    { name: 'webkit',   use: { ...devices['Desktop Safari'] } },
    { name: 'mobile-chrome', use: { ...devices['Pixel 5'] } },
    { name: 'mobile-safari', use: { ...devices['iPhone 13'] } },
  ],
});
```
### Global Level Mnemonic

**"T-T-R-W-F-F-R-E-G"**

TestDir — Timeout — Retries — Workers — FullyParallel — ForbidOnly — Reporter — Expect — GlobalTimeout

Sentence: "Two Tired Runners Walk Fast, Forgetting Roads Every Go"

### Use Block Mnemonic

**"B-A-N-H-S-V-T"**

BaseURL — ActionTimeout — NavigationTimeout — Headless — Screenshot — Video — Trace

Sentence: "Big Angry Ninja Hits Stones Via Throws"

### Har Setting Ka Purpose

**Test-level settings:**

| Option            | Purpose                                                    | Default           |
| ----------------- | ---------------------------------------------------------- | ----------------- |
| `testDir`       | Test files ki location                                     | `./tests`       |
| `timeout`       | Poore test ka max time                                     | `30000ms`       |
| `retries`       | Fail pe auto-retry                                         | `0`             |
| `workers`       | Parallel test processes                                    | CPU cores ka half |
| `fullyParallel` | File ke andar bhi parallel                                 | `false`         |
| `forbidOnly`    | CI mein `test.only()` block                              | `false`         |
| `reporter`      | `'html'`, `'list'`, `'dot'`, `'json'`, `'junit'` | `'list'`        |

**`use` block settings:**

| Option                | Purpose                                                            | Recommended             |
| --------------------- | ------------------------------------------------------------------ | ----------------------- |
| `baseURL`           | Har test mein full URL nahi likhna                                 | App ka URL              |
| `actionTimeout`     | Har `.click()`, `.fill()` ka max wait                          | `15000ms`             |
| `navigationTimeout` | `page.goto()` ka max wait                                        | `30000ms`             |
| `headless`          | Browser UI dikhega ya nahi                                         | `true` CI mein        |
| `screenshot`        | `'on'`, `'off'`, `'only-on-failure'`                         | `'only-on-failure'`   |
| `video`             | `'on'`, `'off'`, `'retain-on-failure'`                       | `'retain-on-failure'` |
| `trace`             | `'on'`, `'off'`, `'on-first-retry'`, `'retain-on-failure'` | `'on-first-retry'`    |

### ⭐ `timeout` vs `actionTimeout` vs `navigationTimeout` — Yeh Confusion Hoti Hai

```
timeout          = poore test ka total budget
                   (sab actions + assertions + navigations combined)

actionTimeout    = ek individual action ka budget
                   (.click(), .fill(), .textContent(), expect() — har ek ke liye)

navigationTimeout = sirf page load ka budget
                   (page.goto(), page.waitForURL() ke liye)
```

**Example:**

```
timeout: 60000        → test ko 60 seconds mein complete karna hai
actionTimeout: 15000  → har ek click/fill 15 seconds mein hona chahiye
navigationTimeout: 30000 → har page load 30 seconds mein hona chahiye
```

## 12. BROWSER → BROWSERCONTEXT → PAGE HIERARCHY ⭐


- **Browser Binaries vs Browser Applications** — Playwright's browser binaries are not the same as browser applications installed on your OS. They are Playwright's own patched engines downloaded via `npx playwright install`.

- **Browser** — when a test runs, Playwright invokes one browser binary instance at the **worker level** (default scope is `worker`). This scope can be changed in custom fixtures.

- **BrowserContext** — inside a single browser instance, a separate **BrowserContext** is created for each test. It is like an **incognito profile** — completely isolated with no shared cookies, sessions, storage, or network state between tests. One browser instance can have multiple browser contexts running simultaneously.

- **Page** — inside a BrowserContext, one or more **Pages** can be created. Each page represents a single **browser tab**.

- **Hierarchy:**

```
Browser (1 process)
├── BrowserContext A (isolated session — incognito jaisa)
│   ├── Page 1 (tab 1)
│   └── Page 2 (tab 2)
└── BrowserContext B (isolated session)
    └── Page 3 (tab 3)
```
- **Real Browser** — if you connect Playwright to a real browser application instead of its own binaries, the browser binary is replaced with the real browser instance but the remaining hierarchy of BrowserContext and Page stays exactly the same.

- **Fixtures** — Playwright provides `browser`, `context`, `page`, `browserName`, and `request` as built-in fixtures directly in the test function — no manual setup needed.

### Browser

The `browser` object represents a single running browser process — Chromium, Firefox, or WebKit. In Playwright Test it is automatically injected via fixtures.

```typescript
test('name', async ({ browser }) => { ... })
```

---

### BrowserContext

`BrowserContext` is an isolated session — exactly like an incognito window. Each context has its own:

- Cookies
- localStorage / sessionStorage
- Cache
- Auth state
- Zero sharing with other contexts

In Playwright Test, every test automatically gets a **fresh BrowserContext** — so tests never affect each other even when running in parallel.

```typescript
const context = await browser.newContext({
  viewport: { width: 1280, height: 720 },
  locale: 'en-IN',
  timezoneId: 'Asia/Kolkata',
  storageState: 'auth/admin.json',  // saved login state
  httpCredentials: { username: 'user', password: 'pass' },
  geolocation: { longitude: 77.2, latitude: 28.6 },
});
```

---

### Page

A `page` represents a single browser tab. Multiple pages can exist inside one context and they share the same session — cookies, localStorage etc.

```typescript
const page = await context.newPage();  // new tab inside that context
```

---

### How Isolation Works in Parallel Tests

```
Test 1 → own BrowserContext → own Page → own cookies
Test 2 → own BrowserContext → own Page → own cookies
Test 3 → own BrowserContext → own Page → own cookies

All 3 running simultaneously — zero interference
```


---

### Browser, Context and Page Level Properties in Playwright Library

When launching a browser via Playwright library, properties are distributed across 3 levels — browser launch, context creation, and page level.

---

### Browser Level — `chromium.launch()`
Controls how the browser process starts.

```typescript
const browser = await chromium.launch({
  headless: false,          // browser visible or not
  slowMo: 500,              // slow down every action by 500ms — useful for debugging
  channel: 'chrome',        // use real Chrome or Edge instead of Playwright's Chromium
  args: ['--start-maximized'], // browser startup arguments
  devtools: true,           // auto open DevTools
  timeout: 30000,           // max time to launch browser
  executablePath: '/path/to/browser', // custom browser binary path
});
```

---

### Context Level — `browser.newContext()`
Controls the isolated session — like configuring an incognito profile.

```typescript
const context = await browser.newContext({
  baseURL: 'https://example.com',       // relative URLs possible in goto()
  viewport: { width: 1280, height: 720 }, // browser window size
  locale: 'en-IN',                      // language/locale setting
  timezoneId: 'Asia/Kolkata',           // timezone
  geolocation: { latitude: 28.6, longitude: 77.2 }, // mock location
  permissions: ['geolocation'],         // grant browser permissions
  storageState: 'auth/admin.json',      // load saved login state
  httpCredentials: { username: 'u', password: 'p' }, // basic auth
  ignoreHTTPSErrors: true,              // ignore SSL certificate errors
  recordVideo: { dir: 'videos/' },      // record video for this session
  extraHTTPHeaders: { 'x-token': 'abc' }, // attach headers to every request
  offline: true,                        // simulate offline mode
  colorScheme: 'dark',                  // dark or light mode
});
```

---

### Page Level — `context.newPage()`
Controls individual tab behavior after page is created.

```typescript
const page = await context.newPage();

await page.setViewportSize({ width: 1920, height: 1080 }); // override viewport
await page.setExtraHTTPHeaders({ 'x-custom': 'value' });   // extra headers for this page only
await page.setDefaultTimeout(15000);        // default timeout for all actions on this page
await page.setDefaultNavigationTimeout(30000); // timeout for goto() and navigation
await page.emulateMedia({ media: 'print' }); // emulate print or screen media
```

---

### Key Point
- **Browser level** — controls the process and how it starts
- **Context level** — controls the session, auth, network, and environment
- **Page level** — controls individual tab behavior and overrides

| Engine   | Browser              | Details                                                       |
| -------- | -------------------- | ------------------------------------------------------------- |

**Playwright ka Chromium ≠ Google Chrome:**

**WebKit on Windows/Linux:**
System pe Safari installed nahi hai — lekin Playwright ka patched WebKit Windows aur Linux pe bhi chal sakta hai. Toh bina Mac ke bhi Safari behavior test kar sakte ho.

## 14. PLAYWRIGHT TEST vs PLAYWRIGHT LIBRARY


**`playwright`** — npm package that provides only the core browser automation API. It gives you `chromium`, `firefox`, and `webkit` launch methods but has no test runner, no assertions, no fixtures, no reporters, and no parallel execution. Used for scripting, web scraping, or data extraction where a testing framework is not needed.

**`@playwright/test`** — npm package that is the complete test framework. It includes everything — test runner, assertions, fixtures, reporters, parallel execution, retries, and the core browser automation API as well. This is what you always use for test automation.

---

### Invoking Browsers via Playwright Library

```typescript
import { chromium, firefox, webkit } from 'playwright';

// Chromium engine
const browser = await chromium.launch();

// Firefox engine
const browser = await firefox.launch();

// WebKit engine
const browser = await webkit.launch();

// Chrome or Edge real browser application — via channel
const browser = await chromium.launch({ channel: 'chrome' });
const browser = await chromium.launch({ channel: 'msedge' });
```

- Import is from **`playwright`** package
- Everything has to be managed manually — newContext(), newPage(), browser.close()
- No auto-cleanup, no assertions, no retries

---

### Invoking Browsers via Playwright Test Runner

```typescript
import { test, expect } from '@playwright/test';

test('example', async ({ page }) => {
  await page.goto('/');
  await expect(page).toHaveTitle(/Example/);
});
```

- Import is from **`@playwright/test`** package
- `test` is the test runner function and `expect` is the assertion library
- `page`, `context`, `browser` are injected automatically via **fixtures** — no manual setup or cleanup needed

**Enabling browsers via `playwright.config.ts` projects block:**

```typescript
projects: [
  { name: 'chromium', use: { ...devices['Desktop Chrome'] } },
  { name: 'firefox',  use: { ...devices['Desktop Firefox'] } },
  { name: 'webkit',   use: { ...devices['Desktop Safari'] } },

  // Real browser applications via channel
  { name: 'chrome',   use: { channel: 'chrome' } },
  { name: 'edge',     use: { channel: 'msedge' } },
]
```

- Engine-based (chromium, firefox, webkit) — uses **Playwright's patched browser binaries**
- Channel-based (chrome, msedge) — uses the **real browser application** installed on your OS

### Chromium, Firefox, and WebKit in Playwright

- **`chromium`** — Playwright's patched Chromium engine. This is not Google Chrome — it is the open-source Chromium base with Playwright's CDP injected into it.
- **`firefox`** — Playwright's patched Firefox engine with CDP-like protocol support injected by Playwright.
- **`webkit`** — Playwright's patched WebKit engine (Safari's rendering engine) that can run on Windows and Linux as well.

These three are imported from the **`playwright`** core library package and are used to manually launch browser instances.

---

### Import Statement Difference

```typescript
import { chromium, firefox, webkit } from 'playwright';
// Use this when using Playwright as a library — manual browser control
// No test runner, no assertions, no fixtures

import { test, expect } from '@playwright/test';
// Use this when writing tests via Playwright Test framework
// test = test runner function, expect = assertion library
// fixtures like page, context, browser are auto-injected
```

---

### Launching Chromium via Playwright Library and Performing Actions

```typescript
import { chromium } from 'playwright';

const browser = await chromium.launch({ headless: false }); // headless: false = browser visible
const context = await browser.newContext();
const page    = await context.newPage();

await page.goto('https://example.com');
await page.locator('#username').click();

await browser.close(); // manual cleanup required
```

## 15. BUILT-IN FIXTURES

### Fixtures in Playwright

- Fixtures are Playwright Test's **dependency injection system**. Whatever you write inside the curly braces of `test()` — those are fixtures.
- Playwright automatically **creates** the fixture before the test, **injects** it into the test, and **destroys** it after the test.
- Fixtures replace the need for `beforeEach` / `afterEach` — setup, usage, and teardown are all managed in one place by Playwright.

```typescript
test('name', async ({ page, context, browser, request, browserName }) => {
  //                   ↑      ↑        ↑        ↑         ↑
  //              tab  session  instance  API     which browser?
});
```

---

### Purpose of Each Built-in Fixture

- **`page`** — gives a fresh browser tab inside a fresh context. Most commonly used fixture in day-to-day test writing.

- **`context`** — gives the BrowserContext that owns the page. Use this when you need to open multiple tabs in the same session, handle new tab events, or when you need to share cookies and auth state across tabs.
```typescript
  // Example — same user logged in, opening a new tab
  test('new tab', async ({ page, context }) => {
    const [newPage] = await Promise.all([
      context.waitForEvent('page'),
      page.getByRole('link', { name: 'Open in new tab' }).click()
    ]);
    await newPage.waitForLoadState();
  });
```

- **`browser`** — gives the raw browser instance. Use this when you need to create **multiple contexts manually** — for example, testing two different users logged into the same application simultaneously.
```typescript
  // Example — two users logged in at the same time
  test('two users', async ({ browser }) => {
    const adminContext = await browser.newContext({ storageState: 'auth/admin.json' });
    const userContext  = await browser.newContext({ storageState: 'auth/user.json' });
    const adminPage    = await adminContext.newPage();
    const userPage     = await userContext.newPage();
  });
```

- **`request`** — gives a direct HTTP API request context. Use this for API testing or setting up test data via API calls before UI tests run.

- **`browserName`** — gives the name of the current browser as a string — `'chromium'`, `'firefox'`, or `'webkit'`. Use this for writing conditional logic when behavior differs across browsers.

---

### When to Use `page` vs `context` vs `browser`

- Use **`page`** — when your test involves a single tab and single user session. This covers 90% of test cases.
- Use **`context`** — when your test needs multiple tabs but for the **same user session** — like opening a link in a new tab.
- Use **`browser`** — when your test needs multiple isolated sessions — like **two different users** logged into the same application at the same time.


### Har Fixture Ka Purpose

| Fixture         | Type                  | Purpose                                                                      |
| --------------- | --------------------- | ---------------------------------------------------------------------------- |
| `page`        | `Page`              | Fresh browser tab in fresh context — most commonly used                     |
| `context`     | `BrowserContext`    | Context jo page ko own karta hai — multiple tabs ke liye                    |
| `browser`     | `Browser`           | Browser instance — custom contexts banana ho to                             |
| `request`     | `APIRequestContext` | Direct HTTP API calls — API testing ya test data setup                      |
| `browserName` | `string`            | `'chromium'` \| `'firefox'` \| `'webkit'` — conditional logic ke liye |

## 16. TEST ANATOMY — BASIC STRUCTURE

```typescript
import { test, expect } from '@playwright/test';

// Simple test
test('test name', async ({ page }) => {
  await page.goto('/login');
  await expect(page).toHaveTitle('Login Page');
});

// Grouped tests
test.describe('Feature: Login', () => {

  test.beforeAll(async ({ browser }) => {
    // Ek baar — poore describe block se pehle
    // Heavy setup — DB seed, server start
  });

  test.afterAll(async () => {
    // Ek baar — poore describe block ke baad
    // Cleanup
  });

  test.beforeEach(async ({ page }) => {
    // Har test se pehle
    await page.goto('/login');
  });

  test.afterEach(async ({ page }) => {
    // Har test ke baad
    // Screenshot ya logs
  });

  test('valid login', async ({ page }) => { ... });
  test('invalid password', async ({ page }) => { ... });
});

// Special modifiers
test.only('sirf yeh test chale', async ({ page }) => { ... }); // baaki skip
test.skip('yeh skip karo', async ({ page }) => { ... });
test.fixme('broken test', async ({ page }) => { ... });        // known failure mark
```

## 18. DEBUG TOOLS — COMPLETE GUIDE

### Tool 1 — Playwright Inspector

Live step-through debugger. Browser headed mode mein khulta hai + ek panel jahan se actions control karte hain.

**3 Tarike kholne ke:**

```bash
# Tarika 1 — CLI flag
npx playwright test --debug

# Tarika 2 — Environment variable
PWDEBUG=1 npx playwright test

# Tarika 3 — Code mein likhdo
await page.pause();  // yahan Inspector khulega
```

**Inspector mein kya kar sakte ho:**

- **Resume** → test normally continue karo
- **Step Over** → ek action execute karo, phir ruko
- **Pick Locator** → browser mein element pe hover karo → Playwright best locator suggest karta hai
- **Explore** → element ka ARIA role, accessible name, attributes inspect karo
- **Locator bar** → apna locator type karo → dekhte hain match hota hai ya nahi

> ⚠️ `await page.pause()` sirf debugging ke liye use karo — production code mein mat chhodna

### Tool 2 — `--headed` Mode

```bash
npx playwright test --headed
```

Browser visible ho jaata hai — step-by-step control nahi hota, test normally chalta hai but tum dekh sakte ho kya ho raha hai. Quick visual check ke liye.

### Tool 3 — UI Mode

```bash
npx playwright test --ui
```

Visual test runner — browser mein ek complete GUI:

- Sab tests listed hote hain — individually run karo
- Real-time results dekhte hain
- Locator picker built-in
- Watch mode — file save pe auto re-run
- Time-travel debugging

### Tool 4 — Trace Viewer

Post-mortem debugging tool — test run hone ke baad. Test ka complete recording ek ZIP file mein save hoti hai.

**Trace file mein kya hota hai:**

- Har action ka pehle aur baad ka screenshot
- DOM snapshot — us moment ka complete HTML
- Network requests aur responses
- Browser console logs
- Source code line jo execute ho raha tha

**4 Trace Options:**

| Option                  | Kab Record                        | Kab Use Karo                   |
| ----------------------- | --------------------------------- | ------------------------------ |
| `'on-first-retry'`    | Sirf retry pe                     | CI standard ✅                 |
| `'retain-on-failure'` | Fail hone pe — retry ke bina bhi | Agar retries nahi hain         |
| `'on'`                | Hamesha                           | Local debugging — heavy files |
| `'off'`               | Kabhi nahi                        | Fastest — production          |

**CI mein recommended combination:**

```typescript
retries: 2,
use: {
  trace: 'on-first-retry',   // fail pe retry hoga → trace milegi
}
```

**Trace dekhne ke 2 tarike:**


## 📋 COMMANDS SUMMARY

### 1. Project Setup

| Command | Purpose |
| --- | --- |
| `npm init playwright@latest` | Fresh Playwright project banao — folder structure, config, sample tests generate hote hain |
| `npm install -D @playwright/test` | Existing project mein Playwright add karo — sirf package install, no config generated |

### 2. Browser Installation

| Command | Purpose |
| --- | --- |
| `npx playwright install` | Teeno browsers download karo — Chromium, Firefox, WebKit |
| `npx playwright install chromium` | Sirf Chromium install karo |
| `npx playwright install firefox` | Sirf Firefox install karo |
| `npx playwright install webkit` | Sirf WebKit install karo |
| `npx playwright install chromium firefox` | Specific multiple browsers ek saath install karo |
| `npx playwright install --with-deps` | Browsers + OS-level system dependencies install karo (Linux CI mein zaroori) |
| `npx playwright install chromium --with-deps` | Chromium + uske OS system dependencies install karo |
| `npx playwright install --dry-run` | Kya install hoga preview karo — bina actually install kiye |

### 3. Verification

| Command | Purpose |
| --- | --- |
| `npx playwright --version` | Installed Playwright version check karo |
| `node -v` | Node.js version check karo (v18+ required) |
| `npm -v` | npm version check karo |

### 4. Running Tests

| Command | Purpose |
| --- | --- |
| `npx playwright test` | Sab tests, sab configured browsers pe run karo |
| `npx playwright test login.spec.ts` | Specific test file run karo |
| `npx playwright test login auth` | Multiple files — partial name match se filter karo |
| `npx playwright test --grep "login"` | Test name se filter karo |
| `npx playwright test --grep-invert "slow"` | Specific tests exclude karo |
| `npx playwright test --project=chromium` | Specific browser pe run karo |
| `npx playwright test --project=chromium --project=firefox` | Multiple specific browsers pe run karo |
| `npx playwright test --headed` | Browser visible mode mein run karo |
| `npx playwright test --headless` | Browser hidden mode mein run karo (default) |
| `npx playwright test -x` | First failure pe execution stop karo |
| `npx playwright test --retries=3` | Retry count config se override karo |
| `npx playwright test --workers=2` | Parallel worker count override karo |
| `npx playwright test --timeout=60000` | Timeout override karo (milliseconds) |
| `npx playwright test --repeat-each=3` | Har test N baar repeat karo |
| `npx playwright test --reporter=list` | Reporter config se override karo |

### 5. Debugging

| Command | Purpose |
| --- | --- |
| `npx playwright test --debug` | Playwright Inspector mode mein test run karo |
| `PWDEBUG=1 npx playwright test` | Environment variable se Inspector enable karo |
| `npx playwright test --ui` | Visual UI mode mein tests run karo |
| `await page.pause()` | Test code mein is line par Inspector open hoga (debugging ke liye) |
