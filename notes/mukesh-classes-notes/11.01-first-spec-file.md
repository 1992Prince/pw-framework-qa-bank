# 11.01 — Writing Your First Spec File

---

## 1. SPEC FILE KYA HOTA HAI?

Spec file ek test file hoti hai — jisme hum apne actual test cases likhte hain.

- Extension `.spec.ts` ya `.spec.js` hoti hai — convention hai jo Playwright follow karta hai.
- Playwright by default `tests/` folder mein `.spec.` wali files dhundhta hai — yeh `testDir` se configure hota hai `playwright.config` mein.

---

## 2. IMPORT STATEMENT — `@playwright/test` SE IMPORT KARNA

```javascript
import { test, expect } from '@playwright/test';
```

- **`test`** — object hai jisse hum test cases define karte hain
- **`expect`** — function hai jisse hum assertions likhte hain (expected outcomes verify karte hain)
- Dono `@playwright/test` module se aate hain — jo `node_modules` mein installed hai

### Yeh import kyun zaroori hai?

- Default JavaScript/Node.js mein `test` ya `expect` nahi hote — Playwright apne module ke through yeh provide karta hai.
- Jab humne `npm install -D @playwright/test` ya `npm init playwright@latest` run kiya — tab `@playwright/test` package `node_modules` mein aa gaya.
- `node_modules/@playwright/test` ke andar yeh sab functions aur objects export kiye hue hain — hum `import` statement se wahan se unhe le aate hain.

### Named Import (ES6) — `{}` Curly Braces Kyon?

```javascript
import { test, expect } from '@playwright/test';
//       ↑     ↑
//   named imports — sirf yahi chahiye hain
```

- `{}` curly braces mein sirf wahi names likhte hain jo chahiye hain — ise **named import** kehte hain (ES6 destructuring).
- `@playwright/test` module mein aur bhi cheezein exported hain jaise `devices`, `defineConfig` — unhe bhi isi tarah import kar sakte hain.

```javascript
import { test, expect, devices } from '@playwright/test';
import { defineConfig } from '@playwright/test';
```

### CommonJS vs ES Modules

| | ES Modules (Modern) | CommonJS (Older) |
| --- | --- | --- |
| Syntax | `import { test } from '...'` | `const { test } = require('...')` |
| `package.json` setting | `"type": "module"` | default (no type field) |
| Best Practice | ✅ Recommended | Avoid for new projects |

> **Best Practice** — ES Modules (`import`) use karo. CommonJS (`require`) purana style hai.

## 3. `test()` FUNCTION — TEST CASE LIKHNA

```javascript
test('test name here', async ({ page }) => {
  // test body — yahan actual test steps likhte hain
});
```

### Dono Arguments Samjho

**Argument 1 — Test Name (string)**

- Pehla argument test ka naam/title hota hai.
- Yahi naam Playwright report mein, terminal output mein, aur HTML report mein dikhta hai.
- Descriptive naam likho — kya verify ho raha hai yeh clear karo.

```javascript
test('Verify login page title', async ({ page }) => { ... });
test('TC001 - User can submit login form', async ({ page }) => { ... });
```

**Argument 2 — Async Callback Function**

- Doosra argument ek async function hai — iske andar actual test steps likhte hain.
- **Async kyun?** — Browser pe kiye jaane wale saare actions (click, goto, fill) Promises return karte hain — yeh asynchronous operations hain. `async/await` se hum ensure karte hain ki ek action poora hone ke baad hi agla shuru ho — yani sequential execution.

```javascript
test('example', async ({ page }) => {
  await page.goto('https://playwright.dev');   // wait karo jab tak page load na ho
  await page.locator('#btn').click();          // wait karo jab tak click na ho
  await expect(page).toHaveTitle(/Playwright/); // wait karo assertion pass hone tak
});
```

**`{ page }` — Fixture Injection**

- `{ page }` ek **built-in fixture** hai jo Playwright automatically inject karta hai.
- Yeh ek browser tab (Page object) represent karta hai — isi pe saari actions karte hain.
- Playwright ke fixtures Dependency Injection system se kaam karte hain — hum sirf naam likhte hain, Playwright khud setup aur cleanup karta hai.

## 4. PEHLA SPEC FILE — COMPLETE EXAMPLE

```javascript
import { test, expect } from '@playwright/test';

test('Verify Playwright page title', async ({ page }) => {
  await page.goto('https://playwright.dev');
  await expect(page).toHaveTitle(/Playwright/);
});
```

Yahi minimum structure hai ek working Playwright test ke liye.

---

## 5. `page` OBJECT — NAVIGATION AUR ACTIONS

`page` object Playwright ka **sabse important object** hai. Yeh ek browser tab ko represent karta hai.

### Navigation

```javascript
// URL pe navigate karo
await page.goto('https://example.com');

// ⚠️ IMPORTANT — hamesha https:// ya http:// lagao
await page.goto('https://www.google.com');   // ✅ correct
await page.goto('www.google.com');           // ❌ Protocol Error — invalid URL
```

`page.goto()` tab resolve hoti hai jab page ek certain load state reach kar le — by default yeh `load` event ka wait karta hai.

### Fixed Wait (Sparingly Use Karo)

```javascript
await page.waitForTimeout(15000); // 15 seconds wait karo
```

> **Note** — `waitForTimeout` avoid karo jahan possible ho. Playwright ki auto-waiting mechanism better hai — yeh unnecessary delays add karta hai aur tests slow karta hai.

## 6. BEHIND THE SCENES — TEST RUN HONE PE KYA HOTA HAI ⭐

Jab `npx playwright test` run karte hain aur test mein `{ page }` fixture use ho, toh internally yeh hota hai:

```
npx playwright test
    ↓
Worker process start hota hai
    ↓
Browser instance invoke hota hai (worker level pe — Playwright ka patched binary)
    ↓
Har test ke liye → nayi BrowserContext create hoti hai (isolated session)
    ↓
Context ke andar → nayi Page (tab) create hoti hai
    ↓
Yeh page { page } fixture ke through test function mein inject hoti hai
    ↓
Test steps execute hote hain
    ↓
Cleanup: Page close → BrowserContext close (har test ke baad)
    ↓
Browser close (sab tests ke baad)
```

### Key Points

- **Browser** — worker level pe ek baar launch hota hai.
- **BrowserContext** — **har test ke liye alag** create hoti hai — isolated session, alag cookies, alag storage. Agar ek spec file mein 10 tests hain, toh same browser instance mein 10 alag contexts ek ke baad ek (sequential) ya ek saath (parallel) banenge.
- **Page** — context ke andar ek tab — yahi `{ page }` fixture ke roop mein milta hai.
- **Hum manually browser open/close nahi karte** — yeh sab Playwright internally hooks mein handle karta hai.

## 7. HOOKS — SETUP AUR CLEANUP

```javascript
import { test, expect } from '@playwright/test';

test.beforeEach(async ({ page }) => {
  // Har test se pehle run hota hai — setup ke liye
  await page.goto('https://example.com');
});

test.afterEach(async ({ page }) => {
  // Har test ke baad run hota hai — cleanup ke liye
});

test('TC001 - Verify title', async ({ page }) => {
  await expect(page).toHaveTitle(/Example/);
});

test('TC002 - Verify URL', async ({ page }) => {
  await expect(page).toHaveURL('https://example.com');
});
```

- `beforeEach` mein `page.goto()` likhte hain taaki har test fresh URL pe start kare.
- `afterEach` mein screenshots, logs, ya cleanup likha ja sakta hai.

## 8. ASSERTIONS — `expect()` FUNCTION

### Retry-able Assertions (hamesha `await` use karo)

Yeh assertions ek baar check karke nahi rukti — yeh **baar baar retry karti hain** jab tak condition pass na ho ya timeout na ho jaye.

```javascript
await expect(page).toHaveTitle(/Playwright/);        // page title check
await expect(page).toHaveURL('https://example.com'); // page URL check
await expect(locator).toBeVisible();                 // element visible hai?
await expect(locator).toHaveText('Hello');           // element ka text check
await expect(locator).toBeEnabled();                 // element enabled hai?
await expect(locator).toHaveValue('abc');            // input field ki value
```

> **`await` kyun zaroori hai?** — Retry-able assertions Promises return karti hain — `await` se Playwright tab tak wait karta hai jab tak assertion pass na ho.

### Non-Retry Assertions (`await` nahi chahiye)

Yeh **instant checks** hain — ek baar evaluate hoti hain aur pass ya fail ho jaati hain. Yeh synchronous hain.

```javascript
expect(200).toBeGreaterThan(100);
expect(200).toBeLessThan(500);
expect(200).toBe(200);
expect(200).toBeCloseTo(200);
expect(['Amit', 'Bhupi', 'abc']).toContain('Amit');
expect(jsObject).toBeDefined();
```

### `.not` Modifier — Assertion Ko Negate Karo

Kisi bhi assertion ke pehle `.not` lagao to opposite check ho jaata hai:

```javascript
expect('playwright').not.toEqual('selenium');
await expect(locator).not.toBeVisible();
await expect(page).not.toHaveURL('/error');
```

### Retry vs Non-Retry — Quick Reference

| Type | `await` | Retries | Use For |
| --- | --- | --- | --- |
| Retry-able | ✅ Required | ✅ Yes — jab tak timeout | Browser/page/element state check |
| Non-Retry | ❌ Not needed | ❌ No — instant | Numbers, strings, objects check |

## 9. COMPLETE SPEC FILE EXAMPLE

```javascript
import { test, expect } from '@playwright/test';

test.beforeEach(async ({ page }) => {
  await page.goto('https://playwright.dev');
});

test('TC001 - Verify page title', async ({ page }) => {
  await expect(page).toHaveTitle(/Playwright/);
});

test('TC002 - Verify page URL', async ({ page }) => {
  await expect(page).toHaveURL('https://playwright.dev/');
});

test('TC003 - Verify docs link is visible', async ({ page }) => {
  await expect(page.getByRole('link', { name: 'Docs' })).toBeVisible();
});
```
