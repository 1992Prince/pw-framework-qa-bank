# Playwright Fixtures — Complete Notes

---

## Topic 1: Fixtures Kya Hote Hain

Fixture ek setup mechanism hai jo test run hone se pehle automatically environment prepare karta hai aur test khatam hone ke baad cleanup bhi karta hai. Jaise agar tumhe har test mein browser open karna hai, page navigate karna hai, ya login karna hai — yeh sab kaam fixture karta hai automatically.

Playwright mein jab tum test likhte ho toh `test('...', async ({ page }) => {})` mein jo `page` aata hai curly braces ke andar — woh ek fixture hai. Tumne manually kuch nahi kiya, playwright ne automatically browser khola, page create kiya aur tumhare test function mein de diya.

**Summary:**
- Fixture = test ka pre-setup + post-cleanup ek jagah
- Test function ko sirf test logic likhni chahiye, setup ki chinta fixture karta hai
- Fixtures reusable hote hain — ek baar likho, hazaron tests mein use karo
- Fixtures isolated hote hain — ek test ka fixture doosre test ko affect nahi karta

## Topic 2: Built-in Fixtures Playwright Mein

Playwright 5 built-in fixtures deta hai jo tum directly test mein use kar sakte ho bina kuch configure kiye.

### `page` — Type: `Page`
Ek naya browser tab/page milta hai har test ko. Iska scope test-level hota hai matlab har test ko fresh page milti hai.

```javascript
test('example', async ({ page }) => {
  await page.goto('https://example.com');
});
```

**Use case:** jab tumhe kisi URL pe navigate karna ho, element dhundhna ho, click karna ho.

---

### `context` — Type: `BrowserContext`
Ek isolated browser session hota hai. Iske andar multiple pages ho sakti hain. Cookies, localStorage is context ke andar bound hote hain.

```javascript
test('example', async ({ context }) => {
  const page1 = await context.newPage();
  const page2 = await context.newPage();
});
```

**Use case:** jab multiple tabs ek hi session mein test karne hon, ya cookies set karni hon test se pehle.

---

### `browser` — Type: `Browser`
Poora browser instance. Chromium, Firefox ya WebKit. Iska scope worker-level hota hai matlab ek worker process mein share hota hai.

```javascript
test('example', async ({ browser }) => {
  const context = await browser.newContext();
  const page = await context.newPage();
});
```

**Use case:** jab tumhe custom browser context banana ho apni settings ke saath.

---

### `browserName` — Type: `string`
Jo browser chal raha hai uska naam milta hai — `'chromium'`, `'firefox'`, ya `'webkit'`.

```javascript
test('example', async ({ browserName }) => {
  if (browserName === 'firefox') {
    // firefox specific logic
  }
});
```

**Use case:** jab tumhe browser-specific conditional code likhna ho.

---

### `request` — Type: `APIRequestContext`
Isse tum directly API calls kar sakte ho browser ke bina. REST APIs test karne ke liye.

```javascript
test('api test', async ({ request }) => {
  const response = await request.get('https://api.example.com/users');
  expect(response.status()).toBe(200);
});
```

**Use case:** backend APIs test karna, ya test setup mein data create karna API ke through.

---

**Summary:**
- 5 built-in fixtures hain: `page`, `context`, `browser`, `browserName`, `request`
- `page` aur `context` har test ke liye fresh hote hain — fully isolated
- `browser` worker-level pe share hota hai
- `browserName` se conditional browser logic likh sakte ho
- `request` se bina browser ke API test kar sakte ho

## Topic 3: Curly Braces Mein Fixtures Kyun Likhte Hain

Jab tum test function likhte ho toh aisa dikhta hai:

```javascript
test('login test', async ({ page, context }) => { ... });
```

Yahan curly braces JavaScript ka **destructuring** hai. Playwright internally ek bada object pass karta hai jisme saare fixtures hote hain. Tum sirf woh fixtures destructure karte ho jo tumhe chahiye.

Agar tum sab fixtures ek saath lena chaho toh technically nahi le sakte ek variable mein — kyunki fixtures **lazy** hote hain, matlab sirf wahi instantiate hota hai jo tum request karte ho. Agar tumne `browser` maanga toh sirf browser banega, agar `page` maanga toh page banega aur internally `context` aur `browser` bhi ban jayenge (kyunki page ke liye zaroori hain).

Tum multiple fixtures ek saath le sakte ho:

```javascript
test('full test', async ({ page, context, browserName, request }) => {
  // sab available hain
});
```

**Summary:**
- Curly braces = JavaScript destructuring
- Playwright ek fixtures object pass karta hai, tum sirf jo chahiye wo lo
- Multiple fixtures ek hi test mein le sakte ho
- Sirf jo fixtures destructure karo wo hi initialize hote hain — efficient hai

## Topic 4: Ek Test Mein Multiple Fixtures

Haan, ek test mein tum jitne chahiye utne fixtures use kar sakte ho. Built-in bhi aur custom bhi ek saath.

```javascript
test('complete test', async ({ page, browserName, request, loginPage }) => {
  // page       = browser tab
  // browserName = konsa browser chal raha hai
  // request    = API calls
  // loginPage  = tumhara custom fixture
});
```

**Summary:**
- Ek test mein multiple built-in + multiple custom fixtures saath use kar sakte ho
- Sab comma se separate karo curly braces ke andar
- Har fixture apna kaam karta hai independently

## Topic 5: Custom Fixtures — Kab Aur Kyun Banate Hain

**Problem:** Maan lo tumhare paas 10 spec files hain aur har spec file ke har test mein login karna hai. Agar tum `beforeEach` mein login likhte ho toh woh sirf us ek spec file ke tests ke liye kaam karega. Doosri spec file mein phir se likhna padega. Yeh code duplication hai.

**Solution:** Custom fixture banao jisme login logic likho. Phir us fixture ko kisi bhi spec file ke kisi bhi test mein use karo.

### Custom Fixture Banane Ka Structure

```javascript
// fixtures/auth.fixture.js
import { test as base } from '@playwright/test';

export const test = base.extend({
  loggedInPage: async ({ page }, use) => {

    // ── PRE-ACTION: setup karo ──
    await page.goto('/login');
    await page.fill('#email', 'user@test.com');
    await page.fill('#password', 'password123');
    await page.click('#submit');
    await page.waitForURL('/dashboard');

    // ── TEST KO PAGE DO ──
    await use(page);

    // ── CLEANUP: teardown karo ──
    await page.click('#logout');
  },
});

// Re-export expect bhi karo taaki test files sirf ek jagah se import karen
export { expect } from '@playwright/test';
```

Phir kisi bhi spec file mein:

```javascript
// spec1.spec.js
import { test, expect } from '../../fixtures/auth.fixture.js'; // playwright ka nahi, apna import

test('dashboard test', async ({ loggedInPage }) => {
  await expect(loggedInPage).toHaveURL('/dashboard');
});
```

```javascript
// spec2.spec.js
import { test, expect } from '../../fixtures/auth.fixture.js'; // same fixture, different file

test('profile test', async ({ loggedInPage }) => {
  await loggedInPage.goto('/profile');
});
```

### `use()` Function Ka Matlab

| Code Position | Equivalent |
|---------------|------------|
| `use()` se pehle | `beforeEach` ki tarah — setup |
| `await use(page)` | Test runs yahan |
| `use()` ke baad | `afterEach` ki tarah — cleanup |

**Summary:**
- `beforeEach`/`afterEach` sirf ek spec file ke andar kaam karta hai
- Custom fixture poori project mein use ho sakta hai, kisi bhi spec file mein
- `use()` ke pehle = setup, `use()` pe = test runs, `use()` ke baad = cleanup
- Ek fixture mein setup + test handoff + cleanup teeno cheezein hoti hain

## Topic 6: Custom Fixtures Ke Aur Use Cases

### Login Fixture
Jaise upar dekha, multiple spec files mein reuse hota hai.

---

### Excel/CSV Data Reading Fixture

```javascript
export const test = base.extend({
  testData: async ({}, use) => {
    const data = await readExcelFile('./test-data/users.xlsx');
    await use(data);
    // cleanup agar zaroori ho
  },
});
```

---

### Database Connection Fixture

```javascript
export const test = base.extend({
  db: async ({}, use) => {
    const connection = await createDBConnection(config);
    await use(connection);
    await connection.close(); // cleanup automatically
  },
});
```

---

### Conditional Logic Fixture

```javascript
export const test = base.extend({
  authPage: async ({ page, browserName }, use) => {
    if (browserName === 'firefox') {
      // firefox pe alag login flow
    } else {
      // default login flow
    }
    await use(page);
  },
});
```

---

### Multiple Custom Fixtures Ek Saath

```javascript
export const test = base.extend({
  loginPage: async ({ page }, use) => { /* ... */ },
  testData: async ({}, use) => { /* ... */ },
  db: async ({}, use) => { /* ... */ },
});

// Test mein:
test('test', async ({ loginPage, testData, db }) => { /* ... */ });
```

**Summary:**
- Fixtures se login, data reading, DB connection — sab handle ho sakta hai
- Ek hi `extend()` mein multiple custom fixtures define kar sakte ho
- Fixture ke andar built-in fixtures bhi inject kar sakte ho (jaise `page`, `browserName`)
- Cleanup automatically hoti hai `use()` ke baad — no manual `afterEach` needed

## Topic 7: Fixtures Internally Kaise Kaam Karte Hain

Playwright ka fixture system **dependency injection** pe based hai. Jab tum `{ page }` likhte ho test mein, playwright dekhta hai ki `page` ke liye kya chahiye. `page` ko `context` chahiye, `context` ko `browser` chahiye. Playwright automatically yeh chain build karta hai.

### Fixture Scopes

| Scope | Behavior | Example |
|-------|----------|---------|
| `test` (default) | Har test ke liye naya instance — test khatam toh destroy | `page`, `context` |
| `worker` | Ek worker process mein sab tests ke beech share hota hai | `browser` |

### Isolation

- Har test ka apna `page` aur `context` hota hai — ek test ka kuch bhi doosre test ko affect nahi karta
- Agar ek test fail ho aur browser crash bhi kare, doosra test fresh page se start karta hai

### `expect` Re-export Kyun Karte Hain

```javascript
export { expect } from '@playwright/test';
```

Isse yeh fayda hota hai ki test files sirf ek hi jagah se import karti hain — tumhari custom fixture file. Playwright ka `test` replace ho gaya tumhare custom `test` se, aur `expect` bhi saath aata hai. Koi confusion nahi ki kahan se import karna hai.

**Summary:**
- Fixtures dependency injection se kaam karte hain — playwright khud resolve karta hai chain
- Test-scope fixtures = fully isolated, har test fresh start karta hai
- Worker-scope fixtures = shared for performance (jaise browser instance)
- `expect` re-export karo taaki imports clean rahein poori project mein
- Fixture ek test mein initialize hota hai, doosre test mein fresh initialize hota hai — koi sharing nahi test-scope mein
