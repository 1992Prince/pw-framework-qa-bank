# Playwright — Execution Order, Priority & testInfo

---

## Part 1 — Execution Order of Tests

### Rule: Tests Run in Declaration Order (Top to Bottom)

Playwright tests **exactly usi order mein run karte hain jis order mein tum ne unhe file mein likha hai** — upar se neeche. Alphabetical order nahi hota.

### Example — Declaration Order

```javascript
import { test, expect } from "@playwright/test";

test("TC003 - Logout", async ({ page }) => {
  console.log("▶ TC003 running");
});

test("TC001 - Login", async ({ page }) => {
  console.log("▶ TC001 running");
});

test("TC002 - Dashboard", async ({ page }) => {
  console.log("▶ TC002 running");
});
```

**Output:**
```
▶ TC003 running   ← pehle — kyunki file mein pehle likha hai
▶ TC001 running   ← doosra
▶ TC002 running   ← teesra
```

Naam mein `TC001`, `TC002`, `TC003` hai — lekin order naam se nahi, **position se** decide hota hai.

---

### describe Groups Ka Order

Groups bhi top-to-bottom chalte hain:

```javascript
import { test, expect } from "@playwright/test";

test.describe("Cart Tests", () => {
  test("TC001 - Add item", async ({ page }) => {
    console.log("▶ [Cart] TC001");
  });
});

test.describe("Login Tests", () => {
  test("TC001 - Valid login", async ({ page }) => {
    console.log("▶ [Login] TC001");
  });
});

test("TC000 - Homepage", async ({ page }) => {
  console.log("▶ TC000 - Independent");
});
```

**Output:**
```
▶ [Cart] TC001         ← Cart group pehle (file mein upar hai)
▶ [Login] TC001        ← Login group doosra
▶ TC000 - Independent  ← Independent test teesra
```

---

### Parallel Mode Mein Order

Jab `workers > 1` ho (parallel execution), tests **different workers pe simultaneously** chalte hain — toh strict order guarantee nahi hota. Lekin ek hi worker pe order top-to-bottom hi rahega.

```
Workers = 1 (serial):  TC001 → TC002 → TC003 → TC004
Workers = 4 (parallel): TC001 + TC002 + TC003 + TC004 (saath mein, alag alag workers pe)
```

## Part 2 — Can We Set Priority?

### Short Answer

**Playwright mein built-in "priority" property nahi hai.** Koi `test.priority(1)` ya `{ priority: 1 }` jaisi cheez nahi hoti.

Lekin tum **3 tarike se priority achieve kar sakte ho:**

---

### Tarika 1 — Declaration Order Se (Sabse Simple)

Jo test pehle chalana chahte ho, usse file mein upar likho:

```javascript
import { test, expect } from "@playwright/test";

// ─── High Priority Tests (upar hain — pehle chalenge) ───
test("CRITICAL - Payment gateway working", async ({ page }) => {
  console.log("▶ CRITICAL test running first");
  await page.goto("https://example.com/pay");
  await expect(page.locator(".payment-form")).toBeVisible();
});

test("CRITICAL - Login working", async ({ page }) => {
  console.log("▶ CRITICAL test running second");
  await page.goto("https://example.com/login");
  await expect(page.locator("form")).toBeVisible();
});

// ─── Low Priority Tests (neeche hain — baad mein chalenge) ───
test("LOW - Footer links check", async ({ page }) => {
  console.log("▶ LOW priority test");
  await page.goto("https://example.com");
  await expect(page.locator("footer")).toBeVisible();
});
```

---

### Tarika 2 — Tags + --grep Se (Most Practical)

High priority tests ko `@smoke` tag do, low priority ko `@regression`. Phir CI/CD mein pehle smoke run karo:

```javascript
import { test, expect } from "@playwright/test";

test("TC001 - Login", { tag: "@smoke" }, async ({ page }) => {
  console.log("▶ [HIGH] TC001 smoke test");
  await page.goto("https://example.com/login");
  await expect(page).toHaveTitle(/Login/);
});

test("TC002 - Payment", { tag: "@smoke" }, async ({ page }) => {
  console.log("▶ [HIGH] TC002 smoke test");
  await page.goto("https://example.com/pay");
  await expect(page.locator(".payment-form")).toBeVisible();
});

test("TC003 - Profile picture upload", { tag: "@regression" }, async ({ page }) => {
  console.log("▶ [LOW] TC003 regression test");
  // slow, non-critical test
});

test("TC004 - Email preferences", { tag: "@regression" }, async ({ page }) => {
  console.log("▶ [LOW] TC004 regression test");
  // slow, non-critical test
});
```

**CI/CD Pipeline mein:**
```bash
# Step 1: High priority (fast feedback)
npx playwright test --grep @smoke

# Step 2: Low priority (full coverage)
npx playwright test --grep @regression
```

## Part 3 — What is testInfo?

### Definition

`testInfo` ek **object** hai jo Playwright automatically provide karta hai — isme currently running test ke baare mein saari metadata hoti hai.

Think of it like an **ID card** of the test — test ka naam, status, duration, file path, retry count — sab kuch isme hota hai.

---

### testInfo Kahan Available Hota Hai?

```javascript
// 1. Test body mein
test("TC001", async ({ page }, testInfo) => {
  console.log(testInfo.title);
});

// 2. afterEach mein
test.afterEach(async ({ page }, testInfo) => {
  console.log(testInfo.status);
});

// 3. beforeEach mein (limited use — status abhi available nahi)
test.beforeEach(async ({ page }, testInfo) => {
  console.log(testInfo.title);  // title available hai
});
```

> **Note:** `testInfo.status` (pass/fail) sirf `afterEach` ya test body ke end mein meaningful hota hai — `beforeEach` mein test abhi chala hi nahi hota.

---

### testInfo Properties — Complete Reference

```javascript
test.afterEach(async ({ page }, testInfo) => {
  // ─── Basic Info ───
  console.log(testInfo.title);           // "TC001 - Login test"
  console.log(testInfo.titlePath);       // ["Login Tests", "TC001 - Login test"]
  console.log(testInfo.file);            // "D:/git/pw-fusion/tests/login.spec.js"
  console.log(testInfo.line);            // 15 (test declaration ki line number)

  // ─── Status Info ───
  console.log(testInfo.status);          // "passed" | "failed" | "timedOut" | "skipped" | "interrupted"
  console.log(testInfo.expectedStatus);  // "passed" (usually — unless test.fail() use kiya ho)

  // ─── Timing Info ───
  console.log(testInfo.duration);        // 1234 (milliseconds)
  console.log(testInfo.startTime);       // Date object — test kab start hua

  // ─── Retry Info ───
  console.log(testInfo.retry);           // 0 = first attempt, 1 = first retry, 2 = second retry
  console.log(testInfo.repeatEachIndex); // 0 (--repeat-each flag ke saath use hota hai)

  // ─── Worker Info ───
  console.log(testInfo.workerIndex);     // 0, 1, 2 (konse worker pe chal raha hai)
  console.log(testInfo.parallelIndex);   // same as workerIndex

  // ─── Error Info (agar test fail hua) ───
  console.log(testInfo.error);           // Error object (message, stack)
  console.log(testInfo.errors);          // Array of errors (multiple failures ke liye)

  // ─── Output Path ───
  console.log(testInfo.outputDir);       // "test-results/TC001-Login-test-chromium"
  console.log(testInfo.snapshotDir);     // Snapshots ka directory
});
```

---

### testInfo Methods — Complete Reference

```javascript
test.afterEach(async ({ page }, testInfo) => {
  // ─── Attach files/screenshots to HTML report ───
  await testInfo.attach("screenshot", {
    body: await page.screenshot(),
    contentType: "image/png",
  });

  // ─── Attach text content ───
  await testInfo.attach("api-response", {
    body: JSON.stringify({ status: "ok" }),
    contentType: "application/json",
  });

  // ─── Add annotation (visible in report) ───
  testInfo.annotations.push({
    type: "issue",
    description: "https://jira.example.com/BUG-123",
  });

  // ─── Mark test as slow (3x timeout milega) ───
  testInfo.slow();

  // ─── Output path for a file ───
  const screenshotPath = testInfo.outputPath("screenshot.png");
  // Returns: "test-results/TC001-.../screenshot.png"
  await page.screenshot({ path: screenshotPath });
});
```

## Part 4 — Practical Examples: Where & How to Use testInfo

### Example 1 — Screenshot on Failure (Most Common Use)

```javascript
import { test, expect } from "@playwright/test";

test.beforeEach(async ({ page }) => {
  await page.goto("https://example.com/login");
});

test("TC001 - Valid login", async ({ page }) => {
  await page.fill('input[name="email"]', "user@example.com");
  await page.fill('input[name="password"]', "pass123");
  await page.click("button:has-text('Login')");
  await expect(page).toHaveURL(/dashboard/);
});

test("TC002 - Invalid login", async ({ page }) => {
  await page.fill('input[name="email"]', "wrong@example.com");
  await page.fill('input[name="password"]', "wrong");
  await page.click("button:has-text('Login')");
  await expect(page.locator(".error")).toBeVisible();
});

test.afterEach(async ({ page }, testInfo) => {
  // Sirf failure pe screenshot attach karo
  if (testInfo.status !== testInfo.expectedStatus) {
    const screenshot = await page.screenshot({ fullPage: true });
    await testInfo.attach(`${testInfo.title} - failure`, {
      body: screenshot,
      contentType: "image/png",
    });
    console.log(`📸 Screenshot attached for failed test: ${testInfo.title}`);
  }
});
```

---

### Example 2 — Retry-Aware Logging

```javascript
import { test, expect } from "@playwright/test";

test.beforeEach(async ({ page }, testInfo) => {
  if (testInfo.retry > 0) {
    console.log(`🔄 Retry attempt #${testInfo.retry} for: "${testInfo.title}"`);
  } else {
    console.log(`▶ First attempt for: "${testInfo.title}"`);
  }
  await page.goto("https://example.com");
});

test("TC001 - Homepage", async ({ page }) => {
  await expect(page).toHaveTitle(/Example/);
});

test.afterEach(async ({ page }, testInfo) => {
  const icon = testInfo.status === "passed" ? "✅" : "❌";
  console.log(
    `${icon} "${testInfo.title}" — ${testInfo.status} ` +
    `(${testInfo.duration}ms, retry: ${testInfo.retry})`
  );
});
```

**Output (agar TC001 pehle fail ho, phir retry pe pass ho):**
```
▶ First attempt for: "TC001 - Homepage"
❌ "TC001 - Homepage" — failed (3012ms, retry: 0)

🔄 Retry attempt #1 for: "TC001 - Homepage"
✅ "TC001 - Homepage" — passed (812ms, retry: 1)
```

---

### Example 3 — Bug Report Annotation

```javascript
import { test, expect } from "@playwright/test";

test("TC001 - Coupon validation", async ({ page }, testInfo) => {
```

## Part 5 — Can We Push Custom Metadata into testInfo?

### Short Answer

**Directly testInfo pe custom properties add nahi kar sakte** — yeh ek read-only object hai jiske properties Playwright define karta hai. Lekin `testInfo.annotations` array ke through tum apna **custom key-value metadata** push kar sakte ho. Yeh HTML report mein bhi dikhta hai.

---

### Tarika 1 — testInfo.annotations (Official Way)

`annotations` ek array hai jisme tum apne custom objects push kar sakte ho. Har object mein `type` (label) aur optional `description` (value) hoti hai.

```javascript
import { test, expect } from "@playwright/test";

test("TC001 - Checkout test", async ({ page }, testInfo) => {
  // Custom metadata push karo
  testInfo.annotations.push({ type: "testCaseId",   description: "TC-789" });
  testInfo.annotations.push({ type: "sprint",        description: "Sprint-24" });
  testInfo.annotations.push({ type: "module",        description: "Checkout" });
  testInfo.annotations.push({ type: "testedBy",      description: "Rahul" });
  testInfo.annotations.push({ type: "environment",   description: "staging" });

  await page.goto("https://example.com/checkout");
  await expect(page.locator(".checkout-form")).toBeVisible();
});
```

**HTML Report mein:**
```
TC001 - Checkout test   ✅
  Annotations:
    testCaseId   : TC-789
    sprint       : Sprint-24
    module       : Checkout
    testedBy     : Rahul
    environment  : staging
```

---

### Tarika 2 — beforeEach Mein Global Metadata

Agar har test pe same metadata chahiye (jaise environment, build number), toh `beforeEach` mein push karo:

```javascript
import { test, expect } from "@playwright/test";

test.beforeEach(async ({}, testInfo) => {
  // Har test pe yeh metadata automatically add hogi
  testInfo.annotations.push({ type: "environment", description: process.env.ENV || "staging" });
  testInfo.annotations.push({ type: "buildNumber", description: process.env.BUILD_NUMBER || "local" });
  testInfo.annotations.push({ type: "appVersion",  description: "v2.4.1" });
});

test("TC001 - Login", async ({ page }) => {
  await page.goto("https://example.com/login");
  await expect(page).toHaveTitle(/Login/);
});

test("TC002 - Dashboard", async ({ page }) => {
  await page.goto("https://example.com/dashboard");
  await expect(page.locator("h1")).toBeVisible();
});
```

Dono tests ke report mein automatically environment + buildNumber + appVersion annotation dikhegi.

---

### Tarika 3 — Dynamic Metadata Based on Test Result

```javascript
import { test, expect } from "@playwright/test";

test("TC001 - Payment flow", async ({ page }, testInfo) => {
  // Test-specific metadata
  testInfo.annotations.push({ type: "jiraTicket",  description: "PAY-101" });
  testInfo.annotations.push({ type: "priority",    description: "P1-Critical" });

  await page.goto("https://example.com/payment");
  await page.fill('input[name="card"]', "4111111111111111");
  await page.click("button:has-text('Pay')");
  await expect(page).toHaveURL(/success/);
});

test.afterEach(async ({}, testInfo) => {
  // Result ke baad dynamic metadata add karo
  testInfo.annotations.push({
    type: "executionTime",
    description: `${testInfo.duration}ms`,
  });

  testInfo.annotations.push({
    type: "retryCount",
    description: `${testInfo.retry}`,
  });

  if (testInfo.status === "failed") {
    testInfo.annotations.push({
      type: "failureReason",
      description: testInfo.error?.message || "Unknown error",
    });
  }
});
```

**HTML Report (agar test fail hua):**
```
TC001 - Payment flow   ❌
  Annotations:
    jiraTicket     : PAY-101
    priority       : P1-Critical
    executionTime  : 5234ms
    retryCount     : 0
    failureReason  : expect(received).toHaveURL(expected)
```

### Tarika 4 — testInfo.outputPath() Se Custom Files Save Karo

Custom metadata ko ek file mein save karke report ke saath attach bhi kar sakte ho:

```javascript
import { test, expect } from "@playwright/test";
import fs from "fs";

test.afterEach(async ({ page }, testInfo) => {
  // Custom metadata object banao
  const metadata = {
    testId:       testInfo.title,
    status:       testInfo.status,
    duration:     `${testInfo.duration}ms`,
    worker:       testInfo.workerIndex,
    retry:        testInfo.retry,
    timestamp:    new Date().toISOString(),
    environment:  process.env.ENV || "staging",
    buildNumber:  process.env.BUILD_NUMBER || "local",
  };

  // File mein save karo
  const metaFilePath = testInfo.outputPath("test-metadata.json");
  fs.writeFileSync(metaFilePath, JSON.stringify(metadata, null, 2));

  // HTML report mein attach karo
  await testInfo.attach("test-metadata", {
    body: JSON.stringify(metadata, null, 2),
    contentType: "application/json",
  });

  console.log("📋 Metadata saved:", JSON.stringify(metadata));
});

test("TC001 - Homepage", async ({ page }) => {
  await page.goto("https://example.com");
  await expect(page).toHaveTitle(/Example/);
});
```

**HTML Report mein:**
```
TC001 - Homepage   ✅
  Attachments:
    📎 test-metadata   (application/json)
       {
         "testId": "TC001 - Homepage",
         "status": "passed",
         "duration": "412ms",
         "worker": 0,
         "retry": 0,
         "timestamp": "2026-04-25T10:30:00.000Z",
         "environment": "staging",
         "buildNumber": "local"
       }
```

### Summary — Custom Metadata Options

| Approach | How | Visible In Report? | Best For |
|----------|-----|--------------------|----------|
| `testInfo.annotations.push()` | Key-value push | ✅ Yes (Annotations section) | JIRA IDs, sprint, module, priority |
| `testInfo.attach()` with JSON | File/text attach | ✅ Yes (Attachments section) | Detailed structured data |
| `testInfo.outputPath()` + `fs.write` | Save to file | Only if attached | Large metadata, logs |
| `beforeEach` annotations | Global per test | ✅ Yes | Environment, build number |

---

## Summary

| Topic | Key Point |
|-------|-----------|
| Execution order | Top-to-bottom declaration order — alphabetical nahi |
| Groups order | Groups bhi top-to-bottom chalte hain |
| Parallel mode | Order guarantee nahi — different workers pe chalte hain |
| Priority | Built-in priority nahi hai — order, tags, ya serial mode se achieve karo |
| `testInfo` | Currently running test ki metadata — title, status, duration, retry, worker, etc. |
| `testInfo.status` | `"passed"`, `"failed"`, `"timedOut"`, `"skipped"` |
| `testInfo.retry` | `0` = first run, `1` = first retry |
| `testInfo.attach()` | Screenshots, files ko HTML report mein attach karo |
| `testInfo.annotations` | Report mein extra info add karo (bug links, env info) |
| `testInfo.workerIndex` | Parallel runs mein unique data generate karne ke liye use karo |
| Custom metadata | `testInfo.annotations.push()` ya `testInfo.attach()` — direct property add nahi kar sakte |
