# Data Driven Testing — Top Interview Questions & Answers

---

**Q1. What is Data Driven Testing? Why do we use it?**

**Answer:**  
Data Driven Testing is an approach where we write one test and run it multiple times with different sets of data. Instead of creating 10 separate tests with the same steps but different values, we keep the data externally — in a JSON file, CSV, array, or database — and loop through it. The framework picks one record at a time, runs the test with that data, then moves to the next record.

We use it when we need to validate the same functionality with multiple inputs — like testing a login form with valid credentials, invalid credentials, blank fields, special characters, etc. It saves time, reduces code duplication, and makes tests easier to maintain — if the test logic changes, we update it in one place, not in 10 tests.

---

**Q2. Where do you store test data in your Playwright framework? Why JSON?**

**Answer:**  
In our framework we primarily use JSON files to store test data. We keep them in a separate `testdata/` folder outside the `tests/` folder for clean separation.

We chose JSON because JavaScript and TypeScript natively support JSON — we can import it directly with a simple import statement, no extra library or parsing needed. It's easy to read, easy to edit, and works well with arrays and objects. For simpler cases we sometimes use arrays directly inside the spec file, but for real projects with large data sets, JSON files are the standard approach.

---

**Q3. How do you implement Data Driven Testing in Playwright? Walk me through the approach.**

**Answer:**  
It's a 3-step process:

1. We create a JSON data file with test records — each record is an object with fields like name, email, password, expected result, etc.
2. In our spec file, we import the JSON file using a simple import statement — `import testdata from "../testdata/registration.json"`. This gives us a JavaScript array in memory.
3. We wrap our test inside a `for...of` loop — it iterates through each record and creates a separate test for each one. We use the record's unique field (like test case ID) in the test name to keep test names unique — Playwright requires unique test names otherwise it throws an error.

When we run `npx playwright test`, Playwright unrolls the loop and treats each iteration as an independent test. In the report, each record appears as a separate test with its own pass/fail status.

---

**Q4. If one test data record fails, do the remaining records still run?**

**Answer:**  
Yes. Each record's test is completely independent. If record 2 fails, record 3, 4, 5 still execute normally. In the report, each record shows as a separate test — the failed one is marked red and the passed ones are marked green. This is one of the biggest advantages of data driven testing — one failure doesn't block the rest.

---

**Q5. Why must test names be unique in a data driven loop? What happens if they're not?**

**Answer:**  
Playwright requires every test in a spec file to have a unique name. If two tests have the same name, Playwright throws an error and refuses to run.

In a data driven loop, since the same `test()` block runs multiple times, we need to make each test name unique by including something from the data — like the test case ID or a combination of fields:

```typescript
test(`Test ${data.id}`, ...)           // ✅ unique — TC001, TC002, TC003
test(`Login test`, ...)                // ❌ all iterations get same name — error
```

**Q6. Explain this line — `import { test, expect } from "@playwright/test"` — why curly braces?**

**Answer:**  
This uses **named imports** — a JavaScript concept. The `@playwright/test` library exports many things by name — `test`, `expect`, `Page`, `Browser`, `Locator`, and more. When we import named exports, we must use curly braces and write the exact name as the library defined it. We can't rename `test` to something else because the library exported it with that specific name.

---

**Q7. Explain this line — `import testdata from "../testdata/registration.json"` — why no curly braces? Can we change the name?**

**Answer:**  
This uses a **default import**. A JSON file exports its entire content as one single thing — it doesn't have multiple named exports. For default imports, we don't use curly braces.

And yes, we can name it anything we want — `testdata`, `registrationData`, `myData`, `records` — all will work because with default imports the name is our choice. The content stays the same regardless of what we call it.

---

**Q8. What is the difference between Named Import and Default Import?**

**Answer:**  
**Named import** uses curly braces, requires the exact name from the library, and is used when a module exports multiple things by name.  
Example: `import { test, expect } from "@playwright/test"`.

**Default import** has no curly braces, we can give any name we want, and is used when a module exports one single thing.  
Example: `import testdata from "../testdata/file.json"`.

Simple way to remember — curly braces mean "pick specific items by name from the shelf." No curly braces mean "take the whole box — call it whatever you want."

---

**Q9. If I modify the imported `testdata` inside my spec file, will it change the original JSON file on disk?**

**Answer:**  
No. When we import a JSON file, JavaScript reads its content and creates an in-memory copy. The variable `testdata` is just a copy in memory — it's not a live connection to the file. Any changes we make to `testdata` in our spec file only affect the runtime copy. The original JSON file on disk remains completely untouched. Once the test execution finishes, the in-memory copy is gone.

---

**Q10. What if your JSON is an object of objects instead of an array? Can you still use `for...of`?**

**Answer:**  
No, not directly. `for...of` works only on iterables like arrays, strings, maps, and sets. Plain JavaScript objects are not iterable.

If the JSON is an object of objects, we need to convert it to an array first using JavaScript built-in methods:

- `Object.entries(testdata)` — gives an array of `[key, value]` pairs
- `Object.values(testdata)` — gives an array of just the values
- `Object.keys(testdata)` — gives an array of just the keys

Then we can use `for...of` on the resulting array. In our framework, we prefer keeping JSON as an array of objects because it works directly with `for...of` without any conversion.

**Q11. What are the different sources you can use for test data in Playwright?**

**Answer:**  
We can use multiple sources:

- **Arrays** — directly inside the spec file. Simple but not scalable, used for quick small tests.
- **JSON files** — most common in Playwright. Native support, no extra library needed, easy to read and maintain.
- **CSV files** — need a parsing library like `csv-parse` or `papaparse`. Used when data comes from Excel exports or business teams provide data in CSV.
- **Excel files** — need a library like `exceljs` or `xlsx`. Used when business teams maintain test data in spreadsheets.
- **Database** — connect via a DB client, fetch records, and use them. Used in enterprise projects where test data lives in a central DB.
- **API response** — fetch data from an API and use it as test data. Useful for dynamic data driven tests.

In real projects, JSON is the go-to choice for most Playwright frameworks.

**Q12. How do you handle data driven tests where some records should pass and some should fail intentionally?**

**Answer:**  
We add an `expected` result field in our test data — like `"expectedResult": "success"` or `"expectedMessage": "Invalid email"`. Then inside the test, we write conditional assertions based on that field:

```json
[
  { "id": "TC001", "email": "valid@test.com", "password": "pass123", "expected": "success" },
  { "id": "TC002", "email": "invalid",        "password": "pass123", "expected": "Invalid email format" },
  { "id": "TC003", "email": "",               "password": "",        "expected": "Fields cannot be empty" }
]
```

```typescript
if (data.expected === "success") {
  await expect(page.locator('.success-msg')).toBeVisible();
} else {
  await expect(page.locator('.error-msg')).toHaveText(data.expected);
}
```

This way positive and negative test scenarios all live in the same data file — each record knows what its expected outcome should be.

**Q13. What are the advantages of Data Driven Testing?**

**Answer:**

- **No code duplication** — one test handles all data variations.
- **Easy to add new test cases** — just add a new record in the JSON file, no code change needed.
- **Data and logic are separated** — test logic is in the spec file, test data is in the JSON file. If logic changes, update the spec. If data changes, update the JSON. Neither affects the other.
- **Independent execution** — each record runs as a separate test. One failure doesn't block others.
- **Clear reporting** — each record appears as a separate test in the report with its own pass/fail status and name.
- **Maintenance is easy** — updating 50 test cases means editing a JSON file, not 50 test methods.
