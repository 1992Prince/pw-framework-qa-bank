# 11.02 — Annotations, Parallel Execution, testInfo & Config

---

## 1. ANNOTATIONS — TEST EXECUTION CONTROL KARNA

Annotations special modifiers hain jo Playwright ko batate hain ki ek specific test ke saath kya karna hai — run karo, skip karo, ya sirf yahi run karo.

### `test.only()` — Sirf Yeh Test Run Karo

```javascript
test.only('only this test runs', async ({ page }) => {
  // yeh test run hoga
});

test('normal test', async ({ page }) => {
  // yeh test SKIP ho jayega jab test.only present ho
});
```

- Jab ek file mein `test.only()` hota hai — sirf wahi test run hota hai, baaki sab tests skip ho jaate hain.
- Local debugging ke liye useful hai — sirf ek specific test isolate karke run karna ho.

> ⚠️ **WARNING — `test.only` kabhi commit mat karo!**
> Agar koi developer accidentally `test.only` push kar de toh CI mein baaki saare tests skip ho jaayenge — silent failure. Isse prevent karne ke liye `playwright.config` mein yeh set karo:
> ```javascript
> forbidOnly: !!process.env.CI
> ```
> CI environment mein `test.only` detect hone pe build fail ho jaayegi — guard lagaa deta hai.

---

### `test.skip()` — Is Test Ko Skip Karo

```javascript
test.skip('this test is skipped', async ({ page }) => {
  // yeh test execute nahi hoga
});
```

- Test skip ho jaata hai — na run hota hai, na fail hota hai.
- Report mein "skipped" show hota hai.
- Kab use karo — jab test kisi known bug ki wajah se fail ho raha ho aur temporarily skip karna ho, ya feature abhi ready nahi ho.

---

### `test.fixme()` — Broken Test Mark Karo

```javascript
test.fixme('known broken test', async ({ page }) => {
  // yeh bhi skip ho jaata hai — lekin "fixme" status ke saath
});
```

- `test.skip` ki tarah hi skip karta hai — lekin intent alag hai.
- `fixme` clearly communicate karta hai ki yeh test broken hai aur fix hona chahiye — tracking ke liye better hai.

---

### Annotations Quick Reference

| Annotation | Kya Karta Hai | Kab Use Karo |
| --- | --- | --- |
| `test.only()` | Sirf yeh test run hoga — baaki skip | Local debugging, isolation |
| `test.skip()` | Yeh test skip ho jaata hai | Temporary skip, WIP feature |
| `test.fixme()` | Skip + "needs fixing" intent | Known bug, broken test |
| `test()` | Normal execution | Har normal test case |

## 2. PARALLEL EXECUTION — TESTS PARALLEL KAISE CHALTE HAIN

### Default Behavior

- Playwright by default tests ko **parallel** mein run karta hai — multiple **worker processes** use karke.
- Har worker ek alag Node.js process hota hai — apna alag browser instance chalata hai.
- Har test ko apna **isolated BrowserContext** milta hai — toh parallel tests ek doosre ko affect nahi karte.

### `fullyParallel: true`

```javascript
// playwright.config.js
export default defineConfig({
  fullyParallel: true,   // ek hi file ke andar ke tests bhi parallel chalenge
});
```

- By default, alag-alag files parallel chalti hain — lekin ek file ke andar tests sequentially chalte hain.
- `fullyParallel: true` set karne se ek file ke andar ke tests bhi parallel ho jaate hain.

### `workers` — Parallel Worker Count

```javascript
export default defineConfig({
  workers: 4,    // 4 parallel processes chalenge
  workers: 1,    // sequential — ek test complete ho phir agla shuru ho
});
```

- `workers: 1` — tests sequentially chalte hain. CI environments mein useful jab tests ek shared resource (database, API) use kar rahe hon aur conflicts aa rahe hon.
- Default — CPU cores ka half automatically set hota hai.

### Isolation Ka Faida

```
Worker 1: BrowserContext A → Test 1 (apne cookies, session)
Worker 2: BrowserContext B → Test 2 (apne cookies, session)
Worker 3: BrowserContext C → Test 3 (apne cookies, session)

Teeno ek saath chal rahe hain — zero interference
```

Har test ka apna fresh context hota hai — ek test ka login state, cookies, ya storage doosre test pe bilkul asar nahi dalta.

## 3. `testInfo` OBJECT — TEST METADATA

Jab test ke andar test ke baare mein information chahiye — `testInfo` use karte hain.

```javascript
test('example', async ({ page }, testInfo) => {
  console.log(testInfo.title);      // test ka naam
  console.log(testInfo.outputDir);  // test output directory ka path
  console.log(testInfo.status);     // 'passed' | 'failed' | 'skipped'
});
```

- `testInfo` test function ka **doosra parameter** hai — `{ page }` ke baad likhte hain.
- Playwright automatically inject karta hai — manually create nahi karna.

### Commonly Used `testInfo` Properties

| Property | Type | Purpose |
| --- | --- | --- |
| `testInfo.title` | `string` | Test ka naam |
| `testInfo.status` | `string` | `'passed'` \| `'failed'` \| `'skipped'` |
| `testInfo.outputDir` | `string` | Test artifacts save karne ki directory |
| `testInfo.retry` | `number` | Yeh kaunwi retry hai — `0` first run |
| `testInfo.duration` | `number` | Test kitni milliseconds mein chala |

### Practical Example — `afterEach` Mein `testInfo` Use Karna

```javascript
test.afterEach(async ({ page }, testInfo) => {
  if (testInfo.status === 'failed') {
    await page.screenshot({ path: `${testInfo.outputDir}/failure.png` });
    console.log(`❌ Test failed: ${testInfo.title}`);
  }
});
```

## 4. PAGE TITLE VERIFY KARNA — DO TARIKE

```javascript
import { test, expect } from '@playwright/test';

test('Verify Playwright page title', async ({ page }) => {
  await page.goto('https://playwright.dev');

  // Tarika 1 — title string nikalo aur assert karo
  const title = await page.title();
  expect(title).toContain('Playwright');

  // Tarika 2 — Direct page assertion (PREFERRED — retry-able hai)
  await expect(page).toHaveTitle(/Playwright/);
});
```

### Dono Tarikon Ka Difference

| | Tarika 1 — `page.title()` | Tarika 2 — `expect(page).toHaveTitle()` |
| --- | --- | --- |
| `await` | `await page.title()` se string milta hai | Assertion pe `await` |
| Retry | ❌ Ek baar check — retry nahi | ✅ Retry karta hai timeout tak |
| Preferred | ❌ Avoid | ✅ Use this |

> **Best Practice** — `expect(page).toHaveTitle()` prefer karo. Yeh retry-able assertion hai — agar page title thodi der se aaye toh bhi wait karke check kar lega. `page.title()` snapshot le leta hai — agar title abhi load nahi hua toh test fail ho sakta hai.

## 5. `use` BLOCK — TEST CONFIGURATION OPTIONS

`playwright.config` mein `use` block global test settings define karta hai — jo har test pe automatically apply hoti hain.

```javascript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  use: {
    baseURL: 'https://yourapp.com',      // relative URLs possible — goto('/login')
    actionTimeout: 10000,                // har action (click, fill) ka max wait — 10s
    navigationTimeout: 30000,            // page.goto() ka max wait — 30s
    headless: true,                      // browser visible nahi hoga
    screenshot: 'only-on-failure',       // fail pe screenshot save karo
    video: 'retain-on-failure',          // fail pe video save karo
    trace: 'on-first-retry',             // pehli retry pe trace record karo

    // Mobile emulation — device preset use karo
    ...devices['iPhone 13'],
  },
});
```

### Har Option Ka Purpose

| Option | Purpose | Recommended Value |
| --- | --- | --- |
| `baseURL` | Relative URLs possible — `goto('/login')` likhna kaafi | App ka base URL |
| `actionTimeout` | Har `.click()`, `.fill()`, `.getBy*()` ka max wait | `10000` (10s) |
| `navigationTimeout` | `page.goto()` aur `page.waitForURL()` ka max wait | `30000` (30s) |
| `headless` | `true` = browser hidden, `false` = browser visible | `true` CI mein |
| `screenshot` | `'on'` \| `'off'` \| `'only-on-failure'` | `'only-on-failure'` |
| `video` | `'on'` \| `'off'` \| `'retain-on-failure'` | `'retain-on-failure'` |
| `trace` | `'on'` \| `'off'` \| `'on-first-retry'` \| `'retain-on-failure'` | `'on-first-retry'` |

### `baseURL` Ka Faida

```javascript
// playwright.config mein baseURL set hai: 'https://yourapp.com'

// Bina baseURL ke
await page.goto('https://yourapp.com/login');   // poora URL likhna padta hai

// baseURL set hone ke baad
await page.goto('/login');                       // sirf path likho — baseURL auto prefix hota hai
await page.goto('/dashboard');
await page.goto('/profile');
```

### Mobile Emulation — `devices` Spread

```javascript
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  projects: [
    {
      name: 'iPhone 13',
      use: { ...devices['iPhone 13'] },  // viewport, UA, touch sab auto-set
    },
    {
      name: 'Pixel 5',
      use: { ...devices['Pixel 5'] },
    },
  ],
});
```

- `devices['iPhone 13']` ek preset object hai jisme viewport size, user agent string, `isMobile: true`, `hasTouch: true` sab pre-configured hain.
- `...devices['iPhone 13']` spread operator se yeh sab values `use` block mein merge ho jaati hain.
