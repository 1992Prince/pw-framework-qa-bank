# Playwright Setup — Interview Q&A (Node.js, npm, npx)

---

**Q1. How do you get Node version, npm version, npx version, Playwright version?**

We use simple commands in the terminal:

```bash
node -v               # gives Node.js version
npm -v                # gives npm version
npx -v                # gives npx version
npx playwright --version  # gives Playwright version
```

We can also check the Playwright version inside `package.json` under the `devDependencies` section — it shows which version is installed in the project.

---

**Q2. How do you initialize a fresh Node.js project?**

We run the command `npm init` in the terminal. This creates a `package.json` file in our folder, which is basically the identity card of the project. It asks a few questions like project name, version, description, author, license, etc. If we want to skip all questions and create it instantly with defaults, we use `npm init -y`. Without `package.json`, npm doesn't recognize the folder as a project and we can't install any libraries.

---

**Q3. How do you initialize a fresh Playwright project? What steps do you follow?**

First, we make sure Node.js v18 or above is installed — because Playwright is built on Node.js. Then we follow these steps:

1. Create a new folder and navigate into it.
2. Run `npm init -y` to initialize the Node.js project and create `package.json`.
3. Run `npm init playwright@latest` — this installs Playwright and runs an interactive wizard that asks: TypeScript or JavaScript? Test folder name? Add GitHub Actions? Install browsers?
4. If we say yes to browsers, it automatically downloads Chromium, Firefox, and WebKit.
5. Finally, run `npx playwright test` to verify everything is working.

That's the complete setup — after these steps the project is ready to write and run tests.

---

**Q4. Explain the folder structure generated and purpose of each file.**

When we run `npm init playwright@latest`, it asks a few setup questions and then generates the following folder structure:

- `node_modules/` — where all installed libraries physically live — Playwright and all its dependencies. We never edit this folder manually and never push it to Git. It can be regenerated anytime by running `npm install`.
- `tests/example.spec.ts` — a ready-made sample test file that Playwright generates so we can see how a basic test looks.
- `tests-examples/` — contains more detailed demo tests, like a todo app example, for learning purposes.
- `package.json` — the project's ID card. It holds the project name, version, scripts, and the list of all dependencies with their versions.
- `package-lock.json` — locks the exact version of every installed package. This ensures every teammate and CI server gets the same versions. We never edit this manually.
- `playwright.config.ts` — the main Playwright configuration file. We configure browsers, timeouts, retries, reporters, base URL, parallel execution, and more here.
- `.github/workflows/playwright.yml` — optional. If we chose GitHub Actions during setup, this file gets created for CI/CD — it runs our tests automatically on every push or PR.

---

**Q5. What is the purpose of the `npm init` command?**

The purpose of `npm init` is to initialize a new Node.js project by creating a `package.json` file. Without this file, our folder is just a random folder — npm doesn't recognize it as a project. We can't install libraries, we can't define scripts, and we can't share the project properly with teammates. So `npm init` basically converts a regular folder into a proper Node.js project. We use `npm init -y` if we want to skip the questions and create it with default values instantly.

---

**Q6. What is the difference between npm and npx? When to use what?**

- **npm** stands for Node Package Manager — its job is to install and download packages into our project. For example, `npm install -D @playwright/test` downloads the Playwright library and saves it in `node_modules`.
- **npx** stands for Node Package Executor — its job is to run and execute commands from packages that are already installed. For example, `npx playwright test` runs our Playwright tests.

In simple terms — **npm is the installer, npx is the executor**. npm brings the library into the project, npx uses it. They are partners, not substitutes — we can't use one in place of the other. `npm playwright test` won't work because npm doesn't know how to run tests. `npx install -D @playwright/test` won't work because npx doesn't install packages as dependencies.

---

**Q7. Why do we run Playwright commands with npx and not npm? Can we run Playwright commands with npm?**

Because Playwright provides CLI commands like `playwright test`, `playwright install`, `playwright codegen` — these are commands that need to be **executed**, not installed again. Since Playwright is already installed inside `node_modules`, we use npx to find and run these commands directly from there.

If we try to run them with npm, it would try to install them again instead of executing them, which is not what we want. So no, we cannot run Playwright CLI commands with npm — we always use npx for execution.

---

**Q8. Where are Playwright libraries installed? Explain the `node_modules` folder.**

When we install Playwright using npm, the library files get stored inside the `node_modules/` folder at the project root. This folder contains not just Playwright but also all its internal sub-dependencies like `playwright-core` and others.

This folder can be very large — thousands of files and folders. We never edit it manually, and we never push it to Git — we add it to `.gitignore`. If someone clones the project, they just run `npm install` and this folder gets regenerated from `package.json`.

One important point — **browser binaries are NOT stored in `node_modules`**. When we run `npx playwright install`, the browsers get downloaded to a global cache on the system:
- **Windows:** `C:\Users\<user>\AppData\Local\ms-playwright\`
- **Mac:** `~/Library/Caches/ms-playwright/`
- **Linux:** `~/.cache/ms-playwright/`

---

**Q9. `npm init playwright@latest` vs `npm install -D @playwright/test` — Purpose of each with scenario.**

These two commands serve different purposes:

- **`npm init playwright@latest`** — for fresh project setup from scratch. It installs Playwright, runs the interactive wizard, creates the `tests/` folder, generates `playwright.config.ts`, adds sample test files, and optionally installs browsers. We use this when starting a brand new automation project.
- **`npm install -D @playwright/test`** — only installs the Playwright library. No wizard, no folders, no config file, no sample tests. We use this when we already have an existing project and just need to add the Playwright library — maybe it's missing, broken, or needs to be updated. For example, a teammate gives us a project where Playwright is listed in `package.json` but `node_modules` is not there — we run this command to install just the library.

---

**Q10. Explain `-D`.**

`-D` is the short form of `--save-dev`. When we add `-D` with `npm install`, the library gets installed under `devDependencies` in `package.json` instead of `dependencies`.

`devDependencies` are libraries needed only during development and testing — they are not needed in production. Playwright is a testing tool — real users and production servers don't need it. So we always install it with `-D`.

Without `-D`, the library goes under `dependencies`, which means it's needed to run the application in production — like Express, React, or database drivers.

Simple rule — ask yourself: *"Does the production server need this library?"* If no, use `-D`.

---

**Q11. Where can we see the Playwright runner version in the framework?**

Two places. First, inside `package.json` under the `devDependencies` section — it shows the version like `"@playwright/test": "^1.48.0"`. Second, by running `npx playwright --version` in the terminal — it gives the exact installed version. We can also check `package-lock.json` for the precise locked version.

---

**Q12. `package.json` vs `package-lock.json` — Purpose of each with scenario.**

- `package.json` is the project's ID card — it lists all the libraries our project needs with **flexible version ranges**. For example, `"^1.48.0"` means any compatible version from 1.48.0 onwards.
- `package-lock.json` locks down the **exact version** that was actually installed. For example, `1.48.3` — no flexibility.

Why does this matter? Let's say I install Playwright today and get version `1.48.3`. My teammate installs it next month — by then `1.48.9` might be available. Since `package.json` says `^1.48.0`, npm would install `1.48.9` for them — a different version than mine. This can cause "works on my machine" bugs.

`package-lock.json` solves this — it locks the version to `1.48.3`, so when my teammate runs `npm install`, they get exactly `1.48.3` too. Same versions everywhere — my machine, their machine, CI server.

Both files are pushed to Git. `node_modules` is not.

---

**Q13. `npx playwright install` vs `npx playwright install --with-deps` — What is `--with-deps` and when to use it?**

- `npx playwright install` — downloads the browser binaries (Chromium, Firefox, and WebKit).
- `npx playwright install --with-deps` — does the same thing **plus** installs additional OS-level system libraries that the browsers need to run — things like fonts, graphics libraries, audio/video libraries.

We use `--with-deps` mainly on **Linux CI servers** like GitHub Actions or Jenkins running on Linux. These servers are bare-bones machines that often don't have the system libraries browsers need. Without `--with-deps`, browsers might fail to launch with "missing library" errors.

On Windows or Mac for local development, we generally don't need `--with-deps`.

---

**Q14. Does `npm init playwright@latest` automatically run `npx playwright install`?**

Yes, but it depends on what we choose during setup. The wizard asks: *"Install Playwright browsers?"* If we say yes, it automatically runs `npx playwright install` and downloads all three browsers. If we say no, we'll need to run `npx playwright install` manually later.

---

**Q15. Can we run `npx playwright install` manually? When?**

Yes, absolutely. We run it manually in these situations:

- We skipped browser installation during the initial setup.
- We updated Playwright to a new version — new versions may need new browser binaries.
- A new teammate cloned the repo — `npm install` brings `node_modules` but not browser binaries because they live outside the project in a global cache. So they must run `npx playwright install` separately.
- Browser binaries got corrupted or accidentally deleted.

---

**Q16. `npx playwright install` vs `npx playwright install chromium`**

- `npx playwright install` without specifying a browser installs **all three browsers** — Chromium, Firefox, and WebKit. This takes more time and disk space but is needed when we do cross-browser testing.
- `npx playwright install chromium` installs **only Chromium**. It's faster and lighter. We use this when our tests only need to run on Chrome-based browsers.
- We can also install specific combinations: `npx playwright install chromium firefox` — only those two, no WebKit.

---

**Q17. When do we go with `npm install -D @playwright/test`?**

We use this command when we already have an existing Node.js project and we just need the Playwright library — not the full setup. Common scenarios:

- The project exists but Playwright is missing or not working.
- We want to update Playwright to a newer version.
- We're adding Playwright to a larger project that already has its own folder structure and config — we don't want the wizard to overwrite things.
- A teammate cloned the repo and needs to install all dependencies — they run `npm install` which reads `package.json` and installs everything listed there.

---

**Q18. How can you install all browsers manually, and only specific browsers like Chromium or Firefox or WebKit?**

To install all browsers:

```bash
npx playwright install
```

To install only specific ones:

```bash
npx playwright install chromium        # only Chromium
npx playwright install firefox         # only Firefox
npx playwright install webkit          # only WebKit
npx playwright install chromium firefox   # Chromium and Firefox, no WebKit
```

We pick specific browsers when we don't need all three — saves download time and disk space.

---

**Q19. After running `npx playwright install` — will stable browser apps install or browser engines? What is their path?**

Playwright does **not** install the regular stable browsers we use daily — like the Chrome or Firefox on our desktop. Instead, it downloads its own **patched browser engines** — custom-built versions of Chromium, Firefox (patched), and WebKit that Playwright has modified to support automation through CDP (Chrome DevTools Protocol).

These patched browsers are separate from our regular browsers — they don't interfere with them.

They get stored in a global cache, not inside the project:
- **Windows:** `C:\Users\<user>\AppData\Local\ms-playwright\`
- **Mac:** `~/Library/Caches/ms-playwright/`
- **Linux:** `~/.cache/ms-playwright/`
