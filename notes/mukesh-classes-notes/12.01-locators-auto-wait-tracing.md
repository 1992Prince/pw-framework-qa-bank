# 11.03 — Locators, Auto-Wait, Actionability & Tracing

---

## 1. LOCATORS KYA HOTE HAIN?

> Reference: [playwright.dev/docs/locators](https://playwright.dev/docs/locators)

Locators Playwright ka core mechanism hai webpage pe elements dhundhne aur unke saath interact karne ka.

Locator use karke tum:
- Kisi bhi element ko webpage pe locate kar sakte ho
- Locate hone ke baad us element pe koi bhi operation kar sakte ho — click, fill, type, etc.

### Locators Ki Do Built-in Superpowers

1. **Auto-wait** — action perform karne se pehle element ke ready hone ka automatically wait karta hai
2. **Actionability Checks** — yeh verify karta hai ki element visible hai, enabled hai, aur stable hai — tab action perform karta hai

---

## 2. PLAYWRIGHT KE BUILT-IN LOCATOR METHODS

| Locator Method | Kab Use Karo | Kaise Kaam Karta Hai |
| --- | --- | --- |
| `page.getByRole()` | Best practice — always prefer | DOM **accessibility tree** pe kaam karta hai — raw DOM pe nahi |
| `page.getByText()` | Visible text se dhundho | DOM accessibility tree pe kaam karta hai |
| `page.getByLabel()` | Form fields ke liye — label se dhundho | DOM accessibility tree pe kaam karta hai |
| `page.getByTestId()` | `data-testid` attribute se dhundho | DOM attribute se locate karta hai |
| `page.getByPlaceholder()` | Input field ka placeholder text se | DOM attribute se |
| `page.locator()` | CSS selector ya XPath — last resort | Raw DOM tree pe kaam karta hai |

### Semantic Locators kyun prefer kare?

- `getByRole()`, `getByText()`, `getByLabel()` — yeh **semantic locators** hain — accessibility tree pe kaam karte hain.
- Agar developer UI update kare (CSS class change kare, div restructure kare) — semantic locators tab bhi kaam karte hain kyunki element ka accessible role/name change nahi hota.
- `page.locator()` CSS/XPath use karta hai — DOM tree structure change hone pe yeh **brittle (fragile)** ho jaate hain — test break ho sakta hai.

> **Best Practice** — Priority order: `getByRole` → `getByLabel` → `getByText` → `getByTestId` → `page.locator()` (CSS/XPath last resort)

---

## 3. LOCATORS KAISE USE KARTE HAIN

### Step 1 — Element Locate Karo
### Step 2 — Us Pe Action Perform Karo

```javascript
// Step 1 — element locate karo (await NOT needed)
const ele = page.getByLabel("Name");

// Step 2 — action perform karo (await IS needed)
await ele.fill("John");        // input mein value set karo (fastest)
await ele.click();             // element pe click karo
await ele.clear();             // field clear karo
await ele.type("hello");       // character by character type karo
await ele.selectOption("val"); // dropdown se value select karo
```

### Important — `await` Kab Chahiye?

```javascript
// Locator object banane mein → await NAHI chahiye
const ele = page.getByLabel("Name");   // ✅ no await — returns Locator object, not Promise

// Action perform karne mein → await CHAHIYE
await ele.click();     // ✅ await required — action ek Promise return karta hai
await ele.fill("abc"); // ✅ await required
```

- `page.getByLabel()` — **Locator object** return karta hai — Promise nahi. Isliye `await` nahi chahiye locator banate waqt.
- Locator pe `.click()`, `.fill()` etc. perform karne pe — woh **Promise** return karte hain — isliye `await` chahiye.

## 4. WRONG LOCATOR DE DO — KYA HOGA?

```javascript
// "Name2" label exist nahi karta page pe
await page.getByLabel("Name2").click();
//                     ↑
//           galat locator — page pe nahi hai
```

**Kya hoga:**
1. Playwright element dhundega — nahi milega
2. Playwright **retry karta rahega** — baar baar dhundega
3. `actionTimeout` expire hone tak wait karega (default: 30 seconds)
4. Timeout ke baad → **Timeout Error** throw karega

```
TimeoutError: Timeout 30000ms exceeded.
waiting for getByLabel("Name2")
```

Yeh Playwright ka **auto-wait + auto-retry** behavior hai — immediately error nahi deta, timeout tak try karta rehta hai.

## 5. AUTO-WAIT KE DO TYPES ⭐

Playwright mein do alag auto-wait mechanisms hain:

### Type 1 — Actions Pe Auto-Wait (`click`, `fill`, etc.)

```javascript
// Playwright wait karega element ke actionable hone tak
await page.getByLabel("Name").click();

// Method level timeout override
await page.getByLabel("Name").click({ timeout: 5000 }); // sirf is action ke liye 5 sec
```

- Default timeout: **`actionTimeout`** config se aata hai (default 30 seconds)
- Config level pe ya individual method level pe override kar sakte ho

### Type 2 — `expect()` Assertions Pe Auto-Wait (Web-First Assertions)

```javascript
// expect() bhi retry karta hai — element visible hone tak wait karta hai
await expect(page.getByLabel("Name")).toBeVisible();
```

- Default timeout: **5 seconds** (action timeout se alag)
- Ise **Web-First Assertions** kehte hain — yeh assertion pass hone tak retry karti hain

### Summary

| Auto-Wait Type | Default Timeout | Override Kaise Kare |
| --- | --- | --- |
| Action auto-wait (`click`, `fill`) | `actionTimeout` (30s default) | `{ timeout: 5000 }` method mein |
| `expect()` auto-wait | 5 seconds | `expect(locator, { timeout: 10000 })` |

## 6. ACTIONABILITY CHECKS — PLAYWRIGHT KYA CHECK KARTA HAI

Action perform karne se pehle Playwright automatically yeh checks karta hai:

| Check | Matlab |
| --- | --- |
| **Visible** | Element screen pe dikhna chahiye — hidden nahi hona chahiye |
| **Enabled** | Element disabled nahi hona chahiye — disabled input pe click nahi kar sakta |
| **Stable** | Element animate nahi ho raha ho — CSS animation khatam honi chahiye |
| **Receives Events** | Koi aur element uske upar nahi hona chahiye (overlay/modal block nahi kar raha) |
| **Attached** | Element DOM mein attached hona chahiye |

Yeh saari checks **automatically** hoti hain — hume code mein kuch likhna nahi padta.

> Reference: [playwright.dev/docs/actionability](https://playwright.dev/docs/actionability)

### `force: true` — Actionability Checks Bypass Karo

```javascript
// force: true — actionability checks skip ho jaati hain
await page.getByLabel("Name").click({ force: true });
```

> ⚠️ Caution — `force: true` sirf tab use karo jab zaroorat ho. Visibility/enabled checks bypass ho jaate hain — real-world failures miss ho sakti hain.

## 7. `fill()` vs `type()` vs `pressSequentially()`

Teen tarike hain text input karne ke:

```javascript
// fill() — RECOMMENDED — fastest, field clear karke value set karta hai
await page.getByLabel("Name").fill("John");

// type() — character by character (newer Playwright versions mein deprecated)
await page.getByLabel("Name").type("John");

// pressSequentially() — character by character with configurable delay
await page.getByLabel("Name").pressSequentially("John", { delay: 500 });
// delay: 500 → har character ke baad 500ms ruko
```

### Kab Kya Use Karo

| Method | Speed | Behavior | Kab Use Karo |
| --- | --- | --- | --- |
| `fill()` | ⚡ Fastest | Field clear karta hai, phir poori value set | Default — almost always |
| `type()` | 🐢 Slow | Har character alag-alag type karta hai | Deprecated — avoid |
| `pressSequentially()` | 🐢 Slow | Har character alag + configurable delay | Jab real keyboard typing simulate karni ho (OTP fields, autocomplete) |

> **Best Practice** — `fill()` use karo by default. `pressSequentially()` sirf tab use karo jab field keyboard events pe depend kare (jaise OTP input ya autocomplete field).

## 8. STRICT MODE — EK SE ZYADA ELEMENTS MATCH HONE PE

Playwright strict mode mein kaam karta hai — agar locator **ek se zyada element** match kare toh error throw karta hai.

```javascript
// Agar page pe 2 "Sign Up" text wale elements hain
await page.getByText("Sign Up").click();
// Error: strict mode violation — getByText("Sign Up") resolved to 2 elements
```

**Strict mode isliye hai** — taaki accidentally wrong element pe action na ho jaye.

### Solution — Specific Element Target Karo

```javascript
// .first() — pehla matching element
await page.getByText("Sign Up").first().click();

// .last() — aakhri matching element
await page.getByText("Sign Up").last().click();

// .nth(index) — specific index wala (0-based)
await page.getByText("Sign Up").nth(1).click();

// Ya zyada specific locator use karo
await page.getByRole('button', { name: 'Sign Up' }).click();
```

## 9. DROPDOWN — `selectOption()`

```javascript
// Select by value attribute
await page.getByLabel("Country").selectOption("India");

// Select by visible text (label)
await page.getByLabel("Country").selectOption({ label: "Karnataka" });

// Select by value attribute explicitly
await page.getByLabel("Country").selectOption({ value: "karnataka" });

// Multiple select — array pass karo
await page.getByLabel("Colors").selectOption(["red", "green", "blue"]);
```

> **Note** — `selectOption()` sirf `<select>` HTML element pe kaam karta hai. Custom dropdown components (div-based) ke liye `click()` + `getByText()` approach use karo.

## 10. `isVisible()` AUR `isEnabled()` — STATE CHECK KARNA

```javascript
// isVisible() — element visible hai ya nahi — true/false return karta hai
const visible = await page.getByRole('button', { name: 'Submit' }).isVisible();
console.log(visible); // true ya false

// isEnabled() — element enabled hai ya nahi — true/false return karta hai
const enabled = await page.getByRole('button', { name: 'Submit' }).isEnabled();
console.log(enabled); // true ya false

// Conditional logic mein use karo
if (await page.getByRole('button', { name: 'Submit' }).isVisible()) {
  await page.getByRole('button', { name: 'Submit' }).click();
}
```

> **`isVisible()` vs `expect().toBeVisible()` ka difference:**
> - `isVisible()` — instant check, `true`/`false` return karta hai, retry nahi karta
> - `expect(locator).toBeVisible()` — retry-able assertion, 5 seconds tak wait karta hai — test assertions mein yeh prefer karo

## 11. `await` KYUN CHAHIYE — HAR ACTION PE

Browser pe kiye jaane wale saare Playwright actions **Promises** return karte hain — yeh asynchronous operations hain.

```javascript
// GALAT — await nahi hai — unreliable test
page.click('button');                       // click shuru hua — lekin khatam hone ka wait nahi
expect(page.locator('.result')).toBeVisible(); // click complete hone se pehle yeh run ho sakta hai!

// SAHI — await hai — reliable test
await page.click('button');                       // click poora hone ka wait karo
await expect(page.locator('.result')).toBeVisible(); // tab assertion run karo
```

**`await` ke bina:**
- JavaScript next line pe move ho jaata hai — action complete hue bina
- Test unreliable ho jaata hai — intermittent failures

**`await` ke saath:**
- JavaScript wait karta hai — current action poora hone tak
- Tab agla step run hota hai — sequential, reliable execution

## 12. SCREENSHOTS, VIDEO, TRACE — CONFIG

`playwright.config` ke `use` block mein configure karo:

```javascript
use: {
  screenshot: 'only-on-failure',   // fail pe screenshot lo
  video: 'retain-on-failure',      // fail pe video rakho
  trace: 'on-first-retry',         // pehli retry pe trace record karo
},
```

### Screenshot Options

| Value | Behavior |
| --- | --- |
| `'on'` | Har test ke liye hamesha screenshot |
| `'off'` | Kabhi screenshot nahi |
| `'only-on-failure'` | Sirf fail hone pe ✅ Recommended |

### Video Options

| Value | Behavior |
| --- | --- |
| `'on'` | Hamesha video record karo |
| `'off'` | Kabhi nahi |
| `'retain-on-failure'` | Fail hone pe video rakho ✅ Recommended |
| `'on-first-retry'` | Pehli retry pe record karo |

### Trace Options

| Value | Behavior |
| --- | --- |
| `'on'` | Hamesha trace record karo |
| `'off'` | Kabhi nahi |
| `'on-first-retry'` | Pehli retry pe trace record karo ✅ CI ke liye recommended |
| `'retain-on-failure'` | Fail hone pe trace rakho |

### CI Recommended Combination

```javascript
export default defineConfig({
  retries: 2,
  use: {
    screenshot: 'only-on-failure',
    video: 'retain-on-failure',
    trace: 'on-first-retry',    // test fail → retry hoga → trace milegi
  },
});
```

## 13. TRACE VIEWER — POWERFUL DEBUGGING TOOL ⭐

Trace Playwright ka **sabse powerful post-mortem debugging tool** hai. Test run hone ke baad trace file open karke **har action ke baare mein sab kuch dekh sakte ho**.

### Trace File Mein Kya Hota Hai

- Har action ka **pehle aur baad ka screenshot** (DOM snapshot)
- **Network requests aur responses**
- **Browser console logs**
- **Source code** — kaunsi line execute ho rahi thi
- Errors, attachments — sab ek jagah

### Trace Kaise Open Kare

```bash
# Tarika 1 — locally
npx playwright show-trace trace.zip

# Tarika 2 — online (zip upload karo)
# https://trace.playwright.dev
```

### Trace Viewer Tabs

Har action select karne pe yeh tabs available hote hain:

| Tab | Kya Dikhata Hai |
| --- | --- |
| **Locator** | Kaunsa locator use hua |
| **Call** | Action details — arguments, return value |
| **Log** | Internal Playwright logs |
| **Errors** | Error message agar action fail hua |
| **Console** | Browser console output |
| **Network** | Network requests during that action |
| **Source** | Source code ki line jo execute ho rahi thi |
| **Attachments** | Screenshots, videos agar attached hain |

> Trace sirf failed tests ke liye rakha jaata hai (`retain-on-failure`) ya retry pe (`on-first-retry`). Successful tests ka trace by default save nahi hota.
