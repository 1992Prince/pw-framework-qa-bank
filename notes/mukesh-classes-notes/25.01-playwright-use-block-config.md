# Playwright Configuration — `use` Block in `playwright.config.ts`

## What is the `use` block?

The `use` block in the Playwright config file is a global configuration object that applies browser-level settings to all tests defined under the `testDir` property. It lets you configure things like base URL, geolocation, credentials, download behavior, etc. in one place.

```js
// playwright.config.ts
use: {
  baseURL: "https://example.com",
  geolocation: { latitude: 12.345, longitude: 41.889 },
  acceptDownloads: true,
  httpCredentials: {
    username: "admin",
    password: "admin"
  }
}
```

**Common `use` properties:** `baseURL`, `headless`, `viewport`, `ignoreHTTPSErrors`, `geolocation`, `permissions`, `locale`, `timezoneId`, `storageState`, `httpCredentials`, `acceptDownloads`, `trace`, `screenshot`, `video`, `extraHTTPHeaders`, `colorScheme`, etc.

---

## Overriding at the Project Level

If browser settings vary by environment or application (e.g., staging vs production, desktop vs mobile), you can override `use` properties inside the `projects` array. Each project can have its own `use` block, `testDir`, browser, etc.

**Scenario:** You have a web app that needs testing on both desktop Chrome and mobile Safari, each pointing to a different base URL.

```js
// playwright.config.ts
projects: [
  {
    name: "Desktop Chrome - Staging",
    use: {
      browserName: "chromium",
      baseURL: "https://staging.example.com",
      viewport: { width: 1280, height: 720 }
    },
    testDir: "./tests/desktop"
  },
  {
    name: "Mobile Safari - Production",
    use: {
      ...devices["iPhone 13"],
      baseURL: "https://www.example.com"
    },
    testDir: "./tests/mobile"
  }
]
```

Here each project can point to a different folder of specs, use different browsers, viewports, or base URLs.

---

## Overriding at the Spec File Level — `test.use()`

If you need to change browser settings only for all tests within a specific spec file (not globally and not worth creating a separate project), use `test.use()` at the top of the spec file.

```js
import { test, expect } from '@playwright/test';

test.use({
  ignoreHTTPSErrors: true,
  acceptDownloads: true,
  locale: 'en-US',
  baseURL: "https://freelance-learn-automation.vercel.app"
});

test("Verify test url", async ({ page }) => {
  await page.goto("/"); // uses the baseURL from test.use(), NOT from config
});
```

Every test in this spec file will use these overridden settings. Properties not overridden here will still fall back to the project or global `use` block.

- **Syntax:** `test.use({ ...options })` — called at the top level of a spec file (not inside a test).
- **Purpose:** Override specific browser-level configurations at the spec file level without modifying global config or creating a new project.
- **What can be overridden:** Any property that the global `use` block supports — `baseURL`, `viewport`, `locale`, `geolocation`, `permissions`, `storageState`, `httpCredentials`, `ignoreHTTPSErrors`, `acceptDownloads`, `colorScheme`, `timezoneId`, etc.

---

## Configuration Priority *(Important for Interviews)*

```
spec file test.use()  >  project use block  >  global use block
```

The most specific level wins. If a property is set at the spec level, it takes precedence over the same property in the project or global config. If not set at the spec level, it falls back to the project level, then the global level.
