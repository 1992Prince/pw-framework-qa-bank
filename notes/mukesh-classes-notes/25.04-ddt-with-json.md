# Data Driven Testing (DDT) with JSON in Playwright

## What is Data Driven Testing?

Data Driven Testing is an approach where we run the same test multiple times with different sets of data. Instead of writing separate test cases for each data combination, we write one test and feed it different data records — the test logic stays the same, only the data changes.

---

## Why Do We Need It?

Imagine testing a registration form — we need to validate it with valid data, invalid emails, blank fields, special characters, long values, etc. Without DDT, we'd write 10 separate tests with the same steps just different values. With DDT, we write one test and provide 10 data records — the framework runs the test 10 times automatically, once for each record.

---

## How It Works

- Each row/record in the data = one test execution.
- Each column/field in the data = a parameter (like name, email, password).
- If 10 records exist, the same test runs 10 times — each time picking the next record.
- All 10 runs are independent — if record 2 fails, record 3 still runs.
- In the test report, each record shows as a separate test with its own pass/fail status.

---

## Where Can We Store Test Data?

Arrays, JSON files, CSV files, Excel files, databases, etc. In real projects, JSON files are the most common in Playwright because JavaScript/TypeScript natively supports JSON — no extra libraries needed to read it.

---

## Folder Structure

```
project-root/
├── testdata/
│   └── registration.json       ← test data file
├── tests/
│   └── registration.spec.ts    ← test file
├── package.json
└── playwright.config.ts
```

---

## Step 1 — Create the JSON Data File

`testdata/registration.json`

```json
[
  { "id": "TC001", "name": "Sarika", "email": "sarika@example.com", "password": "password123" },
  { "id": "TC002", "name": "John",   "email": "john@example.com",   "password": "password456" },
  { "id": "TC003", "name": "Alice",  "email": "alice@example.com",  "password": "password789" }
]
```

This is a JSON array containing 3 objects (records). Each object has 4 fields — id, name, email, password. So this file will produce 3 test runs.

---

## Step 2 — Write the Test

`tests/registration.spec.ts`

```typescript
import { test, expect } from "@playwright/test";
import testdata from "../testdata/registration.json";

for (const data of testdata) {
  test(`Test ${data.id}`, async ({ page }) => {
    await page.goto("https://gauravkhurana.com/practise-api/ui/index.html#/practice");

    await page.getByTestId("form-email").fill(data.email);
    await page.getByTestId("form-password").fill(data.password);
    await page.getByTestId("form-confirm-password").fill(data.name);

    if (data.id === 'TC001') {
      await expect(data.id).toBe("123");  // this will FAIL intentionally for TC001
    }

    await page.waitForTimeout(3000);
  });
}
```

---

## Step 3 — What Happens When We Run This?

```bash
npx playwright test
```

Playwright sees the `for` loop, unrolls it into 3 separate tests:

| Test Name  | Data Used                               | Result                                                          |
| ---------- | --------------------------------------- | --------------------------------------------------------------- |
| Test TC001 | Sarika, sarika@example.com, password123 | ❌ Fail (intentional — `expect("TC001").toBe("123")` fails)     |
| Test TC002 | John, john@example.com, password456     | ✅ Pass                                                          |
| Test TC003 | Alice, alice@example.com, password789   | ✅ Pass                                                          |

In the report, each shows as an independent test — TC001 failing does not affect TC002 or TC003.

## Code Explanation — Line by Line

### Line 1: `import { test, expect } from "@playwright/test";`

This line imports two specific things — `test` and `expect` — from the Playwright library.

**Why curly braces `{ }`?** This is a JavaScript concept called **named import**. The `@playwright/test` library exports many things — `test`, `expect`, `Page`, `Browser`, `Locator`, and more. Each of these has a specific name (they are "named exports"). When we import named exports, we must use curly braces and write the exact name as defined in the library:

```typescript
import { test, expect } from "@playwright/test";         // importing 2 named exports
import { test, expect, Page } from "@playwright/test";   // can import more if needed
```

We cannot change the name `test` to `myTest` or `expect` to `check` — it won't find them because the library exported them with those exact names. (JavaScript does allow aliasing with `as` keyword like `import { test as myTest }` but the original name must match first.)

---

### Line 2: `import testdata from "../testdata/registration.json";`

This line imports the entire content of the JSON file.

**Why NO curly braces?** This is a JavaScript concept called **default import**. A JSON file doesn't have multiple named things inside it — it's just one block of data (the array). When something is exported as a single default, we import it without curly braces.

**Can we change the name `testdata`?** Yes, absolutely. With default imports, the name is our choice — we can call it anything:

```typescript
import testdata from "../testdata/registration.json";        // ✅ works
import registrationData from "../testdata/registration.json"; // ✅ works — same data, different name
import myData from "../testdata/registration.json";           // ✅ works
import xyz from "../testdata/registration.json";              // ✅ works (but bad naming)
```

All of these import the same JSON content — only the variable name we use in our code changes.

---

### What is `testdata` — Array or Object?

It depends on what's in the JSON file. Our JSON file starts with `[` and ends with `]` — so it's an array of objects. After import, `testdata` is a regular JavaScript array:

```typescript
testdata[0]  →  { id: "TC001", name: "Sarika", email: "sarika@example.com", password: "password123" }
testdata[1]  →  { id: "TC002", name: "John", ... }
testdata.length  →  3
```

We can use all array methods on it — `for...of`, `.forEach()`, `.map()`, `.filter()`, etc.

---

### Named Import vs Default Import — Summary

| Feature        | Named Import                                      | Default Import                               |
| -------------- | ------------------------------------------------- | -------------------------------------------- |
| Syntax         | `import { test, expect } from "..."`              | `import testdata from "..."`                 |
| Curly braces   | Yes — required                                    | No                                           |
| Can rename?    | No (must match exact export name)                 | Yes — any name works                         |
| When used      | Library exports multiple things by name           | File exports one single thing (like JSON)    |
| Example        | `@playwright/test` exports `test`, `expect`, etc. | JSON file exports one array/object           |

---

## Important Concept: Does Modifying `testdata` in the Spec File Change the JSON File?

**No.** When we import a JSON file, JavaScript reads the file content and creates a copy in memory. The variable `testdata` is an in-memory copy — it is not a live link to the file on disk.

```typescript
import testdata from "../testdata/registration.json";

testdata[0].name = "Changed Name";   // this changes the in-memory copy only
// registration.json on disk is UNTOUCHED — still says "Sarika"
```

So any modification we make to `testdata` inside our spec file only affects the runtime copy in memory. The original JSON file on disk never gets modified. Once the test execution finishes, the in-memory copy is gone.

**Is this good practice — using the imported data directly?** Yes, it's perfectly fine for reading data. Since the JSON file doesn't get modified, there's no risk. However, if we want to be extra safe or need to modify data per test without affecting other tests, we can create a deep copy:

```typescript
const freshData = JSON.parse(JSON.stringify(testdata));  // deep copy — completely independent
```

This way even if one test modifies `freshData`, the original `testdata` and other test iterations remain unaffected.

## Bonus: What If JSON Is an Object of Objects Instead of an Array?

If we change the JSON from an array `[...]` to an object of objects `{...}`:

```json
{
  "TC001": { "name": "Sarika", "email": "sarika@example.com", "password": "password123" },
  "TC002": { "name": "John",   "email": "john@example.com",   "password": "password456" },
  "TC003": { "name": "Alice",  "email": "alice@example.com",  "password": "password789" }
}
```

Import stays the same — no curly braces because it's still a default import:

```typescript
import testdata from "../testdata/registration.json";
```

But `for...of` loop will **NOT** work — because `for...of` works only on iterables (arrays, strings, maps, sets). Plain objects are not iterable.

**How to loop through it:**

```typescript
// Option 1 — Object.entries() gives [key, value] pairs as an array
for (const [id, data] of Object.entries(testdata)) {
  test(`Test ${id}`, async ({ page }) => {
    await page.getByTestId("form-email").fill(data.email);
    // ...
  });
}

// Option 2 — Object.values() gives just the values as an array
for (const data of Object.values(testdata)) {
  test(`Test ${data.name}`, async ({ page }) => {
    // ...
  });
}

// Option 3 — Object.keys() gives just the keys as an array
for (const key of Object.keys(testdata)) {
  const data = testdata[key];
  test(`Test ${key}`, async ({ page }) => {
    // ...
  });
}
```

`Object.entries()`, `Object.values()`, and `Object.keys()` convert the object into an array — and then `for...of` works on that array.

---

## Summary

- **Data Driven Testing** = one test, multiple data records — test runs once per record, all independent.
- **JSON file** is the most common data source in Playwright — no extra libraries needed.
- `import { test, expect }` uses curly braces because these are **named exports** from the library — exact names required.
- `import testdata from "...json"` has **no curly braces** because JSON is a **default export** — name can be anything we choose.
- `testdata` is whatever the JSON contains — array if JSON starts with `[`, object if JSON starts with `{`.
- Modifying `testdata` in spec does **NOT** modify the JSON file — it's an in-memory copy, original file stays untouched.
- `for...of` works on arrays. For objects, use `Object.entries()`, `Object.values()`, or `Object.keys()` to convert to array first.
- Each data record shows as a **separate test** in the report with its own pass/fail status.
- Test names must be unique per record (e.g., `` `Test ${data.id}` ``) — otherwise Playwright throws an error.
