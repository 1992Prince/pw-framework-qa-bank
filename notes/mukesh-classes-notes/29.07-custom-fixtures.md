# Playwright Custom Fixtures — Deep Dive Notes

---

## Part 1 — Hooks Already Hain, Toh Fixtures Kyun Chahiye?

### Interview Speaking Line

> "Hooks jaise `beforeEach` aur `afterAll` sirf us ek spec file ke scope tak limited hote hain — agar maine 10 spec files mein login karna hai toh mujhe har file mein `beforeEach` copy-paste karna padega, jo code duplication hai aur maintenance nightmare ban jata hai. Fixtures is problem ka solution hain — main ek baar fixture file mein login logic likhta hoon aur use poori project ki kisi bhi spec file mein import karke directly test mein use kar sakta hoon. Fixtures reusable, composable aur automatically isolated hote hain — yeh hooks se fundamentally zyada powerful abstraction hai."

### Difference Table

| Feature | Hooks (`beforeEach`) | Custom Fixtures |
|---------|----------------------|-----------------|
| Scope | Sirf ek spec file | Poori project, koi bhi spec file |
| Reusability | ❌ Nahi — copy-paste karna padega | ✅ Haan — ek baar likho, kahin bhi use karo |
| Composability | ❌ Ek hook ek kaam | ✅ Multiple fixtures ek test mein combine karo |
| Isolation | File-level | Test-level — fully isolated |
| Cleanup | Alag `afterEach` mein likhna padta hai | `use()` ke baad same function mein |
| Readability | Setup aur cleanup alag alag blocks mein | Ek jagah — setup + handoff + cleanup |

## Part 2 — Custom Fixture Kaise Banate Hain — Har Line Ka Matlab

Yeh hai humara actual fixture file:

```javascript
import { test as base } from "@playwright/test";

export const test = base.extend({

  loggedInUser: async ({ page }, use) => {

    await page.goto("https://gauravkhurana.com/practise-api/ui/index.html#/practice");
    await page.getByTestId("form-email").fill("gaurav");
    await page.getByTestId("form-password").fill("gaurav");
    await page.getByTestId("form-submit").click();
    console.log("✅ [auth fixture] Logged in successfully");

    await use(page);

    await page.getByTestId("form-reset").click();
    console.log("🔴 [auth fixture] Logged out successfully");
  },

});

export { expect } from "@playwright/test";
```

Ab har line ka matlab samajhte hain —

---

### Line 1 — Import

```javascript
import { test as base } from "@playwright/test";
```

**Kya ho raha hai:**
Playwright ke package se `test` object import kar rahe hain. Lekin hum iska naam `base` rakh rahe hain — `as base` se rename ho gaya.

**`test as base` kyun?**
Kyunki baad mein hum apna khud ka `test` banayenge (`base.extend(...)` se). Agar dono ka naam `test` hota toh conflict hota — pata nahi chalta playwright wala use ho raha hai ya apna. Isliye playwright wale ko `base` naam dete hain — matlab "yeh foundation hai jiske upar main apna kuch banaunga".

```javascript
// Mental model:
// base  = playwright ka original test (foundation)
// test  = hamara custom test (base ke upar bana hua, extra features ke saath)
```

---

### Line 3 — base.extend()

```javascript
export const test = base.extend({
  // ...
});
```

**`base.extend()` kyun?**
`extend` ka matlab hai — "jo pehle se hai use badhaao, usmein kuch aur add karo." Hum playwright ke original `test` object ko tod nahi rahe, balki usme **apne custom fixtures add** karke ek naya `test` object bana rahe hain.

**`base.extend()` kya accept karta hai?**
Ek **JavaScript object** accept karta hai — key-value pairs ke saath:
- **Key** = fixture ka naam (jo tum test mein `{ }` ke andar likhoge)
- **Value** = ek async function jo fixture ka kaam karta hai (setup + handoff + cleanup)

```javascript
base.extend({
  fixtureNaam1: async (...) => { /* logic */ },
  fixtureNaam2: async (...) => { /* logic */ },
  fixtureNaam3: async (...) => { /* logic */ },
})
```

**Result ko `test` variable mein store kyun karte hain?**
Kyunki `base.extend()` ek **naya test object return karta hai** — jo original playwright test jaisa hi hai lekin usme tumhare custom fixtures bhi hain. Agar store nahi kiya toh result kho jayega. Aur naam `test` diya kyunki spec files mein hum `test(...)` likhte hain — same familiar naam.

**`export const` kyun?**
`export` isliye taaki doosri spec files is `test` ko **import** kar sakein. `const` isliye kyunki yeh value change nahi hogi.

---

### Line 5 — Fixture Name: `loggedInUser`

```javascript
loggedInUser: async ({ page }, use) => {
```

**`loggedInUser`** — yeh fixture ka naam hai. Spec file mein jab test likhoge tab `{ loggedInUser }` likha toh playwright samjhega ki tumhe yeh custom fixture chahiye. Naam kuch bhi rakh sakte ho — meaningful naam lo jo fixture ka kaam bataye.

**`async`** — fixture mein `await` use karna hai (page navigate karna, fill karna, click karna — sab async operations hain), isliye function `async` hona zaroori hai.

**`{ page }`** — pehla parameter. Yeh playwright ke **built-in fixtures inject karo** wali jagah hai. Curly braces mein jo bhi likho, playwright woh fixture provide kar deta hai. Humne `page` manga kyunki login karne ke liye browser page chahiye. Yahan `context`, `browser`, `request` — koi bhi built-in fixture maang sakte the.

**`use`** — doosra parameter. Yeh ek **function** hai jo playwright provide karta hai. `await use(...)` call karne par test body run hoti hai. Use ke pehle ka code = setup. Use ke baad ka code = cleanup. Simple rule.

```javascript
//  Parameter 1          Parameter 2
//      ↓                    ↓
async ({ page },           use) => {
//  ↑
// Built-in fixtures yahan inject karo
// Comma se alag multiple bhi le sakte ho: { page, browserName, request }
```

---

### Lines 8–14 — Setup (Login)

```javascript
await page.goto("https://gauravkhurana.com/practise-api/ui/index.html#/practice");
await page.getByTestId("form-email").fill("gaurav");
await page.getByTestId("form-password").fill("gaurav");
await page.getByTestId("form-submit").click();
console.log("✅ [auth fixture] Logged in successfully");
```

**Kya ho raha hai:**
Yeh sab code `await use(page)` se **pehle** hai — isliye yeh test run hone se **pehle** execute hoga. `beforeEach` ki tarah kaam karta hai.

Hum:
1. Login page pe navigate kar rahe hain
2. Email fill kar rahe hain
3. Password fill kar rahe hain
4. Submit click kar rahe hain
5. Log print kar rahe hain

Test shuru hone tak user already logged in hai.

---

### Line 16 — The Hand-off: `await use(page)`

```javascript
await use(page);
```

**Yeh sabse important line hai.**

`use` ek function hai jo playwright ne parameter mein diya tha. Jab hum `await use(page)` call karte hain tab:

1. **Fixture control test ko de deta hai** — test body run hoti hai
2. **`page` argument pass kiya** — matlab test mein curly braces mein `{ loggedInUser }` jo milega woh actually yeh `page` hi hoga — logged in state mein. `use(page)` mein `page` pass kiya isliye `loggedInUser` = logged-in page
3. **Test khatam hone tak yahan ruka rahega** — test complete hone ke baad `use()` return karta hai aur cleanup code chalti hai

```
┌─────────────────────────────────────────┐
│  FIXTURE FUNCTION                       │
│                                         │
│  1. Login code  ← "beforeEach" jaisa   │
│                                         │
│  2. await use(page) ◄─── TEST RUNS     │
│                          YAHAN          │
│                                         │
│  3. Logout code ← "afterEach" jaisa    │
└─────────────────────────────────────────┘
```

**Agar `use()` na likho?**
Test ko kabhi control nahi milega — test run hi nahi hoga. `use()` call karna mandatory hai.

---

### Lines 18–21 — Teardown (Logout)

```javascript
await page.getByTestId("form-reset").click();
console.log("🔴 [auth fixture] Logged out successfully");
```

**Kya ho raha hai:**
Yeh code `await use(page)` ke **baad** hai — isliye test khatam hone ke **baad** automatically execute hoga. `afterEach` ki tarah.

Hum logout kar rahe hain. Test pass ho ya fail — yeh code hamesha chalega, cleanup guaranteed hai.

---

### Last Line — `export { expect }`

```javascript
export { expect } from "@playwright/test";
```

**Kya ho raha hai:**
Playwright ke `expect` function ko apni fixture file se **re-export** kar rahe hain.

**Kyun?**
Spec files mein dono cheezein chahiye — `test` aur `expect`. Agar yeh line na hoti toh spec file mein do alag imports likhne padte:

```javascript
// ❌ Bina re-export ke — do imports likhne padte
import { test } from "../../fixtures/auth.fixture.js";     // custom test
import { expect } from "@playwright/test";                  // playwright ka expect
```

```javascript
// ✅ Re-export ke saath — sirf ek import kaafi hai
import { test, expect } from "../../fixtures/auth.fixture.js";
```

Clean imports, koi confusion nahi.

## Part 3 — Spec File Mein Kaise Use Karte Hain

```javascript
// tests/29-hooks-fixtures/03-auth-fixture.spec.js

import { test, expect } from "../../fixtures/auth.fixture.js";
//         ↑       ↑
//  custom test   re-exported expect
//  (loggedInUser fixture ke saath)

test("TC001 - Verify logged in user can see practice page", async ({ loggedInUser }) => {
  //                                                                      ↑
  //                                              fixture ka exact naam — jo base.extend() mein define kiya tha
  //                                              playwright ne loggedInUser mein woh page de diya jo use(page) mein pass kiya tha

  // Yahan tak aate aate login ho chuka hai — fixture ne kar diya
  await expect(loggedInUser).toHaveURL(/practice/);
});
```

**`{ loggedInUser }` ka matlab:**
Curly braces mein fixture ka wahi naam likhte hain jo `base.extend()` mein key ke taur pe define kiya tha — yahan `loggedInUser`. Playwright automatically is fixture ko run karega aur `use(page)` mein jo `page` pass kiya tha woh `loggedInUser` variable mein aa jayega — logged in state mein.

> **💡 Suggestion — Rename Karo for Readability:**
> `loggedInUser` naam descriptive toh hai lekin test mein `loggedInUser.click(...)`, `loggedInUser.goto(...)` likhna thoda awkward lagta hai. Tum JavaScript destructuring rename syntax se ise `page` ya koi aur naam de sakte ho:
>
> ```javascript
> // Rename karo — loggedInUser fixture ko 'page' naam do locally
> test("TC001", async ({ loggedInUser: page }) => {
>   await expect(page).toHaveURL(/practice/);  // zyada clean lagta hai
> });
> ```
>
> Dono bilkul same kaam karte hain — sirf local naam alag hai. Rename karna best practice hai jab fixture ek page return karta ho.
