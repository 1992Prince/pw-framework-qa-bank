# Topic 8 — Assertions (Continued)

## PART 5 — Soft Assertions

### Pehle Problem Samjho
Normal `expect()` mein — pehli failure pe test immediately stop ho jaata hai.

```
Registration form pe 5 fields validate karni hain:
Field 1 — Name     → FAIL ❌ — test yahan ruk gaya
Field 2 — Email    → never ran
Field 3 — Phone    → never ran
Field 4 — Address  → never ran
Field 5 — Pincode  → never ran

Report mein sirf ek failure dikhegi.
Baaki 4 fields mein kya problem hai — unknown.
Dobara run karo, fix karo, run karo — baar baar.
```

### Soft Assertion Ka Purpose
Failure pe test continue karta hai. Saari failures collect hoti hain, end mein ek saath report.

```
Field 1 — Name     → FAIL ❌ (noted, continuing...)
Field 2 — Email    → FAIL ❌ (noted, continuing...)
Field 3 — Phone    → PASS ✅
Field 4 — Address  → FAIL ❌ (noted, continuing...)
Field 5 — Pincode  → PASS ✅

Test end pe — 3 failures ek saath report
Ek run mein poori picture clear ho gayi.
```

### Syntax

```typescript
// Normal expect — failure pe STOP
await expect(locator).toHaveValue('text');

// Soft expect — failure pe CONTINUE
await expect.soft(locator).toHaveValue('text');

// Custom message bhi de sakte ho
await expect.soft(locator, 'Name field should have correct value').toHaveValue('Gaurav');
```

### Real Use Case — Form Validation

```typescript
test('registration form validation', async ({ page }) => {
  await page.goto('/register');
  await page.getByRole('button', { name: 'Submit' }).click();

  // Saare validations soft assertions se
  await expect.soft(page.getByLabel('Name'), 'Name error should appear').toBeVisible();
  await expect.soft(page.getByLabel('Email'), 'Email error should appear').toBeVisible();
  await expect.soft(page.getByLabel('Phone'), 'Phone error should appear').toBeVisible();
  await expect.soft(page.getByLabel('Password'), 'Password error should appear').toBeVisible();

  // Critical assertion — normal expect
  // Agar form submit ho gaya toh saara test meaningless hai
  await expect(page).toHaveURL('/register');
});
```

### Kab Soft, Kab Normal — Simple Rule

```
Soft assertions use karo jab:
→ Multiple independent fields validate kar rahe ho
→ Ek failure doosre checks ko meaningless nahi banata
→ Saari failures ek run mein dekhni hain

Normal assertions use karo jab:
→ Critical path pe ho — ek fail matlab aage sab fail
→ Login successful hua ya nahi
→ Correct page pe land hua ya nahi
→ Main element visible hai ya nahi
```

## PART 6 — Non-Retrying Assertions

### Yeh kya hain aur purpose
Abhi tak jo assertions dekhe — `expect(locator).toBeVisible()` — yeh DOM ke saath kaam karte hain, locator ke saath.

Non-retrying assertions plain JavaScript values ke saath kaam karte hain — numbers, strings, arrays, objects, booleans. Koi DOM nahi, koi retry nahi, instant check.

```typescript
const count = await page.locator('li').count();  // count already resolved hai
expect(count).toBe(5);                           // simple number check — retry ka koi matlab nahi
```

### Kahan se aate hain
`@playwright/test` ka `expect` — Jest-compatible hai. Jo Jest mein hote hain, wahi yahan bhi.

### Important Ones — Zaroor Jaano

```typescript
// toBe() — strict equality (===)
// Primitives ke liye — numbers, strings, booleans
expect(count).toBe(5)
expect(status).toBe('active')
expect(isLoggedIn).toBe(true)
// Return: void — synchronous, no await needed

// toEqual() — deep equality
// Objects aur arrays ke liye — structure compare karta hai
expect(userData).toEqual({ name: 'Gaurav', role: 'admin' })
expect(items).toEqual(['Home', 'About', 'Contact'])
// Return: void

// toBe vs toEqual — yeh confuse karte hain
expect({ a: 1 }).toBe({ a: 1 })     // ❌ FAIL — alag references hain memory mein
expect({ a: 1 }).toEqual({ a: 1 })  // ✅ PASS — structure same hai

// toContain() — array ya string mein item hai
expect(['admin', 'user', 'viewer']).toContain('admin')
expect('Welcome back Gaurav').toContain('Gaurav')
// Return: void

// toHaveLength() — array ya string ki length
expect(searchResults).toHaveLength(10)
expect('Playwright').toHaveLength(10)
// Return: void

// Number comparisons
expect(responseTime).toBeLessThan(2000)       // 2 second se kam
expect(itemCount).toBeGreaterThan(0)          // zero se zyada
expect(price).toBeGreaterThanOrEqual(100)
expect(discount).toBeLessThanOrEqual(50)
// Return: void

// Truthiness
expect(token).toBeTruthy()      // truthy value — non-empty string, non-zero number, object
expect(error).toBeFalsy()       // falsy value — null, undefined, 0, '', false
expect(value).toBeNull()        // exactly null
expect(value).toBeUndefined()   // exactly undefined
expect(value).toBeDefined()     // not undefined
// Return: void

// String matching
expect(url).toMatch(/dashboard/)          // regex match
expect(errorMsg).toMatch(/required field/i)
// Return: void

// toBeNaN
expect(parseInt('abc')).toBeNaN()
// Return: void
```

### Real Use Cases — Kab Aate Hain Test Mein

```typescript
// API response validate karna
const response = await request.get('/api/users');
const body = await response.json();

expect(response.status()).toBe(200)              // status code check
expect(body.users).toHaveLength(10)             // 10 users aaye
expect(body.users[0]).toEqual({                 // pehla user ka structure
  id: 1,
  name: 'Gaurav',
  role: 'admin'
})
expect(body.total).toBeGreaterThan(0)           // total > 0

// Count check karne ke baad value assertion
const rowCount = await page.locator('tbody tr').count();
expect(rowCount).toBeGreaterThan(0)
expect(rowCount).toBeLessThanOrEqual(20)        // pagination — max 20 rows

// Text extract karke validate karna
const headingText = await page.getByRole('heading').innerText();
expect(headingText).toContain('Dashboard')
expect(headingText).toMatch(/Welcome/i)

// Array of texts validate karna
const allTexts = await page.locator('li').allInnerTexts();
expect(allTexts).toHaveLength(5)
expect(allTexts).toContain('Settings')
expect(allTexts).toEqual(['Home', 'About', 'Services', 'Blog', 'Contact'])
```

### Web-First vs Non-Retrying — Side by Side

```typescript
// SAME THING — do tarike se

// Web-first — locator directly, auto-retry, preferred
await expect(page.locator('tbody tr')).toHaveCount(5)

// Non-retrying — value extract karo, phir check karo, no retry
const count = await page.locator('tbody tr').count()
expect(count).toBe(5)

// Toh kaunsa use karo?
// → Web-first prefer karo jab locator available ho — auto-retry milti hai
// → Non-retrying use karo jab value already resolve ho chuki ho
//   (API response body, extracted text, calculated values)
```

## PART 7 — toPass() — Custom Polling

### Purpose
Built-in assertions ek cheez check karti hain. Lekin kabhi kabhi condition complex hoti hai — multiple checks, custom logic. Tab `toPass()` use karo.

```typescript
// Scenario: Background job chal raha hai
// Items appear hote hain slowly — pehle 0, phir 3, phir 7, phir 10
// Exact timing unknown — kabhi 3 sec mein, kabhi 8 sec mein

await expect(async () => {
  const count = await page.locator('.result-item').count();
  expect(count).toBeGreaterThan(0);   // andar normal value assertion
}).toPass({
  timeout: 15000,              // 15 seconds tak try karo
  intervals: [500, 1000, 2000] // 500ms pe try, phir 1000ms, phir 2000ms
});
```

### Real Scenario — Multiple Conditions

```typescript
// Order status page — 'Processing' se 'Delivered' pe change hota hai
// Exact time unknown

await expect(async () => {
  const status = await page.getByTestId('order-status').innerText();
  const timestamp = await page.getByTestId('delivery-time').innerText();

  expect(status).toBe('Delivered');           // status change hua
  expect(timestamp).not.toBe('Pending');      // timestamp update hua
}).toPass({
  timeout: 30000,
  intervals: [2000, 3000, 5000]
});
```

### toPass() vs Built-in Assertion — kab kya

```typescript
// Simple case — built-in prefer karo
await expect(page.getByText('Success')).toBeVisible();  // ✅ clean

// Complex multi-condition — toPass() use karo
await expect(async () => {
  const errors = await page.locator('.error').count();
  const warnings = await page.locator('.warning').count();
  expect(errors + warnings).toBe(0);  // dono zero hone chahiye
}).toPass({ timeout: 10000 });
```

## PART 8 — Negation

### Purpose
Koi bhi assertion `.not.` se negate ho sakti hai. "Yeh nahi hona chahiye" validate karne ke liye.

```typescript
// Visibility
await expect(errorMsg).not.toBeVisible()      // error nahi dikhna chahiye
await expect(modal).not.toBeVisible()         // modal close ho gaya

// State
await expect(submitBtn).not.toBeDisabled()    // button clickable hai
await expect(checkbox).not.toBeChecked()      // unchecked hai

// Content
await expect(input).not.toHaveValue('')       // empty nahi hai
await expect(heading).not.toHaveText('Error') // error page nahi hai

// URL
await expect(page).not.toHaveURL('/error')    // error page pe nahi gaye
await expect(page).not.toHaveURL(/login/)     // login page se nikal gaye
```

### Real Use Cases

```typescript
// Logout ke baad — dashboard nahi dikhna chahiye
await expect(page.getByRole('heading', { name: 'Dashboard' })).not.toBeVisible();
await expect(page).not.toHaveURL('/dashboard');

// Form submit ke baad — error message nahi hona chahiye
await expect(page.getByTestId('error-banner')).not.toBeVisible();

// Delete ke baad — item list mein nahi hona chahiye
await expect(page.getByText('Deleted Item Name')).not.toBeVisible();
```

## PART 9 — Custom Error Messages

### Purpose
Default Playwright error generic hota hai. Custom message se turant samajh aata hai kya fail hua aur kyun.

```typescript
// Without message
await expect(btn).toBeEnabled();
// Error output:
// "Error: Locator expected to be enabled
//  Received: disabled"
// → Kaunsa button? Kyun disabled? Unknown.

// With message
await expect(btn, 'Submit button should be enabled after all required fields are filled').toBeEnabled();
// Error output:
// "Submit button should be enabled after all required fields are filled
//  Error: Locator expected to be enabled
//  Received: disabled"
// → Ab turant pata hai — submit button hai, required fields ka issue hai
```

### Syntax — bilkul simple

```typescript
expect(locator, 'Your message here').assertion()
expect(page, 'Your message here').assertion()
expect.soft(locator, 'Your message here').assertion()
```

### Real Examples

```typescript
await expect(
  page.getByRole('heading'),
  'Page heading should show username after login'
).toHaveText('Welcome, Gaurav');

await expect(
  page.getByTestId('cart-count'),
  'Cart should have 3 items after adding 3 products'
).toHaveText('3');

await expect(
  page,
  'Should redirect to dashboard after successful login'
).toHaveURL('**/dashboard');

await expect.soft(
  page.getByLabel('Email'),
  'Email validation error should appear for empty submission'
).toBeVisible();
```

> CI mein raat ko failure aaye — message padho, turant samajh aao. Yeh senior code ki pehchan hai.

## PART 10 — Assertion Timeout Configuration

### Default kya hai aur kahan set hota hai

```typescript
// playwright.config.ts
export default defineConfig({
  expect: {
    timeout: 5000   // 5 seconds — default for ALL web-first assertions
  }
});
```

### Increase karo globally — slow apps ke liye

```typescript
expect: {
  timeout: 10000  // 10 seconds — sab assertions ke liye
}
```

### Per-assertion override — specific slow element ke liye

```typescript
// Heavy report table — load hone mein 20 sec lagte hain
await expect(
  page.getByTestId('report-table'),
  'Report table should load within 20 seconds'
).toBeVisible({ timeout: 20000 });

// Fast check — agar 2 sec mein nahi aaya toh actually missing hai
await expect(page.getByTestId('toast')).toBeVisible({ timeout: 2000 });

// Content assertion ka bhi timeout hota hai
await expect(heading).toHaveText('Dashboard', { timeout: 8000 });
```

### Non-retrying assertions mein timeout nahi hota

```typescript
// Yeh instant hain — timeout ka concept nahi
expect(count).toBe(5)                  // no timeout option
expect(arr).toHaveLength(3)            // no timeout option
expect(obj).toEqual({ name: 'x' })    // no timeout option
```
