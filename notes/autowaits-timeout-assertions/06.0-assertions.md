# 🎭 Playwright Assertions — Complete Notes

---

## PART 1 — Assertion Kya Hota Hai? Why Does It Matter?

Assertion ek check hai — *"jo hona chahiye tha, woh hua ya nahi?"* Test mein action karo (`click`, `fill`, `navigate`) phir assert karo ki expected state aa gaya ya nahi.

Playwright mein assertions `expect()` function ke through hoti hain. Yeh **Jest-style API** hai — familiar lagega agar Jest use kiya ho. But Playwright ne iske upar **web-specific superpowers** add kiye hain.

```typescript
import { test, expect } from '@playwright/test';

test('login test', async ({ page }) => {
  await page.goto('/login');
  await page.fill('#email', 'user@example.com');
  await page.fill('#password', 'pass123');
  await page.click('#submit');

  // Assertion — kya dashboard visible hua?
  await expect(page.locator('h1')).toHaveText('Dashboard'); // auto-retry hogi
});
```

## PART 2 — Do Types Ki Assertions — Core Concept

Playwright mein do **fundamentally alag** tarike ke assertions hain. Yeh sabse important distinction hai jo interview mein puchha jaata hai.

---

### Type 1 — Auto-Retrying Assertions (Web-First Assertions)

**Kya hoti hain:** Yeh assertions tab tak retry karti hain jab tak condition pass na ho jaye ya assertion timeout hit na ho jaye. Async hain — `await` karna zaroori hai.

**Default timeout:** 5 seconds (globally configure kar sakte ho).

**Kyon hain:** Web pages asynchronous hain — API call ke baad data load hota hai, animation ke baad element visible hoti hai. Ek baar check karo aur fail karo — wrong approach. Retry karo jab tak ready na ho — right approach.

Ye locator ya page pe hoti hain:

```typescript
// Locator assertions — most common
await expect(page.locator('#status')).toBeVisible();
await expect(page.locator('#status')).toHaveText('Success');
await expect(page.locator('#status')).toContainText('Success');
await expect(page.locator('input')).toBeEnabled();
await expect(page.locator('input')).toBeDisabled();
await expect(page.locator('input')).toBeChecked();
await expect(page.locator('input')).toBeEditable();
await expect(page.locator('#msg')).toBeHidden();
await expect(page.locator('#msg')).toBeAttached();
await expect(page.locator('#msg')).toBeEmpty();
await expect(page.locator('#msg')).toBeFocused();
await expect(page.locator('#msg')).toBeInViewport();
await expect(page.locator('input')).toHaveValue('expected');
await expect(page.locator('input')).toHaveValues(['opt1', 'opt2']); // multi-select
await expect(page.locator('#el')).toHaveAttribute('href', '/home');
await expect(page.locator('#el')).toHaveClass('active');
await expect(page.locator('#el')).toHaveId('submit-btn');
await expect(page.locator('#el')).toHaveCSS('color', 'rgb(0, 0, 0)');
await expect(page.locator('ul li')).toHaveCount(5);
await expect(page.locator('#el')).toHaveRole('button');

// Page-level assertions — also auto-retrying
await expect(page).toHaveTitle('Dashboard | MyApp');
await expect(page).toHaveTitle(/Dashboard/);
await expect(page).toHaveURL('https://app.example.com/dashboard');
await expect(page).toHaveURL(/.*dashboard.*/);
```

---

### Type 2 — Non-Retrying Assertions (Generic Assertions)

**Kya hoti hain:** Yeh ek baar evaluate karti hain — pass ya fail, immediately. No wait, no retry. Synchronous bhi ho sakti hain (kuch cases mein).

**Kab use karo:** Jab value already tumhare paas hai — JavaScript variables, API response data, computed values. DOM elements ke liye avoid karo — timing issues create karengi.

```typescript
// Generic matchers — pure JS values pe
expect(42).toBe(42);
expect('hello').toBe('hello');
expect(null).toBeNull();
expect(undefined).toBeUndefined();
expect(value).toBeDefined();
expect(true).toBeTruthy();
expect(0).toBeFalsy();
expect([1, 2, 3]).toHaveLength(3);
expect('hello world').toContain('world');
expect([1, 2, 3]).toContain(2);
expect({ name: 'Prince' }).toEqual({ name: 'Prince' });
expect(5).toBeGreaterThan(3);
expect(3).toBeLessThan(5);
expect(5).toBeGreaterThanOrEqual(5);
expect(3).toBeLessThanOrEqual(3);
expect(3.14159).toBeCloseTo(3.14, 2);

// Arrays
expect(['a', 'b', 'c']).toEqual(expect.arrayContaining(['a', 'b']));

// Objects
expect({ a: 1, b: 2 }).toMatchObject({ a: 1 });

// Regex
expect('hello world').toMatch(/world/);
```

**The most important comparison — jo interviewer dekhna chahta hai:**

```typescript
// ❌ WRONG — Non-retrying assertion on DOM element
const isVisible = await page.locator('#status').isVisible(); // immediate snapshot
expect(isVisible).toBe(true); // agar element 2s baad aata hai — fail

// ✅ CORRECT — Auto-retrying assertion on DOM element
await expect(page.locator('#status')).toBeVisible(); // retry karega 5s tak

// Non-retrying theek hai sirf pure values ke liye
const response = await page.request.get('/api/user');
expect(response.status()).toBe(200); // response already hai — immediate fine hai
const body = await response.json();
expect(body.name).toBe('Prince'); // variable already hai — immediate fine hai
```

## PART 3 — Auto-Retrying Assertions Ka Internal Working

Jab `await expect(locator).toBeVisible()` call karte ho — internally:

| Step | Description |
|------|-------------|
| **Step 1** | Locator DOM mein evaluate karo. |
| **Step 2** | Condition check karo (visible hai?). |
| **Step 3** | Agar pass — done. |
| **Step 4** | Agar fail — wait karo, dobara Step 1 se. Locator fresh re-evaluate hota hai — stale element problem nahi. |
| **Step 5** | 5 seconds ke andar pass nahi hua — `AssertionError` with clear message. |

> **Retry interval:** Playwright exponentially backoff karta hai — roughly 100ms, 250ms, 500ms, 1000ms intervals pe check karta hai. Exact intervals internal implementation detail hain.

## PART 4 — Complete Auto-Retrying Assertions List with Purpose and Use Case

### Locator Assertions

---

#### `toBeVisible()`

**Kya check karta hai:** Element DOM mein hai aur user ko dikh raha hai.

**Use case:** Page load ke baad content appear hona, action ke baad UI update verify karna.

```typescript
await page.click('#load-data');
await expect(page.locator('.data-table')).toBeVisible();
// Retry karega jab tak table appear na ho jaye
```

---

#### `toBeHidden()`

**Kya check karta hai:** Element visible nahi hai — ya DOM mein nahi ya hidden hai.

**Use case:** Loader/spinner gayab hona, error message dismiss hona.

```typescript
await page.click('#submit');
await expect(page.locator('.loading-spinner')).toBeHidden();
// Confirm karo ki loader gone
```

---

#### `toBeAttached()`

**Kya check karta hai:** Element DOM tree mein present hai (visible nahi bhi ho sakta).

**Use case:** Dynamic component DOM mein add hona verify karna.

```typescript
await page.click('#add-item');
await expect(page.locator('.new-item')).toBeAttached();
```

---

#### `toBeEnabled()`

**Kya check karta hai:** Form element disabled nahi hai.

**Use case:** Multi-step form mein Next button enable hona, required fields fill hone ke baad submit button enable hona.

```typescript
await page.fill('#required-field', 'value');
await expect(page.locator('#submit-btn')).toBeEnabled();
```

---

#### `toBeDisabled()`

**Kya check karta hai:** Form element disabled hai.

**Use case:** Form submit hone ke baad button disable hona (double submission prevention), permission-based UI.

```typescript
await page.click('#submit');
await expect(page.locator('#submit-btn')).toBeDisabled();
// Confirm double submit prevention
```

---

#### `toBeChecked()`

**Kya check karta hai:** Checkbox ya radio button checked hai.

**Options:** `{ checked: false }` pass kar sakte ho unchecked verify karne ke liye.

**Use case:** Terms & conditions checkbox, feature toggle.

```typescript
await page.locator('#terms-checkbox').check();
await expect(page.locator('#terms-checkbox')).toBeChecked();

// Unchecked verify karna
await expect(page.locator('#newsletter')).toBeChecked({ checked: false });
```

---

#### `toBeEditable()`

**Kya check karta hai:** Input field editable hai — not readonly, not disabled.

**Use case:** Edit mode enable hone pe fields editable ho jaayein.

```typescript
await page.click('#edit-profile-btn');
await expect(page.locator('#name-input')).toBeEditable();
```

---

#### `toBeEmpty()`

**Kya check karta hai:** Input field empty hai ya container mein koi content nahi.

**Use case:** Form clear hona verify karna, empty state UI.

```typescript
await page.click('#clear-btn');
await expect(page.locator('#search-input')).toBeEmpty();
```

---

#### `toBeFocused()`

**Kya check karta hai:** Element currently focused hai.

**Use case:** Page load pe auto-focus verify karna, tab navigation testing.

```typescript
await page.goto('/login');
await expect(page.locator('#email-input')).toBeFocused(); // auto-focus on load
```

---

#### `toBeInViewport()`

**Kya check karta hai:** Element viewport mein visible area mein hai.

**Use case:** Scroll-to-element verify karna, lazy loaded content.

```typescript
await page.locator('#footer').scrollIntoViewIfNeeded();
await expect(page.locator('#footer')).toBeInViewport();
```

---

#### `toHaveText()`

**Kya check karta hai:** Element ka text exactly match karta hai. String ya regex dono.

> **Note:** Whitespace normalize karta hai — leading/trailing spaces ignore.

```typescript
await expect(page.locator('h1')).toHaveText('Welcome, Prince!');
await expect(page.locator('h1')).toHaveText(/Welcome/);

// Multiple elements ke liye array pass karo
await expect(page.locator('li')).toHaveText(['Item 1', 'Item 2', 'Item 3']);
```

---

#### `toContainText()`

**Kya check karta hai:** Element ke text mein substring ya pattern exist karta hai. Exact match zaroori nahi.

**Use case:** Dynamic content jahan exact text pata na ho.

```typescript
await expect(page.locator('.notification')).toContainText('saved');
// "Your changes have been saved successfully" — pass hoga

// Array of locators ke liye
await expect(page.locator('tr')).toContainText(['Prince', 'Active']);
```

---

#### `toHaveValue()`

**Kya check karta hai:** Input, textarea, select ka current value.

**Use case:** Form pre-population verify karna, user input verify karna.

```typescript
await expect(page.locator('#email')).toHaveValue('prince@example.com');
await expect(page.locator('#search')).toHaveValue(/playwright/i);
```

---

#### `toHaveValues()`

**Kya check karta hai:** Multi-select dropdown ke selected values.

**Use case:** Multi-select form elements.

```typescript
await expect(page.locator('#skills-select')).toHaveValues(['playwright', 'selenium']);
```

---

#### `toHaveAttribute()`

**Kya check karta hai:** Element ka specific HTML attribute aur uski value.

**Use case:** Link href verify karna, aria attributes, data attributes.

```typescript
await expect(page.locator('a.logo')).toHaveAttribute('href', '/home');
await expect(page.locator('#btn')).toHaveAttribute('aria-disabled', 'true');
await expect(page.locator('#input')).toHaveAttribute('type', 'email');
```

---

#### `toHaveClass()`

**Kya check karta hai:** Element ke CSS classes.

**Use case:** Active state, error state, selected state verify karna.

```typescript
await expect(page.locator('#dashboard-tab')).toHaveClass('active');
await expect(page.locator('#input')).toHaveClass(/error/); // regex bhi works
```

---

#### `toHaveId()`

**Kya check karta hai:** Element ka `id` attribute.

```typescript
await expect(page.locator('h1')).toHaveId('main-heading');
```

---

#### `toHaveCSS()`

**Kya check karta hai:** Element ki computed CSS property.

**Use case:** Visual regression, theme changes, dynamic styling.

```typescript
await expect(page.locator('#error-msg')).toHaveCSS('color', 'rgb(255, 0, 0)');
await expect(page.locator('.sidebar')).toHaveCSS('display', 'none');
```

---

#### `toHaveCount()`

**Kya check karta hai:** Locator se match hone wale elements ki exact count.

**Use case:** Table rows count, list items, search results.

```typescript
await page.click('#load-users');
await expect(page.locator('table tbody tr')).toHaveCount(10);

// Dynamic count
const response = await page.waitForResponse('/api/users');
const users = await response.json();
await expect(page.locator('.user-card')).toHaveCount(users.length);
```

---

#### `toHaveRole()`

**Kya check karta hai:** Element ka ARIA role.

**Use case:** Accessibility testing.

```typescript
await expect(page.locator('#nav')).toHaveRole('navigation');
await expect(page.locator('#submit')).toHaveRole('button');
```

## PART 5 — Complete Non-Retrying Assertions List

Non-retrying assertions tab use karo jab value already hai — DOM se nahi, JavaScript se.

```typescript
// Basic equality
expect(value).toBe(42);            // strict equality (===)
expect(obj).toEqual({ a: 1 });     // deep equality (objects/arrays)
expect(obj).toMatchObject({ a: 1 }); // partial match — extra keys allowed

// Null/undefined checks
expect(value).toBeNull();
expect(value).toBeUndefined();
expect(value).toBeDefined();

// Truthy/falsy
expect(value).toBeTruthy();  // any truthy value
expect(value).toBeFalsy();   // 0, '', null, undefined, false

// Numbers
expect(5).toBeGreaterThan(3);
expect(3).toBeLessThan(5);
expect(5).toBeGreaterThanOrEqual(5);
expect(3.14).toBeCloseTo(3.1, 1); // floating point comparison

// Strings
expect('hello world').toContain('world');
expect('hello').toMatch(/^hel/);
expect('hello').toHaveLength(5);

// Arrays
expect([1, 2, 3]).toHaveLength(3);
expect([1, 2, 3]).toContain(2);
expect([1, 2, 3]).toEqual(expect.arrayContaining([1, 3]));

// Errors
expect(() => throwingFn()).toThrow();
expect(() => throwingFn()).toThrow('expected message');

// API response — non-retrying appropriate hai
const response = await page.request.get('/api/data');
expect(response.status()).toBe(200);
expect(response.headers()['content-type']).toContain('application/json');
const body = await response.json();
expect(body).toHaveLength(10);
expect(body[0]).toMatchObject({ id: expect.any(Number), name: expect.any(String) });
```

## PART 6 — Soft Assertions

**Kya hain:** Normal assertion fail hone pe test immediately stop ho jaata hai. **Soft assertion** fail hone pe test **continue** karta hai — saari assertions run hoti hain, phir end mein report mein sab failures dikhte hain.

**Kab use karo:** Jab ek hi page pe multiple independent things verify karne hain aur sab failures ek saath dekhne hain — puri picture ek run mein milti hai.

```typescript
test('dashboard page verification', async ({ page }) => {
  await page.goto('/dashboard');

  // Soft assertions — sab run hongi chahe koi fail ho
  await expect.soft(page.locator('h1')).toHaveText('Dashboard');
  await expect.soft(page.locator('#user-name')).toContainText('Prince');
  await expect.soft(page.locator('#notification-count')).toBeVisible();
  await expect.soft(page.locator('.sidebar')).toBeVisible();
  await expect.soft(page).toHaveURL(/.*dashboard.*/);

  // Ab check karo ki koi soft failure tha ya nahi
  expect(test.info().errors).toHaveLength(0);
  // Agar koi soft failure tha — yahan hard fail hoga, sab errors report mein dikhenge
});
```

**Soft vs Hard — key differences:**

| Type | Fail Behavior |
|------|---------------|
| **Hard assertion** | Fail → test immediately stop → baaki steps skip → fail report. |
| **Soft assertion** | Fail → test continue → sab steps run → end mein failures compile hokar report mein aate hain. |

**Real use case — form validation testing:**

```typescript
test('all form validation errors appear at once', async ({ page }) => {
  await page.goto('/registration');
  await page.click('#submit'); // empty form submit

  // Sab validation errors ek saath verify karo
  await expect.soft(page.locator('#email-error')).toContainText('Email required');
  await expect.soft(page.locator('#password-error')).toContainText('Password required');
  await expect.soft(page.locator('#name-error')).toContainText('Name required');
  await expect.soft(page.locator('#terms-error')).toContainText('Please accept terms');

  expect(test.info().errors).toHaveLength(0);
});
```

> **ETS scenario:** Assessment submission page pe 8–10 fields ki validation verify karni thi ek run mein. Hard assertions se sirf pehla error pata chalta. Soft assertions se ek run mein sab 10 fields ki validation status milti.

## PART 8 — `expect.poll()` — Custom Polling Assertion

**Kab use karo:** Jab kisi bhi synchronous value ko async polling assertion mein convert karna ho. Non-DOM values ke liye — API state, database state, file system state.

**Default timeout:** 5 seconds. Custom mein set kar sakte ho.

```typescript
// API repeatedly check karna jab tak condition true na ho
await expect.poll(async () => {
  const response = await page.request.get('/api/job-status');
  const body = await response.json();
  return body.status;
}, {
  message: 'Job should complete within timeout',
  timeout: 30000,
  intervals: [1000, 2000, 5000], // custom polling intervals
}).toBe('completed');

// Page URL polling
await expect.poll(() => page.url()).toContain('/dashboard');

// Element count polling
await expect.poll(async () => {
  return await page.locator('.notification').count();
}).toBeGreaterThan(0);
```

**`expect.poll()` vs `waitForFunction()` fark:**

| Method | Runs In | Can Use |
|--------|---------|---------|
| `expect.poll()` | Node.js | Playwright APIs — `page.request`, `locator.count()` etc. |
| `waitForFunction()` | Browser | `window` variables, DOM properties |

---

## Notes

**Web-first assertions (`toBeVisible`, `toHaveText`, etc.)**

- Waits for element to be attached, visible, stable in DOM
- Auto-retries the *element state check*
- Only works with **locators**

**`expect.poll()`**

- Jab condition **UI element se related nahi hai** — API response, DB value, file status, background job result etc.
- Tum ek **callback function** dete ho, wo har interval pe **execute** hota hai
- Uska **return value** expect condition se match kiya jata hai
- Response time uncertain ho (kabhi fast, kabhi slow) — tab ye perfect fit hai

## How to explain in interview (English)

> "Playwright's web-first assertions like `toBeVisible()` have built-in auto-waiting, but they only work on locators — they poll the DOM state of an element until it matches the expected condition.
>
> But in cases where I'm not asserting on a UI element — for example, calling an API and the response time is unpredictable — I use `expect.poll()`. It takes a callback function, re-executes that function at defined intervals until the returned value satisfies the assertion, or until timeout.
>
> **Example use case:** I had a scenario where after triggering an action, a backend job would process asynchronously and update a status field via an API. The processing time varied — sometimes 2 seconds, sometimes 15. Instead of using a hard wait or a manual retry loop, I used:
>
> ```
> await expect.poll(async () => {
>   const res = await request.get('/api/job-status');
>   return (await res.json()).status;
> }, { timeout: 20000 }).toBe('COMPLETED');
> ```
>
> This way, Playwright kept polling the API status until it became `COMPLETED`, with proper timeout and retry handling — avoiding flaky tests and unnecessary fixed waits."

## PART 9 — `expect.toPass()` — Retry Block

**Kab use karo:** Jab multiple assertions ka ek block tab tak retry karna ho jab tak sab pass na ho jayein. Complex conditions ke liye.

> ⚠️ **Important:** `toPass` ka default timeout `0` hai — test timeout inherit karta hai. Custom `expect` timeout respect nahi karta. **Explicitly timeout specify karo.**

```typescript
// Multiple assertions ka block retry karo
await expect(async () => {
  const count = await page.locator('tbody tr').count();
  expect(count).toBeGreaterThan(0);

  const firstRow = page.locator('tbody tr:first-child');
  await expect(firstRow).toBeVisible();
  await expect(firstRow.locator('td:first-child')).not.toContainText('Loading');
}).toPass({
  timeout: 15000,
  intervals: [500, 1000, 2000, 5000]
});

// Navigation + content combination
await expect(async () => {
  await page.goto('/api-status');
  await expect(page.locator('#status')).toHaveText('healthy');
}).toPass({ timeout: 30000 });
```

**`toPass()` vs `expect.poll()` kab kya use karo:**

| Method | Use When |
|--------|----------|
| `expect.poll()` | Ek single value return karo, matcher pe check karo. Simple condition. |
| `toPass()` | Multiple assertions ka block retry karo. Complex multi-step condition. |

```typescript
// poll — single value return karna hai
await expect.poll(async () => {
  const res = await page.request.get('/api/status');
  return (await res.json()).state;
}).toBe('ready');

// toPass — multiple assertions, koi ek fail hua to poora block retry
await expect(async () => {
  await expect(page.locator('#state')).toHaveText('ready');
  await expect(page.locator('#items')).toHaveCount(5);
  await expect(page.locator('#error')).not.toBeVisible();
}).toPass({ timeout: 20000 });
```

## PART 10 — Custom Error Messages

## Notes — Custom Expect Message

**Why needed:**

- Default expect failure sirf generic error deta hai (e.g. "expected true, got false")
- Bina context ke debug karna mushkil ho jata hai, especially jab test suite bada ho
- Custom message se **failure ka reason turant samajh aata hai** — reports mein bhi clearly dikhta hai (pass ho ya fail)

**Syntax:**
Second argument as string in `expect()`:

```
await expect(locator, 'custom message here').toBeVisible();
```

**Examples:**

```
// URL check
await expect(page, 'homepage URL should load correctly').toHaveURL('https://example.com');

// Title check
await expect(page, 'homepage title should match expected value').toHaveTitle('My App - Home');

// Login check
await expect(page.getByText('Welcome'), 'user should be logged in successfully').toBeVisible();

// Soft assertion
expect.soft(price, 'product price should be 499').toBe(499);
```

**Message likhne ka tareeka (best practice):**

- Assertion **kya verify kar raha hai** wo describe karo, error nahi
- ❌ "title is not matching"
- ✅ "homepage title should match expected value"
- Matlab statement **positive/expected behavior** ke form mein likho, taaki pass/fail dono readable ho

## Interview mein kaise bolo (English)

> "By default, if an assertion fails, Playwright only gives a generic error like 'expected true but got false' — which doesn't tell you *what* was being validated, especially in large test suites with many assertions.
>
> So I always pass a custom message as the second argument to `expect()`. It describes the intent of the assertion — like 'homepage title should match expected value' instead of just failing silently. This message shows up in both passing and failing steps in the report, which makes debugging and reporting much clearer for the whole team, not just me."

## PART 11 — Custom Assertions — `expect.extend()`

Baar baar repeat hone wali domain-specific assertions ko custom matcher mein convert karo.

```typescript
// custom-matchers.ts
import { expect } from '@playwright/test';

expect.extend({
  async toBeLoggedIn(page) {
    const userMenu = page.locator('#user-menu');
    const isVisible = await userMenu.isVisible();
    return {
      pass: isVisible,
      message: () => isVisible
        ? 'Expected user to NOT be logged in, but user menu is visible'
        : 'Expected user to be logged in, but user menu is not visible'
    };
  },

  async toHaveTableRows(locator, expectedCount: number) {
    const rows = await locator.locator('tbody tr').count();
    return {
      pass: rows === expectedCount,
      message: () => `Expected ${expectedCount} rows but found ${rows}`
    };
  }
});

// Test mein use karo
test('should be logged in', async ({ page }) => {
  await page.goto('/');
  await expect(page).toBeLoggedIn();
  await expect(page.locator('#users-table')).toHaveTableRows(10);
});
```

## PART 12 — Assertion Timeout Configuration

```typescript
// playwright.config.ts
export default defineConfig({
  expect: {
    timeout: 10000,    // default 5000ms ko 10s kar diya
    toHaveScreenshot: {
      maxDiffPixels: 100,    // visual comparison ke liye
    },
  }
```

## PART 14 — API Assertions

```typescript
test('API response validation', async ({ request }) => {
  const response = await request.get('/api/users');

  // Non-retrying — response already hai
  expect(response.status()).toBe(200);
  expect(response.ok()).toBeTruthy();
  expect(response.headers()['content-type']).toContain('application/json');

  const body = await response.json();
  expect(body).toHaveLength(10);
  expect(body[0]).toMatchObject({
    id: expect.any(Number),
    name: expect.any(String),
    email: expect.stringContaining('@')
  });

  // POST request
  const createResponse = await request.post('/api/users', {
    data: { name: 'Prince', role: 'SDET' }
  });
  expect(createResponse.status()).toBe(201);
  const created = await createResponse.json();
  expect(created.id).toBeDefined();
  expect(created.name).toBe('Prince');
});
```
