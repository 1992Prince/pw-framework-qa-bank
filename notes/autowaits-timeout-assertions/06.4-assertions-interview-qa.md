# Playwright — Assertions Interview Q&A

---

## Q1. What are Auto-retry assertions and how are they different from Non-Retrying assertions? Purpose and where to use what?

Auto-Retry Assertions woh hain jo internally DOM ko baar baar poll karte hain jab tak condition true na ho ya timeout expire na ho. Playwright ka core feature hai — isliye tests flaky nahi hote.

Non-Retrying Assertions woh hain jo ek baar check karte hain, koi wait nahi, instant result. Plain JavaScript values ke saath kaam karte hain.

```typescript
// AUTO-RETRY — DOM asynchronously update hota hai
// Login button click ke baad success message 800ms baad aayega
await loginBtn.click();
await expect(page.getByText('Welcome!')).toBeVisible();
// Playwright 100ms...200ms...800ms — YES, visible → PASS ✅
// Bina waitForTimeout ke handle ho gaya

// NON-RETRYING — value already resolved hai
const count = await page.locator('li').count(); // count ek number hai
expect(count).toBe(5); // instant check — retry ka koi matlab nahi
```

**Kab kya use karo:**
- **Auto-retry** → jab DOM pe assertion karo, async updates ho rahi hoon, spinner, loading, network call ke baad UI update
- **Non-retrying** → jab value already extract ho chuki ho — API response body, count, extracted text, calculated values

## Q2. What do you understand by Boolean State Methods?

Boolean State Methods woh methods hain jo element ka abhi ka state ek baar check karte hain aur true ya false return karte hain. Yeh Locator pe available hain, page pe nahi. Inme koi auto-retry nahi hoti, koi timeout nahi — ek instant DOM check.

**Purpose:** Conditional logic — result ke basis pe aage kya karna hai decide karna. Test assertion ke liye nahi.

```typescript
// Ye sab Locator pe hain:
locator.isVisible()    // → Promise<boolean> — visible hai ya nahi
locator.isHidden()     // → Promise<boolean> — hidden hai ya nahi
locator.isEnabled()    // → Promise<boolean> — disabled attribute nahi hai
locator.isDisabled()   // → Promise<boolean> — disabled attribute hai
locator.isChecked()    // → Promise<boolean> — checkbox/radio checked hai
locator.isEditable()   // → Promise<boolean> — not disabled AND not readonly

// Sahi use — conditional flow
if (await page.getByTestId('cookie-banner').isVisible()) {
  await page.getByRole('button', { name: 'Accept' }).click();
}
// Test continue hoga chahe banner ho ya na ho

// Galat use — timing issue aayega
const result = await successMsg.isVisible();
expect(result).toBe(true); // ❌ DOM abhi update ho raha tha → false mil sakta hai
```

## Q3. What is the difference between isVisible() and toBeVisible()?

Dono visibility check karte hain lekin purpose, behavior aur return type bilkul alag hain.

- `isVisible()` — ek **question** hai: "Abhi visible hai?" Ek baar check, instant answer.
- `toBeVisible()` — ek **guarantee** hai: "Visible hona chahiye." Wait karega jab tak ho ya timeout ho.

```typescript
// isVisible() — instant, boolean milta hai, no retry
const isBannerThere = await banner.isVisible(); // Promise<boolean>
// 0ms pe check kiya — agar DOM update ho raha tha toh false milega

// toBeVisible() — polling, void return, test fails on timeout
await expect(banner).toBeVisible(); // Promise<void>
// 100ms, 200ms, 300ms... retry karta rahega 5s tak

// REAL SCENARIO — login button click, toast 500ms baad aata hai:

// ❌ Wrong — isVisible() se assertion
await loginBtn.click();
const visible = await toast.isVisible(); // immediately check — 500ms nahi gaye
expect(visible).toBe(true); // FAIL — toast abhi ready nahi tha

// ✅ Correct — toBeVisible() handles timing automatically
await loginBtn.click();
await expect(toast).toBeVisible(); // 500ms ke baad appear hua → PASS ✅
```

| | `isVisible()` | `toBeVisible()` |
|---|---|---|
| Return type | `Promise<boolean>` | `Promise<void>` |
| Auto-retry | No | Yes — ~100ms polling |
| Timeout | None | Default 5s |
| On fail | `false` return | Test fail + error |
| Use case | if/else | Test assertion |

> **Golden rule:** Test assertion mein kabhi `isVisible()` mat use karo — timing issues guarantee hain.

## Q4. What is the return type of boolean state methods, auto-retry assertions, and non-retrying assertions?

```typescript
// BOOLEAN STATE METHODS
locator.isVisible()    // → Promise<boolean>  — await zaroori
locator.isHidden()     // → Promise<boolean>
locator.isEnabled()    // → Promise<boolean>
locator.isDisabled()   // → Promise<boolean>
locator.isChecked()    // → Promise<boolean>
locator.isEditable()   // → Promise<boolean>

// AUTO-RETRY ASSERTIONS (Web-First)
expect(locator).toBeVisible()         // → Promise<void>  — await zaroori
expect(locator).toHaveText('x')       // → Promise<void>
expect(locator).toHaveValue('x')      // → Promise<void>
expect(page).toHaveURL('/dashboard')  // → Promise<void>

// NON-RETRYING ASSERTIONS
expect(5).toBe(5)                  // → void  — synchronous — await NAHI lagta
expect({}).toEqual({})             // → void
expect([]).toHaveLength(3)         // → void
expect('text').toMatch(/regex/)    // → void
```

**Key distinction yaad rakho:**
- Boolean → `Promise<boolean>` → await lagao, value milti hai
- Web-First → `Promise<void>` → await lagao, value nahi milti, ya pass ya fail
- Non-Retrying → `void` (synchronous) → await nahi lagta, instant check

## Q5. Explain the internal working mechanism of auto-retry assertions — polling time, default timeout. How to increase globally and for a specific assertion?

**Internal mechanism:**
```
expect(locator).toBeVisible() call hoti hai
        ↓
DOM check → condition true? NO
        ↓ (~100ms wait)
DOM check → condition true? NO
        ↓ (~100ms wait)
DOM check → condition true? YES → PASS ✅

      --- YA ---

5 seconds guzar gaye, kabhi true nahi hua
        ↓
TimeoutError — "Locator expected to be visible"
+ screenshot + trace automatically attach
```

~100ms polling Playwright ka internal default hai — directly change nahi hoti. Jo configure hota hai woh maximum wait time (timeout) hai.

**Default timeout:** 5000ms (5 seconds)

```typescript
// GLOBAL INCREASE — playwright.config.ts mein
export default defineConfig({
  expect: {
    timeout: 10000  // sab assertions ke liye 10 seconds
  }
});

// SPECIFIC ASSERTION ke liye override
await expect(
  page.getByTestId('heavy-report-table')
).toBeVisible({ timeout: 30000 }); // sirf is ek assertion ke liye 30s

// Content assertion ka bhi timeout hota hai
await expect(heading).toHaveText('Dashboard', { timeout: 8000 });
```

## Q6. If assertion timeout is 60 seconds and test timeout is 30 seconds — which takes preference?

**Test timeout preference lega — test pehle khatam ho jaayega.**

Test timeout ek hard ceiling hai. Agar test ka overall timeout 30s hai aur assertion 60s wait karna chahti hai — test 30s pe terminate ho jaayega. Assertion poora wait nahi kar paayegi.

```typescript
// playwright.config.ts
export default defineConfig({
  timeout: 30000,         // test level — 30s
  expect: {
    timeout: 60000        // assertion level — 60s
  }
});
// Result: assertion 30s ke baad test timeout se fail hogi
// 60s ka chance kabhi nahi milega

// FIX — test timeout badhao agar assertion ko 60s chahiye
test('slow report test', async ({ page }) => {
  test.setTimeout(90000); // is specific test ke liye 90s

  await expect(page.getByTestId('report')).toBeVisible({ timeout: 60000 });
});
```

> **Rule:** Assertion timeout ≤ Test timeout hona chahiye. Warna assertion timeout meaningless hai.

## Q8. What is the difference between toHaveValue(), toHaveText(), and toContainText()?

```typescript
// toHaveValue() — input / select / textarea ka CURRENT VALUE
// DOM property padh ta hai — jo user ne type kiya ya jo set hai
// Non-visible elements pe bhi kaam karta hai
await expect(page.getByLabel('Email')).toHaveValue('gaurav@test.com');
await expect(page.getByLabel('Email')).toHaveValue(/gaurav/i); // regex bhi
// Returns: Promise<void>
// Use: form fill ke baad verify karo, dropdown selected value

// toHaveText() — element ka VISIBLE INNERTEXT — EXACT match
// Jo user screen pe dekh sakta hai
// String, RegEx, ya String[] (multiple elements ke liye)
await expect(page.getByRole('heading')).toHaveText('Welcome Back');
await expect(page.getByRole('heading')).toHaveText(/Welcome/i);
await expect(page.getByRole('listitem')).toHaveText(['Home','About','Contact']); // ordered
// Returns: Promise<void>
// Use: headings, paragraphs, button text, list items

// toContainText() — PARTIAL text match kaafi hai
// Exact pura text yaad nahi ya dynamic part ho toh
await expect(page.getByTestId('greeting')).toContainText('Gaurav');
// "Welcome back, Gaurav! You have 3 messages" → PASS ✅
// Returns: Promise<void>

// COMPARISON:
// Element text: "Welcome back, Gaurav!"
// toHaveText('Welcome back, Gaurav!')  → ✅ exact match
// toHaveText('Welcome')               → ❌ FAIL — partial nahi chalega
// toContainText('Welcome')            → ✅ partial enough

// toHaveValue vs toHaveText — input pe:
// toHaveValue('gaurav@test.com')  → input ka value property
// toHaveText('gaurav@test.com')   → input pe nahi chalta — use toHaveValue
```

## Q10. What is the difference between hard assertions and soft assertions? Purpose, implementation, report difference?

**Hard Assertion** — pehli failure pe test immediately stop ho jaata hai. Normal `expect()`.

**Soft Assertion** — failure ke baad test continue karta hai. Saari failures end mein ek saath report.

```typescript
// HARD ASSERTION — pehli failure pe STOP
await expect(page.getByLabel('Name')).toHaveValue('Gaurav');    // FAIL → STOP
await expect(page.getByLabel('Email')).toHaveValue('g@t.com');  // Never runs
await expect(page.getByLabel('Phone')).toHaveValue('9999');     // Never runs
// Report mein: sirf 1 failure dikhegi

// SOFT ASSERTION — failure ke baad CONTINUE
await expect.soft(page.getByLabel('Name'),  'Name field').toHaveValue('Gaurav');   // FAIL, continue
await expect.soft(page.getByLabel('Email'), 'Email field').toHaveValue('g@t.com'); // FAIL, continue
await expect.soft(page.getByLabel('Phone'), 'Phone field').toHaveValue('9999');    // PASS, continue
// Report mein: 2 failures ek saath dikhenge

// IMPLEMENTATION DIFFERENCE
expect(locator).toBeVisible()               // hard — test stops on fail
expect.soft(locator).toBeVisible()          // soft — test continues
expect.soft(locator, 'message').toBeVisible() // soft with custom message
```

**Report difference:**
- Hard assertion ke saath — ek failure, baaki assertions ka pata hi nahi chalega. Dobara run karo, fix karo, run karo — tedious cycle.
- Soft assertions ke saath — ek run mein poori form ki saari problems pata chal jaati hain.

**Kab kya use karo:**
- **Hard** → Critical path pe — login successful hua ya nahi, correct page pe land hua ya nahi. Failure ke baad aage karna meaningless ho.
- **Soft** → Validation checks — form fields, dashboard items, saare independent fields ek run mein check karne hoon.

## Q11. What is the difference between toBe() and toEqual()?

```typescript
// toBe() — Strict equality (===)
// Primitives ke liye — numbers, strings, booleans
// No await — synchronous, void return

expect(5).toBe(5)               // ✅ same primitive value
expect('active').toBe('active') // ✅
expect(true).toBe(true)         // ✅

// toEqual() — Deep equality
// Objects aur arrays ke liye — structure compare karta hai
// No await — synchronous, void return

expect({ name: 'Gaurav' }).toEqual({ name: 'Gaurav' }) // ✅ same structure
expect(['a', 'b']).toEqual(['a', 'b'])                 // ✅ same order + values

// WHY toBe() fails on objects:
expect({ a: 1 }).toBe({ a: 1 })    // ❌ FAIL
// Kyun? JavaScript mein {} === {} false hai — alag memory references
// toBe() reference check karta hai, structure nahi

expect({ a: 1 }).toEqual({ a: 1 }) // ✅ PASS
// toEqual() structure/value compare karta hai — deep equality

// RULE: Number/string/boolean → toBe()
//       Object/array → toEqual()

// Real automation use case:
const body = await response.json();
expect(body.status).toBe(200);         // primitive — toBe
expect(body.user).toEqual({            // object — toEqual
  name: 'Gaurav',
  role: 'admin'
});
```

## Q12. What do you understand by truthiness and falsiness? Explain with examples.

JavaScript mein har value ya toh **truthy** hai ya **falsy** — boolean context mein true ya false treat hoti hai.

- **Falsy values:** `false`, `0`, `''` (empty string), `null`, `undefined`, `NaN`
- **Truthy values:** Baaki sab — non-empty strings, non-zero numbers, objects, arrays (chahe empty ho), functions

```typescript
// toBeTruthy() — value truthy hai
// No await — synchronous, void return
expect('Gaurav').toBeTruthy()    // ✅ non-empty string
expect(1).toBeTruthy()           // ✅ non-zero number
expect({}).toBeTruthy()          // ✅ object — even empty object is truthy!
expect([]).toBeTruthy()          // ✅ array — even empty array is truthy!
expect(true).toBeTruthy()        // ✅

// toBeFalsy() — value falsy hai
expect('').toBeFalsy()           // ✅ empty string
expect(0).toBeFalsy()            // ✅ zero
expect(null).toBeFalsy()         // ✅
expect(undefined).toBeFalsy()    // ✅
expect(false).toBeFalsy()        // ✅
expect(NaN).toBeFalsy()          // ✅

// toBeNull() — exactly null
expect(null).toBeNull()          // ✅
expect(undefined).toBeNull()     // ❌ FAIL — undefined null nahi hai

// toBeUndefined() — exactly undefined
expect(undefined).toBeUndefined() // ✅
expect(null).toBeUndefined()      // ❌ FAIL

// toBeDefined() — not undefined
expect('value').toBeDefined()     // ✅
expect(null).toBeDefined()        // ✅ — null defined hai, undefined nahi
expect(undefined).toBeDefined()   // ❌ FAIL

// Automation mein use case:
const token = await getAuthToken();
expect(token).toBeTruthy();       // token exist karta hai aur empty nahi
// '' empty string bhi fail karegi — meaningful check

const errorMsg = await page.getByTestId('error').textContent();
expect(errorMsg).toBeFalsy();     // null ya empty string — no error shown
```

## Q13. How do you validate primitive values, arrays, objects, length, content, and response time?

```typescript
// ── PRIMITIVE VALUES — toBe(), synchronous, no await ──
const status = await response.status();    // number
const statusText = await response.text();  // string

expect(status).toBe(200)                   // strict ===, void
expect(statusText).toBe('OK')
expect(isLoggedIn).toBe(true)

// ── ARRAY AND OBJECT VALUES — toEqual() ───────────────
const body = await response.json();

expect(body.user).toEqual({                // deep equality, void
  name: 'Gaurav',
  role: 'admin'
})

const texts = await page.locator('li').allInnerTexts();
expect(texts).toEqual(['Home', 'About', 'Contact'])  // exact order + values

// ── ARRAY AND STRING LENGTH — toHaveLength() ──────────
const results = await page.locator('.result').all();
expect(results).toHaveLength(10)            // array length, void

expect('Playwright').toHaveLength(10)       // string length, void

const allTexts = await page.locator('li').allInnerTexts();
expect(allTexts).toHaveLength(5)            // returned string array length

// ── ARRAY AND STRING CONTENT — toContain() ────────────
const roles = ['admin', 'user', 'viewer'];
expect(roles).toContain('admin')            // array mein item hai, void

expect('Welcome back Gaurav').toContain('Gaurav') // string mein substring, void

// ── RESPONSE TIME COMPARISON ──────────────────────────
const startTime = Date.now();
await page.goto('/dashboard');
const responseTime = Date.now() - startTime;

expect(responseTime).toBeLessThan(3000)         // 3 seconds se kam, void
expect(responseTime).toBeGreaterThan(0)          // greater than zero
expect(responseTime).toBeGreaterThanOrEqual(100)
expect(responseTime).toBeLessThanOrEqual(5000)

// API response time:
const start = Date.now();
const response = await request.get('/api/users');
const duration = Date.now() - start;
expect(duration).toBeLessThan(2000)         // API 2 seconds se fast honi chahiye

// ── YEH SAB ASYNC HAI YA NAHI? ───────────────────────
// Non-retrying assertions (toBe, toEqual, toHaveLength, etc.) → SYNCHRONOUS
// No await needed
// But value extract karna async hai:
const count = await page.locator('li').count();  // await — async DOM call
expect(count).toBe(5);                           // no await — sync check
```

## Q14. How does Playwright prevent flakiness? How is Playwright assertion different from Selenium?

**Playwright flakiness kaise rokta hai:**

```typescript
// SELENIUM approach — manual waits, error-prone
driver.findElement(By.id('btn')).click();
Thread.sleep(2000);  // hard sleep — always waits 2s even if ready in 100ms
// Agar 2s enough nahi — test fail, agar zyada — test slow

// PLAYWRIGHT approach — smart waiting
await page.getByRole('button').click();
// Before click: 5 actionability checks automatic:
// 1. Attached   — element DOM mein hai
// 2. Visible    — display:none nahi
// 3. Stable     — animate nahi ho raha (position stable)
// 4. Enabled    — disabled attribute nahi
// 5. Not Covered — koi doosra element overlap nahi kar raha
// Sab pass → click execute. Koi wait likhna nahi pada.

// ASSERTION level flakiness prevention:
await expect(successMsg).toBeVisible();
// Selenium equivalent: custom ExpectedConditions likh karo ya Thread.sleep
// Playwright: built-in polling, no extra code
```

**5 Key Differences — Playwright vs Selenium:**

| | Playwright | Selenium |
|---|---|---|
| Auto-wait | Built-in — every action | Manual — explicit waits |
| Actionability | 5 checks automatic | No built-in concept |
| Assertion retry | `toBeVisible()` auto-polls | Custom `WebDriverWait` likhna padta |
| Flaky tests | Rare — smart waiting | Common — timing issues |
| Error messages | Screenshot + trace automatic | Basic error string |

```typescript
// Playwright — clean, no manual waits
await loginBtn.click();
await expect(page).toHaveURL('**/dashboard');    // auto-waits for navigation
await expect(welcomeMsg).toBeVisible();           // auto-waits for element
await expect(welcomeMsg).toHaveText('Welcome');   // auto-waits for text

// Selenium equivalent:
loginBtn.click();
wait.until(ExpectedConditions.urlContains("dashboard")); // manual
wait.until(ExpectedConditions.visibilityOf(welcomeMsg)); // manual
wait.until(ExpectedConditions.textToBe(welcomeMsg, "Welcome")); // manual
// Har step pe explicit wait — verbose, error-prone
```

**Why Playwright assertions are superior:**
- Auto-retry built-in — koi `waitForXxx` likhna nahi padta
- Detailed failure messages — screenshot, trace, DOM snapshot automatic
- `.not.` chaining — negation assertions bhi retry karti hain
- Timeout configurable globally ya per-assertion
- Soft assertions — ek run mein saari failures
