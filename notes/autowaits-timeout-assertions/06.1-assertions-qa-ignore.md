# 🎭 Playwright Assertions — Interview Questions & Answers

Real scenarios ke saath, senior SDET / Automation Architect interviews ke liye.

---

## Q1: What are the two types of assertions in Playwright? What is the fundamental difference?

**Answer:**

Do types hain: **Auto-retrying (web-first) assertions** aur **Non-retrying (generic) assertions**.

| Type | Behavior | Await | Example |
|------|----------|-------|---------|
| **Auto-retrying** | Tab tak retry karti hain jab tak condition pass na ho ya assertion timeout (5s default) hit na ho. | ✅ Zaroori | `expect(locator).toBeVisible()` |
| **Non-retrying** | Ek baar evaluate karti hain — pass ya fail, immediately. | ❌ | `expect(value).toBe(42)` |

Web pages asynchronous hain isliye auto-retrying assertions exist karti hain. Non-retrying pure JavaScript values ke liye hain jahan timing matter nahi karta.

**Most common interview trap:**

```typescript
// ❌ Wrong — isVisible() non-retrying hai
expect(await page.locator('#el').isVisible()).toBe(true);

// ✅ Correct — toBeVisible() auto-retrying hai
await expect(page.locator('#el')).toBeVisible();
```

> Dono same lagti hain but behavior completely alag hai. Pehli wali ek baar check karti hai — element 2 seconds baad aane wala ho to fail. Doosri wali 5 seconds tak retry karti hai.

## Q2: What are soft assertions? When would you use them in a real project?

**Answer:**

Soft assertions fail hone pe test **continue** karne deti hain — stop nahi hota. End mein saari failures compile hokar report mein aati hain.

```typescript
await expect.soft(page.locator('#title')).toHaveText('Dashboard');
await expect.soft(page.locator('#user')).toContainText('Prince');
await expect.soft(page.locator('#status')).toBeVisible();
expect(test.info().errors).toHaveLength(0);
```

**Real use cases:**

- **Form validation testing** — ek empty form submit karo, sab validation errors ek run mein verify karo. Hard assertions se sirf pehli error dikhti.
- **Smoke test across multiple page sections** — header, footer, navigation, main content — ek page pe 10 cheezein verify karni hain. Ek failure se baaki 9 skip nahi hone chahiye.
- **ETS scenario** — assessment results page pe score, grade, breakdown, feedback — sab ek page pe. Soft assertions se ek run mein poori page health milti thi.

**When NOT to use:**
- Jab assertions dependent hain — ek pass hona zaroori ho doosre ke liye.
- Jab failure critical ho — login fail hua to test stop hona chahiye, aage proceed karna wrong hoga.

## Q3: What is the difference between `toHaveText()` and `toContainText()`?

**Answer:**

- **`toHaveText()`** — exact match karta hai. Element ka poora text expected value se match karna chahiye (whitespace normalize hota hai).
- **`toContainText()`** — substring match karta hai. Expected text element ke text mein anywhere exist karna chahiye.

```typescript
// Element text: "Welcome, Prince! You have 3 notifications."

// ❌ toHaveText — fail karega kyunki exact match nahi
await expect(locator).toHaveText('Welcome');

// ✅ toContainText — pass karega
await expect(locator).toContainText('Welcome');
await expect(locator).toContainText('3 notifications');

// ✅ toHaveText exact
await expect(locator).toHaveText('Welcome, Prince! You have 3 notifications.');

// Regex ke saath flexible
await expect(locator).toHaveText(/Welcome, \w+!/);
```

**Multiple elements ke liye dono array accept karte hain:**

```typescript
// toHaveText — exact match each element
await expect(page.locator('li')).toHaveText(['Item 1', 'Item 2', 'Item 3']);

// toContainText — substring match each element
await expect(page.locator('li')).toContainText(['Item', 'Item', 'Item']);
```

## Q4: When would you use `expect.poll()` vs `expect.toPass()`? What is the difference?

**Answer:**

| Method | Use When | Returns |
|--------|----------|---------|
| `expect.poll()` | Ek function repeatedly call karna hai jo **single value** return kare, phir us value pe matcher run karo. | Single value → matcher |
| `expect.toPass()` | Ek **async block** repeatedly retry karna hai jab tak andar ki saari assertions pass na ho jayein. | Multiple assertions block |

```typescript
// expect.poll() — single value, external state check
await expect.poll(async () => {
  const res = await page.request.get('/api/job/' + jobId);
  return (await res.json()).status;
}, { timeout: 30000 }).toBe('completed');

// toPass() — multiple assertions block, complex condition
await expect(async () => {
  await expect(page.locator('#status')).toHaveText('Done');
  await expect(page.locator('#result-count')).not.toHaveText('0');
  await expect(page.locator('.error-msg')).not.toBeVisible();
}).toPass({ timeout: 20000 });
```

> ⚠️ **Important:** `toPass()` ka default timeout `0` hai — **explicitly specify karo** nahi to test timeout se bound ho jaayega silently. `expect.poll()` ka default 5 seconds hai.

**Rule of thumb:** Single value return kar sakte ho aur ek matcher se check ho sakta hai — `poll()`. Multiple independent assertions ek saath pass honi chahiye — `toPass()`.

## Q5: What is wrong with this code and how do you fix it?

```typescript
// Code given in interview
expect(await page.locator('#status').isVisible()).toBe(true);
```

**Answer:**

**Do problems hain:**

1. **`isVisible()` non-retrying hai.** Agar `#status` element async operation ke baad appear hota hai, yeh immediately `false` return karega aur test fail ho jaayega — even though element 500ms baad aane wala tha.
2. **Playwright best practices ke against hai.** Official docs specifically yeh pattern avoid karne ko kehti hain.

```typescript
// ❌ Problem
expect(await page.locator('#status').isVisible()).toBe(true);

// ✅ Fix — auto-retrying assertion
await expect(page.locator('#status')).toBeVisible();

// ✅ Fix with custom timeout if needed
await expect(page.locator('#status')).toBeVisible({ timeout: 10000 });

// ✅ Fix with message for better reporting
await expect(
  page.locator('#status'),
  'Status element should be visible after form submit'
).toBeVisible();
```

**`isVisible()` ka valid use case** sirf conditional logic mein hai:

```typescript
// Conditional — non-retrying appropriate hai
if (await page.locator('#cookie-banner').isVisible()) {
  await page.locator('#accept-cookies').click();
}
```

## Q6: How does assertion timeout differ from action timeout?

**Answer:**

| Timeout | Config Key | Default | Used For |
|---------|------------|---------|----------|
| **Action timeout** | `use.actionTimeout` | 0 (test timeout se bound) | `click()`, `fill()`, `hover()` — actionability checks |
| **Assertion timeout** | `expect.timeout` | **5 seconds** | `expect().toBeVisible()` — retry window |

```typescript
// playwright.config.ts
export default defineConfig({
  timeout: 60000,              // test: 60s
  expect: { timeout: 10000 }, // assertions: 10s (default 5s se zyada)
  use: {
    actionTimeout: 15000,      // actions: 15s (default 0 se zyada)
  }
});
```

> **Common confusion 1:** Koi sochta hai action timeout assert karne pe bhi apply hota hai — **wrong**. Assertion ka apna separate timeout hai. Dono independently configure hote hain.

> **Common confusion 2:** Assertion timeout `0` karo to kya hoga? Assertion ek baar check karega aur fail kar dega — non-retrying assertion ki tarah behave karega. **Never do this.**

## Q7: How do you structure assertions in a Page Object Model?

> Architecture level question — senior/lead roles ke liye.

**Answer:**

Page Object mein do approaches hain — dono ka proper use case hai.

**Approach 1 — Assertions POM ke bahar (recommended for most cases):**

```typescript
// pages/DashboardPage.ts
export class DashboardPage {
  constructor(private page: Page) {}

  // Locators expose karo
  get heading() { return this.page.locator('h1.dashboard-title'); }
  get userMenu() { return this.page.locator('#user-menu'); }
  get notificationCount() { return this.page.locator('#notif-count'); }

  // Actions
  async navigate() {
    await this.page.goto('/dashboard');
  }

  async openUserMenu() {
    await this.userMenu.click();
  }
}

// test file mein — assertions yahan
test('dashboard loads correctly', async ({ page }) => {
  const dashboard = new DashboardPage(page);
  await dashboard.navigate();

  await expect(dashboard.heading).toHaveText('Dashboard');
  await expect(dashboard.userMenu).toBeVisible();
  await expect(page).toHaveURL(/.*dashboard.*/);
});
```

**Approach 2 — Assertions POM mein (specific scenarios ke liye):**

```typescript
export class DashboardPage {
  // Reusable verification methods
  async verifyPageLoaded() {
    await expect(this.heading).toBeVisible();
    await expect(this.page).toHaveURL(/.*dashboard.*/);
    await expect(this.page).toHaveTitle(/Dashboard/);
  }

  async verifyUserLoggedIn(userName: string) {
    await expect(this.userMenu).toContainText(userName);
  }
}

// test file
test('dashboard', async ({ page }) => {
  const dashboard = new DashboardPage(page);
  await dashboard.navigate();
  await dashboard.verifyPageLoaded();           // reusable
  await dashboard.verifyUserLoggedIn('Prince'); // reusable
});
```

> **Trade-off:** Assertions POM mein daalne se reuse hoti hain but tests less readable hote hain. Hard assertion POM mein fail hui to test stop — might not give full picture.
>
> **Balance:** Generic `verifyPageLoaded()` POM mein, specific business assertions test mein.

## Q8: You see intermittent test failures only on CI, not locally. Assertions are failing. What do you suspect and how do you fix?

> Senior level debugging question.

**Answer:**

CI pe intermittent assertion failures almost always ek cause se hote hain — **timing**. CI server slower hota hai local machine se, network latency zyada hoti hai.

**Step 1 — Check karo non-retrying assertion toh nahi use ki DOM element pe:**

```typescript
// Yeh CI pe fail karega kyunki element load hone mein time lagta hai
expect(await page.locator('#data').isVisible()).toBe(true);
// Fix:
await expect(page.locator('#data')).toBeVisible();
```

**Step 2 — Assertion timeout badhao for CI:**

```typescript
// playwright.config.ts
export default defineConfig({
  expect: {
    timeout: process.env.CI ? 15000 : 5000 // CI pe zyada time
  }
});
```

**Step 3 — `waitForTimeout` remove karo — yeh CI pe scale nahi karta:**

```typescript
// ❌ Local pe works, CI pe fail
await page.waitForTimeout(2000);
await expect(page.locator('#result')).toBeVisible();

// ✅ Deterministic
await expect(page.locator('#result')).toBeVisible({ timeout: 10000 });
```

**Step 4 — Trace enable karo CI pe first retry pe — exact failure point aur timeline milega:**

```typescript
use: {
  trace: 'on-first-retry',
  screenshot: 'only-on-failure',
}
```

**Step 5 — Flaky test annotation temporarily lagao while fixing:**

```typescript
test.fixme('flaky on CI - timing issue', async ({ page }) => {
  // investigating
});
```

## Q9: What is the difference between `toHaveURL()` and `waitForURL()`? When do you use which?

**Answer:**

| Method | Type | Purpose |
|--------|------|---------|
| `waitForURL()` | Navigation wait | Page object pe — URL change hone ka wait. Ek baar URL match → resolve. |
| `toHaveURL()` | Assertion | Current URL expected hai verify karo. Auto-retrying — 5s tak retry. |

```typescript
// waitForURL — navigation flow mein use karo
await page.click('#login');
await page.waitForURL('**/dashboard'); // wait karo redirect ke liye
// Ab dashboard pe actions karo

// toHaveURL — assertion/verification ke liye use karo
await expect(page).toHaveURL('https://app.example.com/dashboard');
await expect(page).toHaveURL(/.*\/dashboard/);
```

**Real test mein dono saath use karte hain:**

```typescript
await page.click('#login');
await page.waitForURL('**/dashboard');          // navigation complete hone ka wait
await expect(page).toHaveURL(/.*dashboard.*/); // explicit assertion for reporting
await expect(page.locator('h1')).toHaveText('Dashboard'); // content verify
```

> **Why both?** `waitForURL` navigation wait karta hai. `toHaveURL` test report mein clear assertion dikhta hai — failure message better hota hai.

## Q10: How do you handle assertion for elements that may or may not exist on a page?

> Real world scenario — optional UI elements.

**Answer:**

Optional elements ke liye **conditional logic** use karo — non-retrying `isVisible()` appropriate hai.

```typescript
// Scenario: Cookie consent banner — kuch environments mein aata hai, kuch mein nahi

// Approach 1 — conditional check
const cookieBanner = page.locator('#cookie-consent');
if (await cookieBanner.isVisible()) {
  await page.locator('#accept-all').click();
  await expect(cookieBanner).toBeHidden(); // dismiss hone ka assert
}

// Approach 2 — fixture mein handle karo (framework level)
// base-fixtures.ts
const test = base.extend({
  page: async ({ page }, use) => {
    await page.goto(BASE_URL);
    const cookieBanner = page.locator('#cookie-consent');
    if (await cookieBanner.isVisible({ timeout: 2000 })) {
      await page.click('#accept-all');
    }
    await use(page);
  }
});

// Approach 3 — soft assertion for optional elements
await expect.soft(page.locator('#promo-banner')).toBeVisible();
// Fail hoga but test continue karega — promo banner optional feature hai
```

> ⚠️ **Mistake jo avoid karni hai:**
> ```typescript
> // ❌ toBeVisible() wait karega 5s — tab bhi element na aaya to fail
> // Optional elements ke liye wrong — har environment pe fail karega
> await expect(page.locator('#optional-widget')).toBeVisible();
> ```

## Quick Revision — Assertion Interview Ke Liye

| Topic | Key Point |
|-------|-----------|
| **Two types** | Auto-retrying (web-first, locator pe, `await` zaroori, 5s default retry) vs Non-retrying (generic, pure values pe, immediate). |
| **Most common mistake** | `expect(await locator.isVisible()).toBe(true)` — always fix to `await expect(locator).toBeVisible()`. |
| **Soft assertion rule** | Multiple independent checks ek page pe → soft. Dependent flow → hard. |
| **`toPass` timeout** | Default is `0` — explicitly specify karo, otherwise test timeout se bound hoga silently. |
| **`toHaveText` vs `toContainText`** | Exact vs substring — regex dono accept karte hain. |
| **`expect.poll` vs `toPass`** | Single value return → `poll()`. Multiple assertions block → `toPass()`. |
| **API assertions** | `response.status()`, `response.ok()`, `response.json()` — non-retrying fine hai kyunki response already resolved hai. |
