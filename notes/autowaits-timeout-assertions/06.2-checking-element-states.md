# Topic 8 — Checking Element States

## Pehle Samjho — Yeh Topic Kya Hai Actually

Real app mein kuch scenarios socho:

- Login button click kiya → button disabled ho gaya (loading) → phir success message aaya
- Form submit kiya → spinner aaya → spinner gaya → data table appear hua
- Cookie banner kabhi aata hai kabhi nahi → agar aaye toh dismiss karo

Playwright mein element ka "state" check karna matlab — visible hai? enabled hai? checked hai? value kya hai? Yeh sab check karne ke liye do alag approaches hain aur dono ka purpose alag hai. Yahi confusion hai jo clear karni hai.

---

## PART 1 — Boolean State Methods

### Purpose
Yeh methods ek simple sawaal ka jawab dete hain — "Is waqt yeh element ka state kya hai?"
Jawab milta hai `true` ya `false`. Ek baar check, koi retry nahi, instant result.

### Kahan milte hain — Locator pe
Yeh `page` pe nahi hote, `Locator` pe hote hain:

```typescript
const btn = page.getByRole('button', { name: 'Submit' });
await btn.isVisible();   // ✅ Locator pe
await btn.isEnabled();   // ✅ Locator pe

await page.isVisible();  // ❌ page pe nahi hota
```

### Kitne available hain aur har ek kya validate karta hai

```typescript
locator.isVisible()   → Promise<boolean>
// Element DOM mein hai AND visible hai (no display:none, no visibility:hidden, no opacity:0)
// Hidden element pe → false

locator.isHidden()    → Promise<boolean>
// isVisible() ka opposite — hidden hai ya DOM mein nahi hai → true

locator.isEnabled()   → Promise<boolean>
// Element pe 'disabled' attribute nahi hai → true
// Button, input, select pe kaam karta hai

locator.isDisabled()  → Promise<boolean>
// isEnabled() ka opposite — 'disabled' attribute hai → true

locator.isChecked()   → Promise<boolean>
// Checkbox ya radio button checked hai ya nahi
// Sirf checkboxes aur radio buttons pe meaningful hai

locator.isEditable()  → Promise<boolean>
// NOT disabled AND NOT readonly — dono conditions satisfy honi chahiye
// Ek bhi ho toh false
```

### Kab use karo — if/else conditional logic mein

```typescript
// Scenario 1 — Cookie banner kabhi aata hai kabhi nahi
const banner = page.getByTestId('cookie-banner');
if (await banner.isVisible()) {
  await page.getByRole('button', { name: 'Accept' }).click();
}
// Test aage badhega regardless — banner tha ya nahi

// Scenario 2 — Optional form field
const middleName = page.getByLabel('Middle Name');
if (await middleName.isVisible()) {
  await middleName.fill('Kumar');
}

// Scenario 3 — Button state check before clicking
const deleteBtn = page.getByRole('button', { name: 'Delete' });
if (await deleteBtn.isEnabled()) {
  await deleteBtn.click();
}
```

### ⚠️ Yeh GALAT use hai — test assertion ke liye mat use karo

```typescript
// ❌ NEVER DO THIS
const visible = await page.getByText('Success').isVisible();
expect(visible).toBe(true);
// Kyu galat? Ek baar check kiya — DOM us waqt update ho raha tha → false mila → test fail
// Timing issue — isko flaky test kehte hain
```

## PART 2 — Web-First Assertions

### Purpose
Boolean methods ne bataya ki abhi kya hai. Web-First Assertions kehte hain — "Yeh condition true honi chahiye, main wait karunga."

Playwright baar baar DOM check karta rehta hai jab tak condition true na ho ya timeout na ho. Isko **polling** kehte hain. Isliye yeh handle karte hain:

- Spinner aaya phir gaya — `expect(spinner).toBeHidden()` wait karega jab tak spinner disappear na ho
- Success message delay se aata hai — `expect(toast).toBeVisible()` wait karega
- Button loading ke baad enable hota hai — `expect(btn).toBeEnabled()` wait karega

Yahi hai flaky tests ka solution. Tum `waitForTimeout(2000)` nahi likhte — assertion khud wait karti hai.

### Return type

```typescript
expect(locator).toBeVisible()  → Promise<void>
// void — matlab koi value nahi milti
// Ya pass ho jaata hai ya test fail ho jaata hai
```

### Polling mechanism — ekdum clear

```
expect(locator).toBeVisible() call hoti hai
         ↓
DOM check → visible? NO
         ↓ ~100ms wait
DOM check → visible? NO
         ↓ ~100ms wait
DOM check → visible? YES ✅ → PASS, test aage badhta hai

         --- YA ---

5 seconds guzar gaye, kabhi YES nahi aaya
         ↓
TimeoutError → test FAIL
"Locator expected to be visible"
+ screenshot + trace automatically
```

### 100ms polling — fixed hai ya change ho sakti hai?
100ms approximate internal default hai, change nahi hoti directly. Jo change hoti hai woh **timeout** hai — kitna wait karunga max.

```typescript
// Default timeout — 5 seconds
// Kahan specified hota hai — playwright.config.ts mein
expect: {
  timeout: 5000   // 5 seconds — sab assertions ke liye
}

// Increase karo globally
expect: {
  timeout: 10000  // 10 seconds
}

// Sirf ek specific assertion ke liye override
await expect(page.getByTestId('report')).toBeVisible({ timeout: 30000 });
// Yeh sirf is ek assertion ke liye 30 seconds wait karega
```

### Spinner, loading, delayed elements — kaise handle hote hain

```typescript
// Scenario: Login click → spinner aata hai → spinner jaata hai → dashboard aata hai

await page.getByRole('button', { name: 'Login' }).click();

// Spinner appear hone ka wait
await expect(page.getByTestId('spinner')).toBeVisible();
// Spinner disappear hone ka wait
await expect(page.getByTestId('spinner')).toBeHidden();
// Dashboard appear hone ka wait
await expect(page.getByRole('heading', { name: 'Dashboard' })).toBeVisible();

// Playwright baar baar check karta hai — tum sirf declare karte ho "yeh hona chahiye"
```

## PART 3 — State Assertions — Complete List

### Yeh kya hain
`expect(locator).toBe___()` form ki assertions. Yeh web-first assertions hain — matlab inme bhi auto-retry hota hai, same 100ms polling, same default 5s timeout.

### Most Important ones — Zaroor Jaano

```typescript
// Visibility — sabse zyada use hoti hain
await expect(locator).toBeVisible()      // element dikhna chahiye
await expect(locator).toBeHidden()       // element nahi dikhna chahiye
await expect(locator).not.toBeVisible()  // same as toBeHidden — dono sahi hain

// Enable/Disable — form interactions pe
await expect(locator).toBeEnabled()      // button click karne se pehle check
await expect(locator).toBeDisabled()     // loading state mein button disabled hai

// Input state
await expect(locator).toBeEditable()     // field mein type kar sakte hain
await expect(locator).toBeEmpty()        // input value ya text empty hai
await expect(locator).toBeChecked()      // checkbox selected hai
await expect(locator).not.toBeChecked()  // checkbox unselected hai
```

### Good to Remember — Kam Use Hoti Hain

```typescript
await expect(locator).toBeFocused()      // keyboard focus is element pe hai
// Use case: Tab key navigation testing, accessibility testing
```

---

### Value aur Content Assertions

#### Inme auto-retry hota hai ya nahi?
Haan, inme bhi auto-retry hota hai — yeh bhi web-first assertions hain. Locator ke saath hain toh polling hogi.

#### Yeh kya validate karte hain

```typescript
// toHaveValue — input field ka current value
// Kahan kaam karta hai: input, textarea, select
// Use case: fill karne ke baad verify karo jo type kiya wahi hai

await expect(page.getByLabel('Email')).toHaveValue('gaurav@test.com')
await expect(page.getByLabel('Email')).toHaveValue(/gaurav/)  // regex pattern bhi
// Return: Promise<void>

// toHaveText — element ka visible text (innerText)
// Kahan kaam karta hai: any element — heading, paragraph, button, div
// Use case: page pe correct content show ho raha hai verify karo

await expect(page.getByRole('heading')).toHaveText('Welcome Back')
await expect(page.getByRole('heading')).toHaveText(/Welcome/i)  // regex, case-insensitive
// Return: Promise<void>

// toContainText — partial text match
// Use case: jab exact pura text yaad na ho ya dynamic part ho

await expect(page.getByTestId('user-greeting')).toContainText('Gaurav')
// "Welcome back, Gaurav! You have 3 messages" → pass karega

// toHaveText — ARRAY FORM — multiple elements ek saath
// Use case: list ke saare items verify karo ek assertion mein

await expect(page.getByRole('listitem')).toHaveText(['Home', 'About', 'Contact'])
// Order matter karta hai — first item 'Home' hona chahiye, second 'About', etc.
// Return: Promise<void>

// toHaveCount — multiple elements ka count
await expect(page.getByRole('listitem')).toHaveCount(5)
// Return: Promise<void>
```

#### toHaveText vs toContainText — ek baar ke liye seedha

```
Element ka actual text: "Welcome back, Gaurav!"

toHaveText('Welcome back, Gaurav!')  → ✅ exact match
toHaveText('Welcome')                → ❌ FAIL — partial nahi chalega
toContainText('Welcome')             → ✅ partial enough
toContainText('Gaurav')              → ✅ yeh bhi
```

### Attribute aur CSS Assertions

#### Inme auto-retry hota hai?
Haan — yeh bhi `expect(locator).` se hain, web-first hain, auto-retry hoga.

#### Important ones

```typescript
// toHaveAttribute — HTML attribute ki value check karo
// Use case: link ka href verify karo bina navigate kiye,
//           data attributes check karo, aria attributes verify karo

await expect(page.getByRole('link', { name: 'Home' }))
  .toHaveAttribute('href', '/home')

await expect(page.getByTestId('user-card'))
  .toHaveAttribute('data-status', 'active')

await expect(page.getByRole('link'))
  .toHaveAttribute('href', /\/profile\/\d+/)  // regex — koi bhi numeric ID
// Return: Promise<void>

// toHaveClass — CSS class check
// Use case: active state verify karo, error state verify karo

await expect(page.getByRole('tab', { name: 'Settings' }))
  .toHaveClass('active')

await expect(btn).toHaveClass(/btn-primary/)  // class mein 'btn-primary' ho
// Return: Promise<void>

// toHaveCSS — computed CSS property check
// Use case: styling tests, dark mode verify karo, hide/show CSS verify karo
// NOTE: computed value chahiye — 'red' nahi, 'rgb(255, 0, 0)' likhna padega

await expect(errorMsg).toHaveCSS('color', 'rgb(255, 0, 0)')
await expect(sidebar).toHaveCSS('display', 'none')
// Return: Promise<void>
```

#### Good to Remember

```typescript
await expect(input).toHaveId('email-field')  // id attribute check
// Use case: form elements ka ID verify karo
```

### Page-Level Assertions

#### Yeh kya hain
`expect(page)` — locator nahi, page object pass karte hain. Page ki overall state check karte hain.

#### Inme auto-retry hota hai?
Haan — navigation asynchronous hoti hai, Playwright wait karta hai.

```typescript
// toHaveURL — current page URL verify karo
// Use case: login ke baad correct page pe land hua, navigation sahi hua

await expect(page).toHaveURL('https://app.example.com/dashboard')
await expect(page).toHaveURL('**/dashboard')     // glob — ** = anything
await expect(page).toHaveURL(/dashboard/)        // regex
// Return: Promise<void>

// toHaveTitle — browser tab ka title verify karo
// Use case: correct page load hua, SEO testing

await expect(page).toHaveTitle('Dashboard — MyApp')
await expect(page).toHaveTitle(/Dashboard/)
// Return: Promise<void>
```

## PART 4 — isVisible() vs toBeVisible() — The Big Difference

### Pehle dono ka purpose samjho alag alag

**`isVisible()` ka purpose:**
Tumhe abhi, is moment, ek yes/no answer chahiye. Koi wait nahi, koi retry nahi. Result ke basis pe tum decide karoge kya karna hai next. Yeh ek **question** hai.

**`toBeVisible()` ka purpose:**
Tum ek guarantee de rahe ho test ko — "yeh element visible hona CHAHIYE." Agar abhi nahi hai toh Playwright wait karega jab tak ho. Agar nahi hua toh test fail karo meaningful error ke saath. Yeh ek **assertion** hai.

### Differences — Table

| | `isVisible()` | `toBeVisible()` |
|---|---|---|
| Nature | Question — "hai kya?" | Assertion — "hona chahiye" |
| Return type | `Promise<boolean>` | `Promise<void>` |
| Auto-retry | ❌ Nahi — ek baar check | ✅ Haan — ~100ms polling |
| Timeout | Koi timeout nahi | Default 5s (configurable) |
| Jab false/fail | `false` return — test chalta hai | Test fail — error + screenshot |
| Error message | Koi message nahi | Detailed Playwright error |
| Use case | `if/else` conditional logic | Test assertion |

### Ek scenario se dono ka sahi use

```typescript
// Scenario: Login karo, success toast 500ms baad appear hoti hai

// ❌ isVisible() galat use — timing issue
await loginBtn.click();
const isToastVisible = await toast.isVisible();
// Problem: click ke turant baad check kiya, toast abhi 500ms baad aayegi
// isToastVisible = false → test ya galat result dega ya fail hoga

// ✅ toBeVisible() sahi use — auto-retry handles 500ms delay
await loginBtn.click();
await expect(toast).toBeVisible();
// Playwright 100ms... 200ms... 300ms... 400ms... 500ms — YES! → PASS ✅

// ✅ isVisible() sahi use — conditional logic
if (await promoModal.isVisible()) {
  await page.getByRole('button', { name: 'Close' }).click();
}
// Yahan result chahiye logic ke liye, assertion nahi
```

### One-line rule to remember forever

> **`isVisible()`** → decide karna ho kya karna hai next
> **`toBeVisible()`** → prove karna ho ki kuch visible hai
