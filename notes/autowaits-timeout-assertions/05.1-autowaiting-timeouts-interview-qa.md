## Category 1 — Auto-Waiting Internals

---

### Q1. How does Playwright's auto-waiting work internally? Walk me through what happens when you call `click()`.

> Almost har senior interview mein puchha jaata hai. Shallow answer nahi chalega — internals batao.

**Answer:**

Jab `await page.locator('#btn').click()` call karta hoon, Playwright ek internal retry loop chalata hai — yeh loop ~100ms ke interval pe element ko re-check karta rehta hai:

| Step | What happens |
|---|---|
| 1 — Locator resolve | DOM mein selector se exactly 1 matching element dhoondta hai. Zero ya multiple mile — retry. |
| 2 — Actionability checks | 5 checks run hoti hain: Attached, Visible, Stable, Enabled, Receives Events (AVSER). |
| 3 — Execute | Sab checks pass — action perform hoti hai. Done. |
| 4 — Retry | Koi bhi check fail — ~100ms wait, Step 1 se dobara. |
| 5 — Timeout | `actionTimeout` expire — `TimeoutError` throw hoti hai with full details. |

**Selenium se key difference:** Playwright ka locator lazy hai — DOM node store nahi karta. React/Angular mein component re-render hone pe fresh DOM node pe automatically re-evaluate karta hai. Isliye Selenium ka `StaleElementReferenceException` Playwright mein exist hi nahi karta. Yeh laziness hi auto-waiting ki backbone hai.

```typescript
// Koi explicit wait nahi likhna — click() khud internally wait karti hai
await page.locator('#submit').click();

// Equivalent to kya hota tha Selenium mein:
// WebDriverWait wait = new WebDriverWait(driver, Duration.ofSeconds(10));
// wait.until(ExpectedConditions.elementToBeClickable(By.id("submit")));
// driver.findElement(By.id("submit")).click();
```

### Q2. What are the 5 actionability checks? Explain each with a real scenario where it would fail.

> Agar AVSER mnemonic pata hai — interviewer impress ho jaata hai. Practical failure scenarios mandatory hain.

**Answer:**

**A — Attached:** Element DOM tree mein present hona chahiye. `display:none` wala element bhi attached hota hai — bas visible nahi hota.

> **Fail scenario:** React component conditionally render hota hai — `{isLoggedIn && <Dashboard />}`. Login se pehle `#dashboard` pe click karo — element DOM mein hai hi nahi, attached fail.

**V — Visible:** User ko actually dikh raha ho. Non-empty bounding box, no `display:none`, no `opacity:0`, no `visibility:hidden`, no invisible parent.

> **Fail scenario:** Form validation error message initially `opacity:0` ke saath exists karta hai DOM mein — attached hai but visible nahi. Submit ke baad `opacity:1` hota hai — tab visible.

**S — Stable:** Element ka position do consecutive frames mein same ho — CSS animation/transition complete ho chuki ho.

> **Fail scenario:** Slide-in modal 300ms animation ke saath aata hai. Animation ke dauran click karo — element move kar raha hai, coordinates wrong, click miss. Stable check animation complete hone tak wait karta hai.

**E — Enabled:** Form element `disabled` attribute se disabled nahi hona chahiye.

> **Fail scenario:** Multi-step form mein "Next" button tab tak disabled hai jab tak required fields fill na ho jayein. `click()` tab tak retry karega jab tak button enable na ho.

**R — Receives Events:** Element ke center coordinate pe pointer events actually us element ko mile — koi overlay block na kar raha ho.

> **Fail scenario:** Page load hone ke baad ek analytics overlay 500ms ke liye aata hai — element visible hai, sab checks pass hain, lekin overlay upar hai. `document.elementFromPoint(x,y)` overlay element return karta hai, target nahi — check fail, retry.

```typescript
// R fail hone pe debug karo:
// Browser console mein:
document.elementFromPoint(x, y); // kaunsa element actually click receive karega?

// Proper fix:
await page.locator('.loading-overlay').waitFor({ state: 'hidden' });
await page.locator('#target-btn').click();
```

> **Important matrix:** `hover()` mein Enabled check nahi hoti — disabled element pe tooltip hover kar sakte ho. `fill()` mein Stable aur Receives Events nahi — text input mein animation irrelevant. `isVisible()` pe zero checks — immediate snapshot, koi wait nahi.

### Q3. Element is visible on screen but `click()` keeps timing out. How do you debug this?

> Real production problem hai — interviewer experience check karta hai. `force:true` ka trap bhi test karta hai.

**Answer:**

Yeh **Receives Events** check fail ho rahi hai — element visible hai lekin pointer events us tak nahi pahunch rahi. Koi cheez uske upar hai.

**Common causes:**
- Loading spinner ya modal backdrop high z-index ke saath upar hai — `opacity: 0.01` bhi browser ke liye valid clickable surface hai
- Sticky header ya cookie banner coordinates pe overlap kar raha hai
- CSS `pointer-events: none` set hai

**Debugging steps:**

```typescript
// Step 1 — Trace enable karo, visual mein dekho kya ho raha hai
// playwright.config.ts:
use: { trace: 'on-first-retry' }

// Step 2 — Browser console se confirm karo overlay hai ya nahi
document.elementFromPoint(x, y);
// Agar target element nahi aaya — overlay confirmed

// Step 3 — force: true se temporarily test karo (SIRF debug ke liye)
await page.locator('#btn').click({ force: true });
// force: true saari actionability checks bypass karta hai
// Agar force se pass ho raha hai — overlay confirm

// Step 4 — Proper fix: overlay hatne ka explicitly wait karo
await page.locator('.loading-overlay').waitFor({ state: 'hidden' });
await page.locator('#btn').click();
```

> `force: true` ko production mein kabhi mat rakho. Yeh actual user behaviour nahi hai — user bhi overlay ke peeche button click nahi kar sakta. Test pass ho jaayega but real bug hide ho jaayega. Proper fix hai overlay ka wait.

### Q4. What is the difference between `locator.waitFor()`, `page.waitForSelector()`, and just calling `locator.click()` directly? When would you use each?

> Ek common confusion hai naye aur experienced dono mein. Interviewer decision-making check karta hai.

**Answer:**

**`locator.click()` — almost always preferred:**

Action methods mein auto-wait built-in hai. Directly `click()` karo — Playwright khud visible + enabled + attached hone ka wait karega internally. Alag se `waitFor` likhne ki zaroorat nahi.

```typescript
// Yeh kaafi hai 90% cases mein:
await page.locator('#submit').click();
```

**`locator.waitFor()` — jab action aur wait alag hoon:**

Jab koi element ek state mein aane ka wait karna ho bina us pe koi action kiye — jaise spinner disappear hona, modal DOM se remove hona.

```typescript
// Spinner hatne ka wait karo, phir alag element pe action
await page.locator('.spinner').waitFor({ state: 'hidden' });
await page.locator('#data-table').click();

// Modal detach hone ka wait
await page.locator('#confirm-dialog').waitFor({ state: 'detached' });
```

Default state `visible` hai. Default timeout `0` — test timeout ki boundary use karta hai.

**`page.waitForSelector()` — legacy, avoid karo:**

Puppeteer-style old API hai jo `ElementHandle` return karta hai — ek specific DOM node ka snapshot reference. React/Angular mein component re-render hone pe yeh stale ho jaata hai — wrong element pe action ya silent failure.

```typescript
// ❌ Legacy — stale element problem
const btn = await page.waitForSelector('#submit');
// React re-renders...
await btn.click(); // old DOM node pe click — undefined behaviour

// ✅ Modern
await page.locator('#submit').waitFor({ state: 'visible' });
await page.locator('#submit').click();
// ya simply:
await page.locator('#submit').click(); // auto-wait handles it
```

> **Rule for new code:** `waitForSelector` kabhi nahi. `locator.click()` directly — jab action karna ho. `locator.waitFor()` — jab sirf state check karna ho.

### Q5. What is the difference between `expect(locator).toBeVisible()` and `locator.isVisible()`?

> Tricky question — bahut log confuse hote hain. Interviewer concept clarity check karta hai.

**Answer:**

| | `expect(locator).toBeVisible()` | `locator.isVisible()` |
|---|---|---|
| Type | Web-first assertion | Snapshot query |
| Waits? | Yes — retry karta hai 5s tak by default | No — immediate result |
| Returns | Pass/fail assertion | `true` / `false` |
| Use for | Test assertions | Conditional logic |

```typescript
// Scenario: button 2 seconds baad appear hota hai

// ✅ expect — retry karega, 2s baad visible hoga, PASS
await expect(page.locator('#delayed-btn')).toBeVisible();

// ❌ isVisible — immediately check, element abhi nahi — false milega
const visible = await page.locator('#delayed-btn').isVisible();
expect(visible).toBe(true); // FAIL — no retry hoti
```

`isVisible()` sahi jagah hai conditional branching mein:

```typescript
// Cookie banner har test mein nahi aata — conditional handle karo
if (await page.locator('#cookie-banner').isVisible()) {
  await page.locator('#accept-all').click();
}
// Proceed regardless
```

> **Rule:** Test mein kuch assert karna ho — hamesha `expect()`. Conditional flow decide karna ho — `isVisible()` / `isEnabled()` theek hai.

## Category 2 — Timeout Configuration

---

### Q6. What are the default timeout values in Playwright? What does action timeout `0` actually mean?

> Numbers yaad hone chahiye. "0 means disabled" trap common hai.

**Answer:**

| Timeout | Default | Kya controls karta hai |
|---|---|---|
| Test timeout | 30,000ms | Ek `test()` block poora complete hone ka time |
| Action timeout | 0 | `click`, `fill`, `hover`, `waitFor` etc. |
| Navigation timeout | 0 | `goto`, `waitForURL`, `waitForLoadState` |
| Assertion timeout | 5,000ms | `expect().toBeVisible()` etc. — retry window |
| Global timeout | None | Poora test suite run |

**Action timeout `0` ka matlab "disabled" nahi hai** — matlab "no independent limit". Action test timeout ki boundary ke andar chalta rehega. Default test timeout 30s hai, toh action maximum 30s tak retry kar sakta hai.

```typescript
// Agar config mein kuch set nahi:
// click() maximum 30s tak retry karega (test timeout boundary)
// expect() 5s tak retry karega (independent assertion timeout)

// Explicitly set karna ho:
export default defineConfig({
  timeout: 60000,              // test: 60s
  expect: { timeout: 10000 }, // assertions: 10s
  use: {
    actionTimeout: 15000,      // actions: 15s
    navigationTimeout: 30000,  // navigation: 30s
  }
});
```

---

### Q7. If I set `actionTimeout: 60000` but `timeout: 30000` in config, what happens?

> Classic follow-up — almost guaranteed agar timeouts pe baat ho.

**Answer:**

**Test timeout wins — always.** Action kabhi apna 60s poora nahi kar paayega.

Test 30s mein terminate ho jaayega — `Test timeout of 30000ms exceeded` error aayega. Action apna extended 60s timeout reach hi nahi kar paata kyunki test process already band ho jaati hai.

```typescript
// WRONG — actionTimeout useless hai yahan
export default defineConfig({
  timeout: 30000,         // test: 30s
  use: {
    actionTimeout: 60000  // action: 60s — test 30s mein khatam, ye 60s kabhi nahi chalega
  }
});

// CORRECT — proper hierarchy
export default defineConfig({
  timeout: 60000,         // test: 60s
  use: {
    actionTimeout: 15000, // action: 15s — test se kam, meaningful limit
    navigationTimeout: 30000, // nav: 30s — test se kam
  }
});
```

> **Golden rule:** `actionTimeout < navigationTimeout < testTimeout`. Yeh hierarchy maintain karo nahi to lower-level timeouts meaningless ho jaate hain.

### Q8. How do you set timeouts at different levels in Playwright? What is the override priority?

> Architecture-level question — Automation Architect interviews mein direct puchha jaata hai.

**Answer:**

Playwright mein 4 levels pe timeouts set kar sakte hain, har level se neeche wale ko override kar sakta hai:

**Test timeout — sabse common, 4 levels:**

```typescript
// Level 1 — Global (poore framework ke liye)
export default defineConfig({ timeout: 60000 });

// Level 2 — Project (specific browser/device ke liye)
projects: [{ name: 'mobile', timeout: 90000 }]

// Level 3 — describe block (group of tests ke liye)
test.describe('slow suite', () => {
  test.setTimeout(90000); // is group ke saare tests
});

// Level 4 — Individual test (highest priority)
test('heavy flow', async ({ page }) => {
  test.setTimeout(120000); // sirf yeh test — kisi bhi upar ke config ko override karta hai
});
```

**Action / Navigation timeout:**

```typescript
// Global use block
use: { actionTimeout: 15000, navigationTimeout: 30000 }

// Project use block — global override
projects: [{ name: 'slow', use: { actionTimeout: 25000 } }]

// Individual action — highest priority
await page.locator('#btn').click({ timeout: 5000 });
await page.goto('/page', { timeout: 60000 });
```

**Assertion timeout:**

```typescript
// Global
expect: { timeout: 10000 }

// Project
projects: [{ name: 'mobile', expect: { timeout: 15000 } }]

// Individual assertion — highest priority
await expect(page.locator('#el')).toBeVisible({ timeout: 20000 });
```

**Priority order (highest → lowest):**

```
Individual call { timeout }
    ↓
test.setTimeout() / describe setTimeout()
    ↓
Project-level config
    ↓
Global config
    ↓
Playwright default values
```

### Q9. How would you configure timeouts for a framework with smoke tests (fast) and regression tests (slow, heavy flows)?

> Practical framework design question — senior SDET / architect ke liye.

**Answer:**

Projects feature use karke alag timeout strategies define karo:

```typescript
// playwright.config.ts
export default defineConfig({
  globalTimeout: 1800000,      // suite: 30 min CI limit

  timeout: 60000,              // global default: 60s
  expect: { timeout: 10000 }, // assertions: 10s
  use: {
    actionTimeout: 15000,
    navigationTimeout: 30000,
  },

  projects: [
    {
      name: 'smoke',
      testMatch: '**/smoke/**',
      timeout: 30000,          // smoke fast hone chahiye — 30s max
      expect: { timeout: 5000 },
      use: {
        actionTimeout: 8000,
        navigationTimeout: 15000,
      }
    },
    {
      name: 'regression',
      testMatch: '**/regression/**',
      timeout: 120000,         // heavy flows — 2 min
      expect: { timeout: 15000 },
      use: {
        actionTimeout: 20000,
        navigationTimeout: 45000,
      }
    }
  ]
});

// Specific genuinely slow test ke liye override:
test('bulk report generation', async ({ page }) => {
  test.setTimeout(180000); // is ek test ke liye 3 min — project 120s override
});
```

> **Design principle:** Global mein sensible defaults. Projects mein category-specific values. Individual test-level override — sirf genuine exceptions ke liye, har test mein nahi.

## Category 3 — Page Level Waits

---

### Q10. What is the difference between `page.waitForURL()` and `page.waitForLoadState()`? When would you use each?

> Navigation waits mein sabse common confusion. SPA scenarios mein important.

**Answer:**

| | `waitForURL()` | `waitForLoadState()` |
|---|---|---|
| Waits for | URL to match a pattern | Page's loading state |
| Use case | Navigation / redirect verify | Resources loading complete |
| Default timeout | navigationTimeout (0) | navigationTimeout (0) |

**`waitForURL`** — URL change hone ka wait. Login ke baad `/dashboard` pe land hua ya nahi, SPA routing sahi hua ya nahi.

```typescript
await page.click('#login-btn');
await page.waitForURL('**/dashboard');                              // glob pattern
await page.waitForURL(/.*\/dashboard.*/);                          // regex
await page.waitForURL(url => url.searchParams.get('tab') === 'settings'); // predicate
```

**`waitForLoadState`** — Page ki resource loading complete hone ka wait. 3 states:
- `domcontentloaded` — HTML parse complete, fastest
- `load` — default after `goto()`, saari resources load
- `networkidle` — 500ms tak zero network requests

```typescript
await page.goto('https://app.example.com');      // 'load' already implicitly wait hoti hai
await page.waitForLoadState('domcontentloaded'); // faster option after click navigation
```

**SPA mein dono ki zaroorat pad sakti hai:**

```typescript
await page.click('#go-to-profile');
await page.waitForURL('**/user/profile');                          // URL change hua
await page.waitForLoadState('domcontentloaded');                   // DOM ready hai
await page.locator('.profile-header').waitFor({ state: 'visible' }); // actual content
```

> `networkidle` avoid karo agar app background polling, WebSocket heartbeats, ya analytics requests karta ho — kabhi trigger nahi hoga, test hang karega.

### Q11. Why do you always need `Promise.all` with `waitForResponse`? What goes wrong without it?

> Network intercept ka fundamental question. Race condition explanation banana chahiye clearly.

**Answer:**

Race condition se bachne ke liye. Network responses JavaScript event loop pe depend karti hain — koi guarantee nahi ki response kab aayega.

**Without `Promise.all` — what actually goes wrong:**

```typescript
// ❌ Race condition — potentially intermittent failure
await page.click('#load-users');
// Agar network fast hai — response already aa chuka click ke baad
// Ab waitForResponse listener register hua — response miss ho chuka
const res = await page.waitForResponse('/api/users'); // HANGS — event already fired
```

CI servers pe yeh intermittent failures deta hai — kabhi pass, kabhi fail. Same code, same app.

**`Promise.all` ka mechanism:**

```typescript
// ✅ Correct — Promise.all ensures listener registers BEFORE action fires
const [response] = await Promise.all([
  page.waitForResponse(r => r.url().includes('/api/users') && r.status() === 200),
  page.click('#load-users')  // yeh action tab execute hogi jab listener ready ho
]);

const data = await response.json();
expect(data.users.length).toBeGreaterThan(0);
```

> **Array destructuring `[response]` kyun:** `Promise.all` array return karta hai — pehli promise (`waitForResponse`) ka result pehle index pe hota hai.

**Same pattern — `waitForRequest` aur `waitForEvent` ke liye bhi:**

```typescript
// New tab
const [newPage] = await Promise.all([
  context.waitForEvent('page'),
  page.click('#open-new-tab')
]);

// File download
const [download] = await Promise.all([
  page.waitForEvent('download'),
  page.click('#export-btn')
]);
```

### Q12. What is `waitForFunction()` and when would you actually need it?

> Kab use karo aur kab nahi — judgment question hai.

**Answer:**

`waitForFunction()` ek custom JavaScript condition browser context mein execute karta hai — jab tak `true` return na kare tab tak retry karta rehta hai. Yeh last resort hai jab Playwright ke built-in waits kaam na karein.

**Default timeout:** test timeout. **Default polling:** `'raf'` — requestAnimationFrame, ~16ms per check.

```typescript
// Syntax
await page.waitForFunction(
  pageFunction,
  arg?,          // Node.js se browser ko value pass karo
  { polling?: number | 'raf', timeout?: number }
);
```

**Real use cases:**

```typescript
// 1. Window/global variable — SPA framework ready signal
await page.waitForFunction(() => window.appReady === true);

// 2. Third-party library initialize hone ka wait
await page.waitForFunction(() => typeof window.Stripe !== 'undefined');

// 3. Node.js se value browser mein pass karo
const threshold = 100;
await page.waitForFunction(
  (n) => document.querySelectorAll('.item').length >= n,
  threshold,
  { polling: 500, timeout: 20000 }
);
```

> **Kab use karo:** Jab koi built-in locator ya assertion kaam na kare — window variables, framework-specific state, complex DOM conditions.
>
> **Kab mat use karo:** Agar `expect(locator).toHaveCount(n)` ya `locator.waitFor()` se kaam chal sakta hai — woh prefer karo. `waitForFunction` mein browser JS likhna padta hai — debug karna harder hai.

### Q13. What is the difference between `page.waitForLoadState('networkidle')` and waiting for a specific element? Which is better and why?

> Senior level — decision-making aur real-world experience.

**Answer:**

`networkidle` — 500ms tak koi bhi network request nahi honi chahiye. Tab trigger hoti hai.

**Problem:** Enterprise apps mein background activity hoti rehti hai — analytics events, polling APIs, WebSocket heartbeats, CDN health checks. Yeh `networkidle` kabhi nahi aane dete. Test indefinitely hang karta hai ya timeout ho jaata hai.

```typescript
// ❌ Fragile in real-world apps
await page.click('#navigate-btn');
await page.waitForLoadState('networkidle'); // background polling hai — kabhi trigger nahi hoga
```

**Specific element wait — better in almost all cases:**

```typescript
// ✅ Reliable — specific meaningful state ka wait
await page.click('#navigate-btn');
await page.locator('h1.page-title').waitFor({ state: 'visible' });

// ✅ URL + element combination — SPA ke liye
await page.waitForURL('**/dashboard');
await page.locator('.dashboard-content').waitFor({ state: 'visible' });

// ✅ API response + UI — most reliable
const [_] = await Promise.all([
  page.waitForResponse('/api/dashboard-data'),
  page.click('#go-to-dashboard')
]);
await expect(page.locator('.data-table')).toBeVisible();
```

> **Best practice as Automation Architect:** Dev team ke saath ek contract establish karo — jab page fully ready ho tab ek specific `data-testid="page-ready"` attribute set karo. Tests us attribute ka wait karein — `networkidle` ya arbitrary waits nahi.

## Category 4 — Common Mistakes & Best Practices

---

### Q14. What are the anti-patterns in waiting strategies? Give examples and correct alternatives.

> Code review ya best practices question — senior role mein common.

**Answer:**

**Anti-pattern 1 — `waitForTimeout` (hard wait)**

```typescript
// ❌ Never in committed code
await page.click('#submit');
await page.waitForTimeout(3000);
await expect(page.locator('.success')).toBeVisible();

// ✅ Let assertions retry
await page.click('#submit');
await expect(page.locator('.success')).toBeVisible(); // 5s tak khud retry karega
```

**Anti-pattern 2 — Unnecessary `waitFor` before action**

```typescript
// ❌ Redundant — click() khud wait karega
await page.locator('#btn').waitFor({ state: 'visible' });
await page.locator('#btn').click();

// ✅ Direct action
await page.locator('#btn').click();
```

**Anti-pattern 3 — `isVisible()` for assertions**

```typescript
// ❌ No retry — instant check, flaky on slow loads
const visible = await page.locator('#result').isVisible();
expect(visible).toBe(true);

// ✅ Web-first assertion — retries automatically
await expect(page.locator('#result')).toBeVisible();
```

**Anti-pattern 4 — `networkidle` in apps with background traffic**

```typescript
// ❌ Hangs on apps with polling/analytics
await page.waitForLoadState('networkidle');

// ✅ Specific state wait
await page.locator('.content-loaded').waitFor({ state: 'visible' });
```

**Anti-pattern 5 — Race condition with `waitForResponse`**

```typescript
// ❌ Response can arrive before listener registered
await page.click('#load');
const res = await page.waitForResponse('/api/data'); // intermittent failure

// ✅ Promise.all ensures listener is registered first
const [res] = await Promise.all([
  page.waitForResponse('/api/data'),
  page.click('#load')
]);
```

### Q15. How do you handle a page where content loads after multiple async API calls complete? You don't know which call finishes last.

> Complex real-world scenario — architecture thinking check karta hai.

**Answer:**

Best approach is to assert on final UI state, not on individual API calls. Let Playwright's retry engine handle the timing.

```typescript
// Option 1 — Assert on final state (simplest, most reliable)
await page.click('#load-dashboard');
// Multiple APIs chal rahi hain — humein individual responses track karne ki zaroorat nahi
// Agar dashboard fully loaded hai to yeh sab true honge:
await expect(page.locator('.user-widget')).toBeVisible();
await expect(page.locator('.analytics-chart')).toBeVisible();
await expect(page.locator('.recent-activity')).toHaveCount(5);
```

```typescript
// Option 2 — Multiple Promise.all agar specific API contracts verify karne hain
const [userRes, analyticsRes] = await Promise.all([
  page.waitForResponse('/api/user-data'),
  page.waitForResponse('/api/analytics'),
  page.click('#load-dashboard')  // action
]);
// Agar dono APIs simultaneously fire hoti hain to yeh sahi hai
// Agar sequentially hain (second API first ke response ke baad call hoti hai) — yeh nahi chalega
```

```typescript
// Option 3 — toPass for complex polling condition
await expect(async () => {
  const chartLoaded = await page.locator('.chart-container canvas').isVisible();
  const tableRows = await page.locator('tbody tr').count();
  const totalAmount = await page.locator('#total-value').textContent();

  expect(chartLoaded).toBe(true);
  expect(tableRows).toBeGreaterThan(0);
  expect(totalAmount).not.toBe('Loading...');
}).toPass({ timeout: 20000 });
```

> **Practical advice:** Option 1 — assertion on final UI state — 90% cases mein kaafi hai. `toPass` tab use karo jab multiple conditions ek saath true honi chahiye aur individual assertions se kaam na chale.

## Quick Recall — What Interviewers Want to Hear

| Topic | Key Points |
|---|---|
| Auto-wait internals | Lazy locator, ~100ms retry loop, AVSER checks, no StaleElement |
| AVSER | Attached · Visible · Stable · Enabled · Receives Events — each with a failure scenario |
| Click timing out but visible | Receives Events failing — overlay/spinner, debug with `elementFromPoint`, fix with `waitFor hidden` not `force:true` |
| Default timeouts | Test=30s, Action=0 (test boundary), Navigation=0 (test boundary), Assertion=5s, Global=none |
| Action timeout 0 | Not disabled — no independent limit, bounded by test timeout |
| `actionTimeout > testTimeout` | Test timeout wins always — action timeout becomes useless |
| `Promise.all` + `waitForResponse` | Race condition — listener must register before action fires |
| `waitForURL` vs `waitForLoadState` | URL = navigation destination, LoadState = resource loading — SPA needs both |
| `networkidle` avoid | Background polling/WebSockets make it unreliable — use specific element wait instead |
| `expect` vs `isVisible` | `expect` = retries 5s, `isVisible` = instant snapshot — assertions use `expect`, conditionals use `isVisible` |
| `waitFor()` default state | `visible` — agar kuch pass nahi kiya |
| Timeout priority | Individual call > `test.setTimeout` > project config > global config > PW default |
