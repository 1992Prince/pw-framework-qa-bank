### **1. What is the role of `playwright.config.ts / .js` in a test framework?**

- "The playwright.config.ts (or .js) file is the central configuration file for a Playwright test framework. 
- The playwright config file is a single source of truth for how your test framework behaves.
- It sits at the root of your project and controls how your tests behave and run ."

#### Here's what it typically controls:

- Where tests live — testDir points to your test/spec folder
- Timeouts — global test timeout, and things like action/navigation/expect timeouts
- Retries — how many times a failed test should automatically re-run
- Reporters — how results get output (list, HTML, JSON, JUnit, etc.)
- Projects — lets you define multiple browser/device combinations (Chromium, Firefox, WebKit, Edge, mobile emulation, etc.) so the same test suite can run across all of them
- The use block — shared settings like baseURL, headless/headed mode, viewport, storageState (for reused login sessions), ignoreHTTPSErrors, permissions, and screenshot/video/trace capture rules
- workers and fullyParallel — control how much parallelism you get across spec files and tests
- globalSetup / globalTeardown — hooks that run once before/after the whole suite (e.g., seeding data, logging in once)

### **2. What are the most important fields you'd configure in `playwright.config.ts`?**

"While Playwright has many configuration options, the core fields I always set up define where the tests live and how they should fail or report."

**Key Points & Syntax:**

* **`testDir`:** Tells Playwright where to find the test files.
* **`timeout`:** Sets the maximum time a single test can run before failing.
* **`retries`:** Defines how many times a flaky test should re-run on CI.
* **`reporter`:** Determines the output format (like HTML, list, or JSON).
* **`projects`:** Configures the different browsers or devices the tests will run against.
* **`workers`:**
* **`fullyParallel`:**

**Example Syntax:**

```javascript
export default defineConfig({
  testDir: './tests',
  timeout: 30000,
  retries: 2,
  reporter: 'html',
  projects: [ /* browser configs */ ]
});

```

### **3. What goes inside the `use` block of the config file?**

- The `use` block configures the actual browser context.
- The use block holds shared settings that apply to every test (or every test within a project, if defined at the project level).


**Key Points & Syntax:**

* **`baseURL`:** The foundational URL for the application so you can use relative paths in tests.
* **`headless`:** A boolean to decide if the browser UI should be visible.
* **`viewport`:** The screen dimensions for the browser window.
* **`storageState`:** Points to a file containing cookies or local storage, which is great for bypassing login steps.
* **`extraHTTPHeaders`:** Useful for passing authorization tokens globally.
* **`permissions`:**
* **`screenshot`:**
* **`video`:**
* **`trace`:**
* **`actionTimeout`:**
* **`navigationTimeout`:**
* **`channel`:**

**Example Syntax:**

```javascript
use: {
  baseURL: 'https://example.com',
  headless: true,
  viewport: { width: 1280, height: 720 },
  storageState: 'auth.json',
  ignoreHTTPSErrors: true,
  permissions: ['geolocation'],
  screenshot: 'only-on-failure',
  video: 'retain-on-failure',
  trace: 'on-first-retry',
}

```

### **5. What are `devices` in Playwright, and how do they help?**

- devices is a built-in object exported by Playwright that contains pre-configured settings for a huge range of real-world browsers and devices — desktop browsers, mobile phones, and tablets.
- Each entry bundles together the right viewport size, user agent string, device scale factor, isMobile/hasTouch flags, and default browser engine for that device.

#### Why it helps:

- No manual viewport math — instead of guessing screen dimensions for "iPhone 12" or "Pixel 5," you just spread the device preset into your use block and it's already correct.
- Consistent emulation — user agent, touch support, and scale factor are set accurately to match how that real device would behave, which plain viewport settings alone wouldn't give you.

* **No Manual Guesswork:** Instead of manually looking up the user agent, screen size, and pixel ratio for an iPhone 12, you just spread the device object.
* **Destructuring:** The ... spread operator copies all the preset properties into our project config..

**Example Syntax:**

```javascript
{  
  name: 'Mobile Safari',  
  use: {    
    browserName: 'webkit',    
    ...devices['iPhone 12']  
  },  
}

```

### **6. Explain `defineConfig` and `export default defineConfig({})`. What is its purpose?**

"`defineConfig` is a helper function provided by Playwright that wraps our configuration object. Its primary purpose is developer experience."

**Key Points:**

* **Type Safety:** It enables TypeScript type checking, meaning your IDE will warn you if you misspell a configuration key.
* **Auto-completion:** Even if you are writing plain JavaScript (`.js`), wrapping your config in `defineConfig` gives you rich Intellisense and auto-complete in editors like VS Code.

### **7. What is head mode vs headless mode? How do you enable/switch between them?**

"Headless mode runs the browser in the background without a graphical user interface, while headed mode actually opens the physical browser window on your screen."

**Key Points & Syntax:**

* **When to use which:** Headless is faster and uses fewer resources, so it's mandatory for CI/CD pipelines. Headed mode is strictly for local debugging when you need to visually see what the test is doing.
* **How to switch:** You can toggle this in the `use` block of the config, or override it via the command line.

**Example Syntax:**

```javascript
// In config:
use: { headless: process.env.CI ? true : false }

// Or via CLI override:
npx playwright test --headed

```

### **8. Where do you set `baseURL` for a framework, and what is its purpose?**

- "The `baseURL` is set inside the `use` block. Its main purpose is to prevent hardcoding full URLs inside our actual test files."
- If you have multiple projects then you can also override the baseURL for each project in use block.

**Key Points & Example:**

* **Cleaner Tests:** Instead of writing `page.goto('https://staging.myapp.com/login')`, you just write `page.goto('/login')`. Playwright automatically appends it to the base URL.
* **Environment Agnostic:** By passing the `baseURL` dynamically (often via `.env` variables), the exact same test script can run against a local build, a QA environment, or Production without changing any test code.

### **9. What is the default timeout for `expect` assertions and element actions?**

"Playwright has a tiered timeout system. The global test timeout defaults to 30 seconds, but the assertions themselves are much shorter."

**Key Points:**

* **Assertions (`expect`):** The default timeout for an `expect` statement (like `expect(locator).toBeVisible()`) is **5 seconds**.
* **Actions:** Action timeouts (like `click()` or `fill()`) generally do not have a separate default limit out-of-the-box; they are bound by the overall test timeout (30 seconds) unless explicitly configured otherwise in the `use` block via `actionTimeout`.

### **10. What do you understand by `workers`? Explain how it works.**

`workers` is a root-level property in `playwright.config.ts` that controls how many parallel processes Playwright uses to run your test files.

- **Default behavior:** `workers: process.env.CI ? 1 : undefined`
  - `undefined` locally means Playwright automatically picks **half of your machine's CPU cores**.
  - Example: 8-core machine → 4 workers.
- **What actually gets parallelized:** By default (without `fullyParallel`), Playwright distributes **spec files** across workers — not individual tests.
  - So if you have 10 spec files and 4 workers, 4 spec files run at the same time, one per worker.
  - Once a worker finishes a spec file, it picks up the next one from the queue.
- **Inside a single spec file:** tests still run one after another, in sequence — a single worker doesn't split tests within the same file across itself.
- **In CI:** it's common to set `workers: 1` to run everything sequentially — this trades speed for stability/predictability (less risk of test interference, easier to debug failures).
- **On strong infra:** you can explicitly bump up `workers` (e.g., `workers: 6`) to speed up execution if the machine/runner can handle it.

You can also override it from the CLI without touching the config file:
```
npx playwright test --workers=4
```

Note: `workers` controls the number of parallel processes; whether it's *files* or *individual tests* that get spread across them depends on whether `fullyParallel` is also enabled (that's a related but separate setting).
