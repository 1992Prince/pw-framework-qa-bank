
# Playwright Interview Prep — Locators & Selector Strategies (Q&A)

---

**1. What locator strategies does Playwright support?**

- **Built-in `getBy` locators** (recommended): `getByRole`, `getByText`, `getByLabel`, `getByPlaceholder`, `getByAltText`, `getByTitle`, `getByTestId`
- **`page.locator()`** — accepts CSS selectors, XPath, and text-based selectors
- **Chained/filtered locators** — built on top of the above using `.filter()`, `.first()`, `.nth()`, `.last()`
- Example: `page.getByRole('button', { name: 'Submit' })` vs `page.locator('#submit-btn')`

---

**2. Why is `getByRole` considered the most recommended locator strategy?**

- It queries the **accessibility tree**, same as how a screen reader or real user perceives the page
- Not dependent on DOM structure, class names, or IDs — so it's resilient to UI/markup changes
- Encourages writing **accessible markup** as a side benefit
- Closest to how an end user actually identifies an element (by role + accessible name)
- Example: `page.getByRole('checkbox', { name: 'Accept terms' })` still works even if the underlying `<input>` classes change.

---

**3. How does `page.locator()` differ from `page.$()` / `page.querySelector()`?**

- `page.locator()` returns a **Locator** — a lazy, reusable reference; it doesn't query the DOM until an action/assertion is performed
- `page.$()` returns an **ElementHandle** — a direct, eager reference to one specific DOM node at that instant
- Locators auto-retry and auto-wait; ElementHandles do not — they can go stale if the DOM re-renders
- Example: `const btn = page.locator('button'); await btn.click();` re-resolves `button` fresh at click time, while `const handle = await page.$('button');` is a snapshot.

---

**4. What's the difference between `page.click()` and `locator.click()` — why is `locator.click()` preferred?**

- `page.click(selector)` resolves the element **once immediately** and clicks — no retry if the element isn't ready
- `locator.click()` re-resolves the element **each time** the action runs and performs actionability checks with auto-retry
- `page.click()` is effectively deprecated in favor of locators for reliability
- Example: `page.locator('#submit').click()` waits/retries until the button is actionable, whereas `page.click('#submit')` may fail if it's not immediately ready.

---

**5. How do you locate elements inside an iframe? Which approach is preferred, and what's the return type of each?**

- `page.frameLocator('iframe#myFrame').getByRole('button')` — **preferred**, returns a `FrameLocator`, which supports lazy chaining and auto-waiting
- `page.frame({ name/url })` — returns a `Frame` object directly; requires the frame to already exist, so it's less resilient to timing issues
- `frameLocator()` is preferred because it behaves like a Locator (lazy, auto-retrying) and doesn't fail if the iframe hasn't loaded yet
- Example: `await page.frameLocator('#paymentFrame').getByLabel('Card number').fill('4111...')`

---

**6. How do you handle Shadow DOM elements in Playwright?**

- Playwright **pierces open shadow DOM automatically** — no special syntax needed for CSS-based locators or `getBy` locators
- `page.locator('css=my-component >> text=Submit')` or simply `getByRole`/`getByText` works across shadow boundaries
- Closed shadow roots cannot be accessed (same limitation as browsers impose)
- Example: `page.getByRole('button', { name: 'Save' })` finds a button even if it's inside a custom web component's shadow root.

---

**7. How do you deal with dynamic/unstable selectors (auto-generated class names, data-driven IDs)?**

- Prefer **role/text/label-based locators** instead of relying on class/ID
- Use **`data-testid`** attributes added specifically for testing (`getByTestId`)
- Use **partial matches** with CSS (`[class*='btn-']`) or regex with text locators when structure is unavoidable
- Avoid deep, brittle XPath chains tied to DOM position
- Example: `page.getByTestId('user-card-123')` stays stable even if surrounding classes change on every build.

---

**8. What is the difference between a selector and a locator?**

- A **selector** is just a **string** (CSS/XPath/text) describing how to find an element — it has no behavior on its own
- A **locator** is an **object** that wraps a selector and adds behavior: auto-waiting, auto-retry, and re-resolution on every action
- Locators are the modern, recommended abstraction; raw selector strings are mostly used internally now
- Example: `'button.submit'` is a selector string, while `page.locator('button.submit')` is a Locator built from it.

---

**9. How can you execute JavaScript code in Playwright?**

- `page.evaluate(fn)` — runs JS in the browser page context and returns a serializable result
- `locator.evaluate(fn)` — runs JS with the specific matched element passed in
- `page.evaluateHandle()` — returns a `JSHandle` when you need to keep a reference to a non-serializable object
- Example: `const title = await page.evaluate(() => document.title);`

---

**10. What is the purpose of `.filter()` and its options — `has`, `hasText`, `hasNot`, `hasNotText`?**

- `.filter()` narrows down a locator that matches multiple elements to the one(s) you actually want
- **`hasText`** — keeps elements whose text content matches a string/regex
- **`hasNotText`** — excludes elements matching given text
- **`has`** — keeps elements that contain a descendant matching another locator
- **`hasNot`** — excludes elements that contain a descendant matching another locator
- Example: `page.locator('li').filter({ hasText: 'Out of stock' })` picks only the out-of-stock list items.

---

**11. Can you chain locators on `page.locator()`?**

- Yes — locators are chainable since each one just narrows the scope of the previous locator
- Chaining creates an implicit "descendant of" relationship
- Combine with `.filter()`, `.first()`, `.nth()`, `.last()` for precise targeting
- Example: `page.locator('table#orders').locator('tr').filter({ hasText: 'Pending' })`

---

**12. What is the return type of `page.locator()` and `page.locator().all()`, and what's the difference?**

- `page.locator()` returns a **single `Locator` object** — even if it matches multiple elements, it stays lazy and unresolved
- `.all()` **immediately resolves** the locator and returns an **array of Locator objects**, one per matched element, as a Promise
- Use `.all()` only when you need to iterate over multiple elements right away, since it breaks the "lazy" auto-retry benefit
- Example: `const rows = await page.locator('tr').all(); for (const row of rows) { ... }`

---

**13. What element states can be validated with `getByRole`?**

- Accessible **name** (via `{ name: '...' }`)
- **checked** state (for checkboxes/radio) via `{ checked: true }`
- **disabled** state via `{ disabled: true }`
- **pressed** state (toggle buttons) via `{ pressed: true }`
- **expanded** state (for expandable widgets) via `{ expanded: true }`
- **level** (for headings h1–h6) via `{ level: 2 }`
- Example: `page.getByRole('checkbox', { checked: true })` finds only the checked checkbox.

---

**14. What are the two DOMs on an HTML page, and how do CSS/XPath/`page.locator()` interact with them versus `getBy` locators?**

- **Regular DOM** — the standard HTML element tree
- **Accessibility Tree (a11y tree)** — a parallel tree built by the browser describing roles, names, and states for assistive tech
- CSS/XPath and generic `page.locator()` selectors query the **regular DOM structure** (tags, attributes, hierarchy)
- `getBy` locators (especially `getByRole`) query the **accessibility tree**, making them behave like a real user/screen reader
- Example: A `<div role="button">Save</div>` is found by `getByRole('button', { name: 'Save' })` via the a11y tree, even though it's a `div` in the regular DOM.

---

**15. Why are `getBy` accessibility locators considered stable and not fragile compared to generic `page.locator()`?**

- They target **role + accessible name**, which rarely changes even when developers refactor markup/CSS classes
- They don't depend on DOM hierarchy, so restructuring `div`/`span` wrappers doesn't break them
- They reflect **user-facing intent**, not implementation detail
- Example: A button's class can change from `.btn-primary` to `.btn-v2` — `getByRole('button', { name: 'Save' })` still works, but a CSS-class-based locator breaks.

---

**16. How does `page.locator()` avoid the "Stale Element Reference" exception seen in Selenium?**

- Selenium's `WebElement` is bound to a specific DOM node reference — if that node is removed/re-rendered, the reference goes stale
- Playwright's `Locator` doesn't hold a reference at all — it **re-queries the DOM fresh** every time an action or assertion runs
- This makes it naturally resilient to re-renders, SPA updates, and dynamic content
- Example: Even if a list re-renders after a filter, `page.locator('li').nth(0).click()` re-finds the current first `<li>` instead of pointing to a removed node.

---

**17. Why is `page.locator()` called "lazy"?**

- Calling `page.locator(selector)` does **not** query the DOM immediately — it just creates a description of how to find the element
- The actual DOM lookup happens only when an **action** (`.click()`, `.fill()`) or **assertion** (`expect(locator)...`) is performed
- This is what enables auto-waiting/auto-retry, since it can keep re-querying until conditions are met
- Example: `const btn = page.locator('#submit');` does nothing yet — the DOM search only happens on `await btn.click();`
