
#### **1. What are the different kinds of assertions in Playwright?**
Playwright has two categories: **auto-retrying (web-first) assertions**, which wait and re-check until a condition is met, and **non-retrying assertions**, which check a plain value once and return immediately. The first kind is used for UI state, the second for computed values like counts, strings, or API responses.

#### **2. What's the difference between auto-retry and non-retry assertions?**
Auto-retry assertions are built for asynchronous UI — the browser might take time to render something, so Playwright polls the DOM repeatedly until the condition becomes true or the timeout runs out. Non-retry assertions assume the value is already known and stable, so they just compare it once.

```typescript
await expect(page.getByText('Order placed')).toBeVisible(); // waits for it to appear
expect(cartItems.length).toBe(3);                             // checked instantly, no waiting
```

If you used a non-retry check on something that renders asynchronously, you'd get flaky failures because the DOM hasn't updated yet.

#### **3. What are Boolean State Methods?**
These are locator methods that return a `Promise<boolean>` immediately — they check the current state once, with zero waiting. They're meant for branching logic in your test, not for asserting outcomes.

```typescript
if (await page.getByTestId('cookie-banner').isVisible()) {
  await page.getByRole('button', { name: 'Accept' }).click();
}
```

Using `isVisible()` as an assertion is a common mistake — it can catch the DOM mid-update and return `false` even though the element shows up a moment later.

#### **4. Difference between `isVisible()` and `toBeVisible()`?**
`isVisible()` is a snapshot — it asks "is it visible right now?" and returns instantly with no retry. `toBeVisible()` is a promise/guarantee — it keeps checking for up to 5 seconds by default until the element becomes visible, and fails the test if it never does.

```typescript
// A toast appears 500ms after clicking login
await loginBtn.click();
const visible = await toast.isVisible();   // checked at 0ms → likely false
expect(visible).toBe(true);                 // fails incorrectly

await loginBtn.click();
await expect(toast).toBeVisible();          // retries until toast appears → passes
```

#### **5. Difference between `toHaveValue()`, `toHaveText()`, and `toContainText()`?**

- `toHaveValue()` reads the current value of form elements (input/select/textarea) — this is the DOM property, not visible text.
- `toHaveText()` requires an exact match of the element's visible inner text.
- `toContainText()` only needs a partial match, useful when part of the text is dynamic.

```typescript
await expect(page.getByLabel('Email')).toHaveValue('gaurav@test.com');
await expect(page.getByRole('heading')).toHaveText('Welcome Back');
await expect(page.getByTestId('greeting')).toContainText('Gaurav'); 
// passes even if full text is "Welcome back, Gaurav! You have 3 messages"
```

#### **6. Difference between `toBe()` and `toEqual()`?**
`toBe()` checks strict equality (`===`), so it works for primitives but fails for objects/arrays unless they're the exact same reference. `toEqual()` does a deep structural comparison, checking that the values match even if they're different object instances.

```typescript
expect(5).toBe(5);                          // passes
expect({ id: 1 }).toBe({ id: 1 });          // fails — different object references
expect({ id: 1 }).toEqual({ id: 1 });       // passes — same structure/values
```

#### **7. What is truthiness and falsiness?**
Every JavaScript value coerces to `true` or `false` in a boolean context. The falsy values are: `0`, `''` (empty string), `null`, `undefined`, `NaN`, and `false`. Everything else — including `'0'` (a string) and empty arrays/objects — is truthy.

```typescript
expect(0).toBeFalsy();
expect('').toBeFalsy();
expect('hello').toBeTruthy();
expect([]).toBeTruthy();  // an empty array is still truthy!
```

#### **8. How do you validate primitives, arrays, objects, length, content, and response time?**

```typescript
expect(response.status()).toBe(200);                  // primitive
expect(tags).toEqual(['bug', 'urgent']);               // array (order matters)
expect(user).toEqual({ id: 1, name: 'Tom' });          // object (deep match)
expect(items).toHaveLength(3);                         // length
expect(pageText).toContain('Payment successful');      // partial content
const start = Date.now();
await page.goto('/dashboard');
expect(Date.now() - start).toBeLessThan(2000);         // response/load time
```

#### **9. List the commonly used auto-retrying assertion methods.**

```typescript
// Locator state
toBeVisible() / toBeHidden() / toBeEnabled() / toBeDisabled()
toBeChecked() / toBeEditable() / toBeEmpty() / toBeFocused()

// Locator content
toHaveValue(v) / toHaveText(t) / toContainText(t) / toHaveCount(n)

// Locator properties
toHaveAttribute(name, val) / toHaveClass(cls) / toHaveCSS(prop, val) / toHaveId(id)

// Page level (called on `page`, not a locator)
expect(page).toHaveURL(url)
expect(page).toHaveTitle(title)
```

All of these return `Promise<void>` and retry internally until they pass or time out.

#### **10. Explain the internal mechanism of auto-retry assertions — polling and timeout, and how to configure it.**
Under the hood, Playwright re-checks the condition roughly every 100ms (this polling interval isn't configurable). What you *can* configure is the maximum time it keeps retrying before giving up — the timeout, which defaults to 5 seconds.

```typescript
// Global — playwright.config.ts
export default defineConfig({
  expect: { timeout: 10000 }   // applies to all assertions
});

// Per-assertion override
await expect(page.getByTestId('heavy-report-table')).toBeVisible({ timeout: 30000 });
```

If it never becomes true, Playwright throws a `TimeoutError` and automatically attaches a screenshot and trace to help debug.

#### **11. If the assertion timeout is 60 seconds but the test timeout is 30 seconds, which one wins?**
The **test timeout** always wins because it's a hard ceiling on the entire test run. Even if you configured an assertion to retry for 60 seconds, Playwright will kill the test at 30 seconds regardless, since the overall test budget ran out first.

#### **12. Difference between hard assertions and soft assertions?**
A hard assertion (`expect`) throws immediately on failure and stops the test right there. A soft assertion (`expect.soft`) records the failure but lets the test keep running, so you can see *all* the issues in one run instead of fixing them one at a time. The test still gets marked as failed at the end if any soft assertion failed.

```typescript
expect.soft(await page.locator('.title').textContent()).toBe('Dashboard');
expect.soft(await page.locator('.subtitle').textContent()).toBe('Overview');
// Both are checked and reported even if the first one fails
```

Use soft assertions when checking multiple independent things on a page (e.g., several fields in a form) where you want a full failure report rather than stopping at the first miss.

#### **13. What is visual/screenshot assertion (`toHaveScreenshot`) and how do you handle dynamic content?**
`toHaveScreenshot()` captures a screenshot and compares it pixel-by-pixel against a saved baseline image, useful for catching unintended UI/CSS regressions. Since things like timestamps or ads change every run, you mask them so they don't cause false failures.

```typescript
await expect(page).toHaveScreenshot('dashboard.png', {
  mask: [page.locator('.timestamp'), page.locator('.ad-banner')],
  maxDiffPixelRatio: 0.02, // allow tiny rendering differences
});
```

Masked regions are blacked out in both the baseline and the current screenshot before comparison, so they're effectively ignored.



#### **14. How do you give a custom error message in an assertion?**
Playwright lets you pass a message as an option to most assertions. This is useful because when a test fails in a CI log, a custom message tells you immediately what business rule broke, instead of just showing a generic locator mismatch.
```typescript
await expect(page.getByTestId('cart-total'), 'Cart total should reflect discount')
  .toHaveText('$45.00');

expect(orderCount, 'Order count should never be negative').toBeGreaterThanOrEqual(0);
```
If this fails, the report shows your custom message alongside the actual vs expected values — much easier to debug than guessing from the locator alone.

**15. What is `expect.poll()` and how is it different from other assertions?**
`expect.poll()` lets you apply auto-retrying behavior to **any custom function or value**, not just built-in Playwright matchers. It's for situations where you're checking something that isn't a locator state — like an API response, a database value, or a computed result — but still need Playwright to retry until it matches, instead of checking once.
```typescript
// Waiting for a value from an API call to eventually reach 200
await expect.poll(async () => {
  const response = await fetch('/api/order-status');
  const data = await response.json();
  return data.status;
}, { timeout: 10000, intervals: [1000, 2000, 5000] }).toBe('completed');
```
The key difference: normal web-first assertions (`toBeVisible`, `toHaveText`, etc.) only work on **locators**, and their polling behavior is baked in. `expect.poll()` works on **any async function you write**, and you control the polling intervals and timeout yourself — it's essentially a generic retry wrapper around arbitrary logic, useful when the thing you're waiting on isn't part of the DOM at all.
