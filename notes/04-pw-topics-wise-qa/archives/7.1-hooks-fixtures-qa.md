
# Playwright Fixtures & Hooks — Interview Revision Notes

---

## 1) What is a Fixture in Playwright Test?

- A fixture is reusable **setup/teardown logic** that the test runner **injects as a parameter** into your test.
- Instead of manually creating `page`, `browser`, or a custom Page Object, you declare them as fixtures — Playwright creates them before the test and cleans them up after.
- Follows the **dependency-injection pattern** — you just ask for what you need in the test signature, Playwright resolves and provides it.

```js
test('example', async ({ page }) => {
  // "page" is a fixture — already created, ready to use
});
```

---

## 2) If Hooks Already Exist, Why Do We Need Fixtures?

- Hooks are **scoped to a single spec file** — cannot be reused across other spec files. Fixtures can be **reused across any number of spec files/tests**.
- With only hooks, setup/teardown code has to be **repeated in almost every spec file**.
- Built-in fixtures already create `Browser`, `BrowserContext`, and `page` — you can directly open a URL and act. With hooks, you'd have to write all this boilerplate yourself.
- Custom fixtures can be built for login, API tokens, or test data — auto-injected wherever needed.

---

## 3) Built-in Fixtures in `@playwright/test`

| Fixture         | Purpose                                                                      |
| --------------- | ---------------------------------------------------------------------------- |
| `browser`     | Shared`Browser` instance — launched once per worker                       |
| `context`     | Isolated`BrowserContext` (like incognito) — separate session per test     |
| `page`        | Fresh`Page` (tab) — new one for every test                                |
| `request`     | `APIRequestContext` — for making API calls without a browser              |
| `browserName` | String (`chromium`/`firefox`/`webkit`) — useful for conditional logic |

---

## 4) How Fixtures Reduce Boilerplate & Provide Isolation

- Using the `page` fixture means you never manually create `Browser`/`BrowserContext` objects — the fixture handles creation and automatic cleanup after the test.
- Fixtures are **independent per test** — no shared state, so one test's data/session never leaks into another, keeping parallel execution safe.

---

## 5) Creating a Custom Fixture with `test.extend()` — Step by Step

**Requirement:**

- Need a fixture that provides an **already logged-in page**.
- Tests shouldn't repeat login code every time.
- Fixture should: (1) log in (setup) → (2) hand over the logged-in `page` to the test → (3) test runs → (4) teardown runs after, **pass or fail**.

**Concept:**

- `loggedInUser` is essentially a `page` fixture, just already logged in.
- Behaves exactly like normal `page` (`.click()`, `.fill()`, `.goto()` all work).
- Test no longer needs to request the built-in `page` fixture — `loggedInUser` is enough.

**Steps:**

1. Create a `fixtures/` folder, file named `auth.fixture.js`.
2. Import Playwright's `test` as `base`:
   ```js
   import { test as base } from "@playwright/test";
   ```

   (renamed to avoid naming conflict with the new custom `test`)
3. Build a new `test` using `base.extend()` — this returns a new `test` with all built-in fixtures **plus** your custom one.
4. Define the fixture with setup → handover → teardown:
   - Code **before** `use()` = **SETUP**
   - `await use(page)` = **HANDOVER** — control passes to the test
   - Code **after** `use()` = **TEARDOWN** — guaranteed to run (Playwright uses a `try/finally`-like mechanism internally)
5. Re-export `expect` from the same file so tests only need one import.
6. In the spec file, import `test`/`expect` from the fixture file, not from `@playwright/test`.
7. Destructure the custom fixture in the test — zero login code in the test body.

### Complete Custom Fixture Code (with comments)

```js
// fixtures/auth.fixture.js

// Step 1: import Playwright's base test, renamed to "base"
// (renaming avoids a naming clash with the new "test" we're about to create)
import { test as base } from "@playwright/test";

// Step 2: extend base test to create a new test with our custom fixture added
export const test = base.extend({

  // "loggedInUser" fixture — depends on the built-in "page" fixture
  loggedInUser: async ({ page }, use) => {

    // ---------- SETUP: runs BEFORE the test ----------
    await page.goto("https://example.com/login");
    await page.getByTestId("form-email").fill("gaurav");
    await page.getByTestId("form-password").fill("gaurav");
    await page.getByTestId("form-submit").click();
    console.log("✅ [auth fixture] Logged in successfully");

    // ---------- HANDOVER: give control (and the page) to the test ----------
    await use(page);

    // ---------- TEARDOWN: runs AFTER the test, pass or fail ----------
    await page.getByTestId("form-reset").click();
    console.log("🔴 [auth fixture] Logged out successfully");
  },
});

// Step 3: re-export expect so spec files import everything from one place
export { expect } from "@playwright/test";
```

```js
// tests/practice.spec.ts

// Import test & expect from the custom fixture file (not from @playwright/test)
import { test, expect } from "../../fixtures/auth.fixture.js";

test("TC001 - Verify logged in user can see practice page", async ({ loggedInUser }) => {
  // No login code here — loggedInUser is already an authenticated page
  await expect(loggedInUser).toHaveURL(/practice/);
});
// Logout (teardown) runs automatically after this test completes
```

## 5b) Multiple Fixtures in the Same File — Possible but Avoid

```js
export const test = base.extend({
  loggedInUser: async ({ page }, use) => { /* ... */ },
  adminUser: async ({ page }, use) => { /* ... */ },
  apiClient: async ({ request }, use) => { /* ... */ },
});
```

- Technically fine, but **avoid** — mixing unrelated fixtures in one file makes purpose unclear.
- **Better practice:** one fixture file per concern (`auth.fixture.js`, `api.fixture.js`, `data.fixture.js`), merge into a common file only if truly needed.

## 5c) API Client Login Fixture (`request`-based)

```js
export const test = base.extend({
  apiLoggedInContext: async ({ request }, use) => {
    // Setup: log in via API instead of UI
    const response = await request.post("/api/login", {
      data: { username: "gaurav", password: "gaurav" },
    });
    const { token } = await response.json();
    console.log("✅ [api fixture] Logged in via API, token received");

    // Handover: give the token to the test
    await use(token);

    // Teardown
    console.log("🔴 [api fixture] Cleanup done");
  },
});
```

- Uses `request` fixture instead of `page` — no browser UI needed, just an API call for the token.
- Much faster than UI login; real frameworks often combine this with `storageState`.

## 5d) Injecting Page Objects via Fixtures

```js
import { test as base } from "@playwright/test";
import { LoginPage } from "../pages/LoginPage";

export const test = base.extend({
  loginPage: async ({ page }, use) => {
    const loginPage = new LoginPage(page);
    await loginPage.goto();
    await use(loginPage);
  },
});

export { expect } from "@playwright/test";
```

- Test no longer writes `new LoginPage(page)` — fixture already hands it over.
- Pattern scales well: one fixture per Page Object.

---

## 5e) Why Re-export `expect` from the Fixture File

- If only `test` is exported (not `expect`), spec files need two imports:
  ```js
  import { test } from "../../fixtures/auth.fixture.js";
  import { expect } from "@playwright/test"; // extra, confusing
  ```
- Re-exporting `expect` keeps everything in one import — cleaner, and if custom matchers are added later, only one place needs updating.

## 7) Fixture Scope — `test` vs `worker`

- **`test` scope (default):** new instance created for every single test. Use when the fixture holds test-specific state that must not leak between tests — e.g. `page`, `context`, a login session tied to one test's flow.
- **`worker` scope:** created **once per worker process**, reused across all tests in that worker. Use for expensive-to-create, safely-shareable resources — e.g. launching `browser`, reading a large static test-data file, DB connection pool setup.

```js
export const test = base.extend<{}, { sharedConfig: Config }>({
  sharedConfig: [async ({}, use) => {
    const config = await loadHeavyConfig(); // expensive, done once per worker
    await use(config);
  }, { scope: 'worker' }],
});
```

**Rule of thumb:** expensive to create + safe to share (read-only/stateless) → `worker`-scoped. Needs to be fresh/isolated per test → `test`-scoped (default).

## 8) Playwright Test Hooks — `beforeAll`, `beforeEach`, `afterEach`, `afterAll`

- **`beforeAll`** — runs once before all tests in a file/describe block. Runs in **worker** context, not test context — so fixtures like `page` are not directly available (must manage browser/context manually, or better, use a worker-scoped fixture).
- **`beforeEach`** — runs before every individual test. Commonly used for navigation, login, or state reset.
- **`afterEach`** — runs after every test, pass or fail — good for cleanup, extra debug screenshots, logging.
- **`afterAll`** — runs once after all tests in that scope finish — final cleanup (closing shared resources, bulk-deleting test data).

**Best practices:**

- Prefer fixtures over `beforeAll` for `page`/`browser`-dependent setup, since `beforeAll` doesn't get test-scoped fixtures.
- Keep `beforeEach` lightweight — heavy setup repeated per test slows the whole suite; move expensive one-time setup to `beforeAll` or a worker fixture.
- Don't put assertions inside hooks that actually belong in the test — hook failures report differently and are harder to debug.
- Use `afterEach` for cleanup that must run even on test failure (fixture post-`use()` code is often cleaner for this).

## 9) Custom Fixture for Login + Test Data (Full Example)

```js
// fixtures/customFixtures.ts
import { test as base, expect } from '@playwright/test';
import { LoginPage } from '../pages/LoginPage';

type TestData = {
  username: string;
  password: string;
  products: string[];
};

type MyFixtures = {
  loginPage: LoginPage;
  authenticatedPage: LoginPage;
  testData: TestData;
};

export const test = base.extend<MyFixtures>({
  // Fixture 1: plain LoginPage, not logged in yet
  loginPage: async ({ page }, use) => {
    const loginPage = new LoginPage(page);
    await use(loginPage);
  },

  // Fixture 2: test data (could load from JSON/DB/API)
  testData: async ({}, use) => {
    const data: TestData = {
      username: 'standard_user',
      password: 'secret_sauce',
      products: ['Sauce Labs Backpack', 'Sauce Labs Bike Light'],
    };
    await use(data);
  },

  // Fixture 3: depends on loginPage + testData, returns an already logged-in page
  authenticatedPage: async ({ page, loginPage, testData }, use) => {
    await page.goto('/login');
    await loginPage.login(testData.username, testData.password);
    await use(loginPage);
    // teardown here, if needed (e.g. logout)
  },
});

export { expect };
```

```js
// tests/checkout.spec.ts
import { test, expect } from '../fixtures/customFixtures';

test('user can add products to cart after login', async ({ authenticatedPage, testData, page }) => {
  for (const product of testData.products) {
    await page.getByText(product).locator('..').getByRole('button', { name: 'Add to cart' }).click();
  }
  await expect(page.locator('.shopping_cart_badge')).toHaveText(String(testData.products.length));
});
```

**Key point — Fixture Chaining:** `authenticatedPage` depends on both `loginPage` and `testData`. A test that only asks for `authenticatedPage` automatically gets a ready, logged-in session — zero setup code in the test body.
