
Here are the answers, interview-ready style:

#### **1. What test annotations in Playwright are you aware of?**
`test.skip()`, `test.fixme()`, `test.only()`, `test.slow()`, `test.fail()`, and conditional variants of these (e.g., `test.skip(condition, reason)`). They're used to control which tests run and how they're treated in reports.

#### **2. fixme vs skip vs only vs slow**

- `test.skip()` — test is not run at all. Use when a feature isn't ready yet or a test is temporarily irrelevant. Can be unconditional (`test.skip()`) or conditional (`test.skip(browserName === 'webkit', 'not supported on webkit')`).
- `test.fixme()` — similar to skip (test doesn't run), but signals "this is broken and needs fixing," not "intentionally excluded." It's a semantic difference for the team — shows up differently in reports so people know it needs attention, not that it's permanently excluded.
- `test.only()` — runs ONLY this test (or describe block) in the file, ignoring all siblings. Used locally during development/debugging. Should never be committed — CI usually has a lint rule or `forbidOnly: true` in config to catch this.
- `test.slow()` — test still runs, but Playwright triples its timeout. Use for tests you know are inherently slow (heavy data setup, large file uploads) instead of bumping the global timeout for everyone.

#### **3. Tagging tests and selective execution**
Tags can be added in the test title as `@tagname` or via the `tag` property:

```js
test('user can login @smoke', async ({ page }) => { ... });
// or
test('user can login', { tag: '@smoke' }, async ({ page }) => { ... });
```

CLI usage:

- Single tag: `npx playwright test --grep @smoke`
- Multiple tags (AND — must match both): `npx playwright test --grep "(?=.*@smoke)(?=.*@regression)"`
- Multiple tags (OR — match either): `npx playwright test --grep "@smoke|@regression"`
- Exclude a tag: `npx playwright test --grep-invert @regression`
- Combine include + exclude: `npx playwright test --grep @smoke --grep-invert @flaky`

#### **4. Grouping tests with `test.describe()`**
`test.describe()` groups related tests under a common block, mainly for organization and readability in reports, and to share a common `beforeEach`/`afterEach` scoped to that group:

```js
test.describe('Login feature', () => {
  test.beforeEach(async ({ page }) => { await page.goto('/login'); });
  test('valid login', async ({ page }) => { ... });
  test('invalid login', async ({ page }) => { ... });
});
```

#### **5. `test.describe.serial()` and `test.describe.parallel()`**

- `test.describe.serial()` forces tests within that block to run in order, in the **same worker**, and if one fails, the rest are skipped (not just run out of order — actually skipped after a failure). Useful when tests share state and are dependent on each other (e.g., step 1 creates a record, step 2 edits it, step 3 deletes it).
- `test.describe.parallel()` explicitly marks a group to run in parallel across workers — this is actually the default behavior at the file level in most configs, but it's useful when parallelism is disabled globally and you want to opt specific groups back in.

**Why you need `serial()` even though a spec file "runs in sequence" already:** by default, tests within a single file *execute one after another*, but they still run in **isolated contexts** and Playwright doesn't guarantee no other file's worker interleaves oddly, and more importantly — if one test fails in a normal sequential file, the **next test still runs**. With `serial()`, a failure causes all *subsequent* tests in that block to be skipped, which is critical when test 2 literally cannot succeed without test 1 having passed (e.g., dependent multi-step flows). It's about failure propagation and worker-pinning, not just ordering.

#### **6. Does Playwright run each test in incognito mode by default?**
Yes. Every test gets a fresh `BrowserContext` (comparable to an incognito window) — no cookies, local storage, or cache carried over from other tests. No extra config needed; this is what gives Playwright strong test isolation and safe parallelism out of the box.

#### **7. `--slowMo` vs `test.slow()`**

- `--slowMo` (or `slowMo` in launch options) **slows down every Playwright action** by a specified number of milliseconds, purely for humans to visually follow along in headed mode while debugging. It doesn't change any timeout — it just delays each step.
- `test.slow()` doesn't affect execution speed at all — it **triples the default timeout** for that specific test, so it has *more time* to complete without being flagged as timed out. Used for tests that are legitimately expected to take longer.

#### **8. What is `test.use()`?**
`test.use()` lets you override fixture values for a specific scope (a `describe` block or file) without redefining the whole fixture:

```js
test.describe('mobile tests', () => {
  test.use({ viewport: { width: 375, height: 667 } });
  test('mobile menu', async ({ page }) => { ... });
});
```

Commonly used to override `baseURL`, `viewport`, `storageState`, or `locale` for a subset of tests.

#### **9. `test.step()` vs `testInfo()`**

- `test.step()` wraps a block of actions into a named, collapsible step that shows up distinctly in the HTML report and trace viewer — purely for readability/debugging structure:

```js
await test.step('Login', async () => {
  await page.fill('#user', 'admin');
  await page.click('#submit');
});
```

- `testInfo` is an object (injected as a fixture) giving you metadata and control **about the currently running test** — title, file, project name, retry count, status, annotations, and lets you attach files/screenshots or set a custom timeout at runtime:

```js
test('example', async ({ page }, testInfo) => {
  console.log(testInfo.title, testInfo.retry);
  await testInfo.attach('screenshot', { path: 'shot.png' });
});
```

So `test.step()` is about **structuring/reporting the test's actions**, while `testInfo` is about **introspecting/controlling the test's runtime metadata**.

#### **10. What is the `test` function, and how do you import it?**
`test` is the core function from `@playwright/test` used to define a single test case. Import syntax:

```js
import { test, expect } from '@playwright/test';

test('test title', async ({ page }) => {
  // test body — page is a built-in fixture
  await page.goto('https://example.com');
  await expect(page).toHaveTitle(/Example/);
});
```

Syntax breakdown: `test(title: string, testFunction: (fixtures, testInfo) => Promise<void>)`. The fixtures object (`{ page }`, `{ browser }`, etc.) is destructured automatically by Playwright's test runner based on what you request, and `testInfo` is an optional second parameter for metadata as covered above.

#### **11. Explain `test.fail()`**

`test.fail()` marks a test as **expected to fail**. If the test does fail, Playwright reports it as "expected failure" (shown differently from a real failure, usually not breaking the CI build). If the test unexpectedly *passes*, Playwright flags that as an error — because you told it "this should fail" and it didn't, which means either the bug got fixed (and you forgot to remove the annotation) or something's off.

```js
test('checkout with expired coupon', async ({ page }) => {
  test.fail(); // known bug — checkout currently allows expired coupons
  await page.fill('#coupon', 'EXPIRED10');
  await page.click('#apply');
  await expect(page.locator('.error')).toBeVisible(); // this currently doesn't show, so test fails as expected
});
```

Conditional form is also supported:
```js
test.fail(browserName === 'webkit', 'known webkit rendering bug');
```

**Use case:** tracking known bugs without deleting the test or leaving it in a permanently red state — it documents "yes, we know this is broken" while still keeping the test executable so you get notified the moment it's fixed (unexpectedly passes) or if the failure mode changes.

Difference from `skip`/`fixme`: those don't run the test body at all; `fail()` **does run it**, it just changes how the pass/fail outcome is interpreted.
