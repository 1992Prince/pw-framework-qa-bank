
#### **1. What are the different types of waits in Playwright?**

Playwright's biggest strength is that it **auto-waits** by default before every action — before clicking, it waits for the element to be visible, stable (not animating), enabled, and receiving events. This alone removes most of the manual `sleep()` calls you'd need in older frameworks like Selenium. On top of that, Playwright gives you explicit wait mechanisms for cases where auto-wait isn't enough (e.g., waiting for a network call, a URL change, or a custom condition).

**Auto-waiting (built-in, always on):**
Before actions like `click()`, `fill()`, `check()`, Playwright automatically waits for the element to be attached, visible, stable, and not obscured — no code needed.

```typescript
await page.getByRole('button', { name: 'Submit' }).click(); // waits internally before clicking
```

**Wait at Action level** — waits tied to a specific interaction:

```typescript
await locator.click({ timeout: 10000 });          // wait for actionability, override timeout
await locator.fill('text', { timeout: 5000 });      // wait for editable state
await locator.hover();                              // wait for visible + stable
await locator.check();                              // wait for enabled + visible checkbox
```

Conditions checked: attached to DOM, visible, stable (no animation), enabled, receives pointer events.

**Wait at Locator level** — waiting for a locator's state directly:

```typescript
await locator.waitFor({ state: 'visible' });   // default
await locator.waitFor({ state: 'hidden' });
await locator.waitFor({ state: 'attached' });
await locator.waitFor({ state: 'detached' });
```

Useful when you need to just wait without performing an action right after.

**Wait at Navigation level** — waiting for page loading/navigation events:

```typescript
await page.waitForURL('**/dashboard');                    // wait for URL to match
await page.waitForLoadState('load');                        // full page load
await page.waitForLoadState('domcontentloaded');            // DOM ready
await page.waitForLoadState('networkidle');                 // network settled
await page.waitForNavigation();                              // (legacy, mostly replaced by waitForURL)
```

**Wait at Assertion level** — auto-retrying assertions covered earlier:

```typescript
await expect(locator).toBeVisible();     // retries up to timeout
await expect(page).toHaveURL(/dashboard/);
await expect.poll(async () => getStatus()).toBe('done');
```

**Wait at Page level** — waiting on network/browser events at the page:

```typescript
await page.waitForResponse('**/api/orders');       // wait for a specific network response
await page.waitForRequest('**/api/save');           // wait for a specific request to fire
await page.waitForEvent('popup');                   // wait for a new tab/popup
await page.waitForFunction(() => window.appReady);  // wait for custom JS condition in browser
```

Together, these give you layered control: auto-wait handles 90% of cases silently, and the explicit waits handle network timing, navigation, and custom conditions that auto-wait can't see.

#### **2. What is `page.waitForLoadState()`, and what's the difference between `load`, `domcontentloaded`, and `networkidle`?**

`waitForLoadState()` pauses your test until the page reaches a certain loading milestone. It's useful right after a navigation or form submission when you want to be sure the page has settled before interacting with it.

- **`domcontentloaded`** — fires as soon as the HTML is parsed and the DOM tree is built, but before images, stylesheets, or async scripts finish loading. Fastest, but risky if you need JS-rendered content.
- **`load`** — fires when the full page has loaded, including images, CSS, and other subresources. This is the default Playwright waits for after `page.goto()`.
- **`networkidle`** — fires when there have been no network connections for at least 500ms. Useful for SPAs that keep fetching data after the initial load, but it's discouraged as a general-purpose wait because some apps have persistent background requests (analytics, polling) that never let the network go idle.

```typescript
await page.goto('/dashboard');
await page.waitForLoadState('domcontentloaded'); // DOM ready, fast
await page.waitForLoadState('load');              // full resources loaded
await page.waitForLoadState('networkidle');       // no network activity for 500ms
```

In practice, most teams avoid relying on `networkidle` and instead wait for a specific element or API response — it's more deterministic than guessing when "the network is quiet."

#### **3. How do `page.waitForResponse()` / `page.waitForRequest()` help synchronize UI and network activity?**

These let you tie your test's timing directly to a specific network call instead of guessing based on page load state. This matters most for SPAs where clicking a button triggers an API call, and the UI only updates once that response comes back — auto-waiting on the DOM alone might check too early.

```typescript
// Wait for a specific API call to resolve before continuing
const [response] = await Promise.all([
  page.waitForResponse(resp => resp.url().includes('/api/orders') && resp.status() === 200),
  page.getByRole('button', { name: 'Place Order' }).click(),
]);
const order = await response.json();
expect(order.status).toBe('confirmed');

// Wait for a request to be sent (e.g. verify an analytics call fired)
await page.waitForRequest('**/api/track-event');
```

The `Promise.all` pattern is important here — you start waiting for the response *before* triggering the click, so you don't miss the response if it comes back very fast.

#### **4. What are the different timeout types (test, action, navigation, expect, global), and how do you configure each?**

Playwright has multiple independent timeout layers, and understanding which one governs what stops a lot of confusion when debugging failures.

- **Test timeout** — max time a single test is allowed to run overall. If exceeded, the test is marked failed regardless of what's still executing.
- **Action timeout** — max time for a single action (click, fill, etc.) to complete its auto-wait.
- **Navigation timeout** — max time for `goto()`, `waitForURL()`, and similar navigation calls.
- **Expect timeout** — max time an assertion will retry before failing.
- **Global timeout** — max time for the *entire test run* (all tests combined), used as a safety net in CI.

```typescript
// playwright.config.ts
export default defineConfig({
  timeout: 30000,              // test timeout — per test
  globalTimeout: 3600000,      // global timeout — entire suite, 1 hour cap
  expect: { timeout: 5000 },   // expect timeout — per assertion
  use: {
    actionTimeout: 10000,      // action timeout — per action
    navigationTimeout: 15000,  // navigation timeout — per navigation
  },
});

// Per-call overrides
await page.click('button', { timeout: 5000 });
await page.goto('/slow-page', { timeout: 20000 });
await expect(locator).toBeVisible({ timeout: 8000 });
```

As covered earlier, the **test timeout always wins** over any inner timeout — if actions/assertions individually haven't timed out but the total test duration exceeds the test timeout, Playwright kills it anyway.

#### **5. What's the difference between `page.setDefaultTimeout()` and `page.setDefaultNavigationTimeout()`?**

Both set default timeouts at the page-instance level (overriding config for that page object), but they apply to different categories of operations.

- **`setDefaultTimeout()`** — sets the default timeout for most page operations: actions like `click()`, `fill()`, and general waits like `waitForSelector()`. It does *not* affect navigation calls.
- **`setDefaultNavigationTimeout()`** — specifically overrides the timeout for navigation-related calls like `goto()`, `waitForURL()`, `goBack()`, `goForward()`.

```typescript
page.setDefaultTimeout(15000);            // affects click(), fill(), waitForSelector(), etc.
page.setDefaultNavigationTimeout(30000);  // affects goto(), waitForURL(), etc. only

await page.goto('/slow-loading-page');    // uses the 30000ms navigation timeout
await page.click('button');               // uses the 15000ms general timeout
```

This separation exists because navigations are often inherently slower (server response, redirects, large payloads) than simple UI interactions, so it makes sense to tune them independently rather than using one blanket timeout for everything.
