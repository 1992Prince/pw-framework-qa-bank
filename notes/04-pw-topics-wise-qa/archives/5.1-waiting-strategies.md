# Playwright Explicit Waits — By Level

Playwright auto-waits before most actions (`click`, `fill`, `check`) — it checks the element is visible, stable, enabled, and not obscured before acting, retrying until the **default action timeout (30s)** expires. Explicit waits are for the cases auto-wait doesn't cover: waiting *without* acting, waiting on the page/network, or waiting on browser-level events.

---

## 1) Locator-Level Waits

Use when you want to wait for a specific **element state**, without performing an action on it yet.

### `locator.waitFor()`

```ts
await locator.waitFor({
  state: 'visible' | 'hidden' | 'attached' | 'detached', // default: 'visible'
  timeout: number // optional, default: inherits actionTimeout (30s)
});
```

- `attached` — element is in the DOM (may not be visible)
- `detached` — element removed from the DOM
- `visible` — in DOM **and** has non-zero size, no `visibility:hidden`
- `hidden` — not visible, or not attached at all

**Example — wait for spinner to disappear before interacting:**

```ts
await page.locator('.loading-spinner').waitFor({ state: 'hidden' });
await page.locator('#data-table').click(); // now safe to interact
```

> Note: `page.waitForSelector()` does the same job but is the **older, discouraged API** — it returns an `ElementHandle` instead of a `Locator`, which is harder to chain and less stable. Prefer `locator.waitFor()` (or better, just let `.click()`/`.fill()` auto-wait) in new code. `waitForSelector` isn't deprecated, but it's legacy.

---

## 2) Page-Level Waits

### a) Wait for navigation to a URL

```ts
await page.waitForURL('https://app.example.com/dashboard', { timeout: 15000 });
```

Verifies navigation landed on the expected URL (supports string, glob, regex, or a predicate function).

- **Default timeout:** inherits `navigationTimeout` (config default `0` → falls back to the global test timeout).
- **Config:** `use.navigationTimeout` in `playwright.config.ts`.

### b) Wait for page load state

```ts
await page.waitForLoadState('domcontentloaded' | 'load' | 'networkidle');
```

| State                | Meaning                                                                                                                      |
| -------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `domcontentloaded` | HTML parsed, DOM ready; images/CSS/JS may still be loading                                                                   |
| `load` (default)   | Everything loaded — images, CSS, JS                                                                                         |
| `networkidle`      | No network requests for 500ms — use sparingly, discouraged by Playwright for SPAs with polling/analytics that never go idle |

**Default timeout:** inherits `navigationTimeout`.

### c) Wait for a network call to resolve

```ts
await page.click('#fetch-data');          // trigger the API call
const res = await page.waitForResponse('/api/users'); // wait for the response
```

- `page.waitForResponse(urlOrPredicate, options?)` — waits for a matching response
- `page.waitForRequest(urlOrPredicate, options?)` — waits for a matching request to be *sent*
- Both accept a string, glob, regex, or a predicate `(response) => boolean` for more control (e.g. matching status code too).

```ts
const res = await page.waitForResponse(
  res => res.url().includes('/api/users') && res.status() === 200
);
```

### d) Wait for a browser-level event

```ts
page.waitForEvent(event, optionsOrPredicate?)
```

Used for things that fire asynchronously outside the normal action flow — dialogs, new tabs/popups, downloads, console messages, etc.

**Handling a `dialog` (alert/confirm/prompt):**

```ts
page.once('dialog', dialog => {
  console.log(dialog.message());
  dialog.accept(); // or dialog.dismiss()
});
await page.click('#delete-button'); // triggers the confirm() dialog
```

*(For dialogs, `page.on('dialog', ...)` is the idiomatic pattern rather than `waitForEvent`, since the dialog must be handled the instant it appears or it blocks the page.)*

**Handling a new tab/popup:**

```ts
const [popup] = await Promise.all([
  page.waitForEvent('popup'),
  page.click('#open-new-tab'), // action that triggers the popup
]);
await popup.waitForLoadState();
console.log(await popup.title());
```

**Handling a download:**

```ts
const [download] = await Promise.all([
  page.waitForEvent('download'),
  page.click('#download-button'),
]);
await download.saveAs('/path/to/save/' + download.suggestedFilename());
```

> Pattern to remember: `waitForEvent` must be started **before** the triggering action, so it's almost always paired with `Promise.all([waitForEvent(...), action()])` — otherwise there's a race where the event fires before you start listening.

### e) Hard wait (timeout-based — last resort)

```ts
await page.waitForTimeout(10000); // waits 10 seconds, no matter what
```

Not condition-based — just a fixed delay. Avoid in real test suites (flaky/slow); acceptable only for quick local debugging.

### f) Wait for a specific selector's state (legacy API)

```ts
await page.waitForSelector('#data-table', { state: 'visible', timeout: 10000 });
```

Waits for one specific element (not all elements — useful since AJAX-loaded elements appear at different times). Functionally the same purpose as `locator.waitFor()`, but returns an `ElementHandle`. **Prefer `locator.waitFor()` for new code.**

### g) Wait for a custom JS condition

```ts
await page.waitForFunction(() => document.querySelectorAll('.row').length > 10);
```

Polls an arbitrary JS predicate in the browser context until it returns truthy. Useful when the condition isn't a simple element state — e.g. waiting for a global JS variable, a counter value, or a computed style.

```ts
// with an argument passed in
await page.waitForFunction(
  (expectedCount) => document.querySelectorAll('.row').length === expectedCount,
  15 // arg
);
```

## 3) Navigation-Level Waits

These specifically concern page navigations (as opposed to waiting for elements/network on an already-loaded page).

- `page.waitForURL(url, options)` — covered above; the modern way to confirm navigation landed correctly.
- `page.waitForLoadState(state)` — covered above; confirms *how much* of the destination page has loaded.
- `page.waitForNavigation()` — **legacy/deprecated.** Used to be paired with an action like:

  ```ts
  await Promise.all([
    page.waitForNavigation(),
    page.click('#submit'),
  ]);
  ```

  Replaced by `page.waitForURL()` after the action, or by simply relying on auto-waiting locator actions, since `waitForNavigation` had race-condition issues similar to `waitForEvent`.
- **Default timeout for all of these:** `navigationTimeout` (config default `0` → falls back to the global test/action timeout).

---

## 4) Assertion-Level Waits

Playwright's `expect()` assertions (the "web-first assertions") are **auto-retrying** — they are not a single instant check, but poll the condition until it passes or the timeout expires. This means assertions themselves double as explicit waits.

```ts
await expect(locator).toBeVisible();
await expect(locator).toHaveText('Order Confirmed');
await expect(locator).toHaveCount(5);
await expect(page).toHaveURL(/dashboard/);
await expect(locator).toBeEnabled();
```

- **Default timeout:** `expect.timeout` — **5 seconds** by default (separate config from the 30s action timeout).
- **Config (global):**
  ```ts
  export default defineConfig({
    expect: { timeout: 10000 },
  });
  ```
- **Per-assertion override:**
  ```ts
  await expect(locator).toBeVisible({ timeout: 15000 });
  ```

Because assertions retry, `expect(locator).toBeVisible()` is often a better explicit wait than `locator.waitFor({ state: 'visible' })` when you're about to assert anyway — one call does both the waiting and the checking.

**Q1. What are the different types of waits available in Playwright?**

* Playwright **auto-waits by default** before every action — checks element is visible, stable, enabled, and receiving events
* Explicit waits are needed only when auto-wait isn't enough — network calls, URL changes, custom conditions
* **Locator level:** `locator.waitFor({ state })` — wait for visible/hidden/attached/detached
* **Navigation level:** `page.waitForURL()`, `page.waitForLoadState()`
* **Page level:** `page.waitForResponse()`, `page.waitForRequest()`, `page.waitForEvent()`, `page.waitForFunction()`

**Q2. What is `page.waitForLoadState()`, and what's the difference between `load`, `domcontentloaded`, and `networkidle`?**

* `waitForLoadState()` pauses the test until the page hits a loading milestone — used after navigation or form submit
* `domcontentloaded` — HTML parsed, DOM built; images/CSS/async scripts may still be loading
* `load` (default) — everything loaded: images, CSS, JS
* `networkidle` — no network activity for 500ms
* `networkidle` is generally discouraged — SPAs with polling/analytics never truly go idle; better to wait on a specific element or API response

#### **Q3. How do `page.waitForResponse()` / `page.waitForRequest()` help synchronize UI and network activity?**

* Ties test timing to a specific network call instead of guessing from page load state
* Needed for SPAs where a click triggers an API call and UI updates only after response returns
* `waitForResponse()` — waits for a matching response (URL, glob, regex, or predicate with status check)
* `waitForRequest()` — waits for a matching request to be sent
* Must start the wait **before** the triggering action, wrapped in `Promise.all()` — otherwise a fast response can be missed

#### **Q4. What are the different timeout types (test, action, navigation, expect, global), and how do you configure each?**

* **Test timeout** — max time for one test overall; fails the test if exceeded regardless of what's still running
* **Action timeout** — max time for a single action (click, fill) to complete its auto-wait
* **Navigation timeout** — max time for `goto()`, `waitForURL()`, etc.
* **Expect timeout** — max time an assertion retries before failing (default 5s)
* **Global timeout** — max time for the entire test suite/run combined, used as a CI safety net
* Configured via `playwright.config.ts`: `timeout`, `globalTimeout`, `expect.timeout`, `use.actionTimeout`, `use.navigationTimeout`
* Can also override per call: `page.click('button', { timeout: 5000 })`
* **Test timeout always wins** — even if individual actions/assertions haven't timed out, the whole test is killed once total duration exceeds the test timeout

### **Q5. What's the difference between `page.setDefaultTimeout()` and `page.setDefaultNavigationTimeout()`?**

* Both set default timeouts at the page-instance level, overriding config for that page object
* `setDefaultTimeout()` — applies to general operations: `click()`, `fill()`, `waitForSelector()`; does **not** affect navigation
* `setDefaultNavigationTimeout()` — applies only to navigation calls: `goto()`, `waitForURL()`, `goBack()`, `goForward()`
* Kept separate because navigations are typically slower (server response, redirects) than simple UI actions, so each needs independent tuning

### **Q6. What's the difference between `locator.waitFor({ state: 'attached' })` and `{ state: 'visible' }`?**

* `attached` — element only needs to exist in the DOM; could still be hidden via CSS
* `visible` — element must be in the DOM **and** have a non-zero bounding box with no hiding CSS (e.g. `display:none`, `visibility:hidden`)

### **Q7. When would you use `page.waitForFunction()`?**

*  when you have your own custom JS function/logic in the script and you want Playwright to wait until that function's condition returns true — you simply pass that function (or its check) inside waitForFunction(), and Playwright keeps polling it in the browser context until it resolves truthy or the timeout hits

### **Q8. What's the difference between `page.waitForSelector()` and `locator.waitFor()`? Which one is not preferred, and why?**

* `waitForSelector()` — older, page-level API; still supported, returns an `ElementHandle`
* `locator.waitFor()` — modern, locator-based equivalent
* `locator.waitFor()` is preferred — it integrates with Playwright's  **auto-retrying locator model** , while `waitForSelector()` returns a static handle that doesn't re-query the DOM
