
# Mocking Interview Questions (Playwright)

## 1. What is Mocking?

**Answer:**

Mocking is the process of creating a fake implementation of a real dependency, such as an API or service, so that we can test our application without calling the actual system.

Instead of sending the request to the real backend, we intercept it and return a predefined response. This allows us to test different scenarios in a controlled and predictable way.

---

## 2. Why do we use Mocking?

**Answer:**

We use mocking to remove dependencies on external systems and make our tests more reliable.

Some common reasons are:

- Backend service is not available (e.g., third-party payment APIs may not be available in lower environments).
- Backend functionality is still under development.
- We want to avoid calling third-party services.
- We need predictable test data.
- We want to simulate success, failure, timeout, or error scenarios.
- Tests become faster and more stable.

---

## 3. Where have you used Mocking in your automation project?

**Answer:**

In my project, I use **Playwright's API mocking** using the `page.route()` method.

For example, when the browser sends a request to the `/payment` API, I intercept that request and return a mocked response instead of allowing it to reach the real backend.

The mocked response contains values like:

```json
{
  "message": "Transaction Successful",
  "transactionId": "TXN12345"
}
```

This allows me to test the payment success flow without depending on the actual payment service.

---

## 4. How does Playwright Mocking work?

**Answer:**

The flow is:

```text
Browser
    │
Clicks Pay
    │
Request → /payment
    │
Playwright intercepts request
    │
Returns mocked response
    │
Browser receives mocked response
    │
UI behaves as if backend returned success
```

The real backend is never called.

---

## 5. Why did you use Mocking instead of the real API?

**Answer:**

The real API may be unavailable, unstable, slow, or return changing data.

By mocking the API:

- My tests become independent of backend availability, especially in lower environments.
- I always get the expected response.
- Tests execute faster.
- I can easily test different scenarios like success, failure, or server errors.

---

## 6. What is a Mock?

**Answer:**

A mock is a fake implementation of a real object or service that returns predefined data instead of executing the actual logic.

For example, instead of calling the real payment API:

```http
POST /payment
```

The mock returns:

```json
{
  "message": "Transaction Successful"
}
```

---

## 7. What is a Mock Server?

**Answer:**

A mock server is a server that mimics the behavior of the real backend.

Instead of processing actual business logic, it listens for requests and returns predefined responses.

Examples include:

- Mountebank
- WireMock
- MockServer

---

## 8. What is a Stub?

**Answer:**

A stub is simply a predefined response that is returned when a request matches certain conditions.

Think of it as:

> **"If this request comes, return this response."**

Example:

Request:

```http
GET /users/101
```

Stub Response:

```json
{
  "id": 101,
  "name": "Prince"
}
```

---

## 9. What is the difference between a Mock and a Stub?

**Answer (Speakable):**

A **stub** only returns predefined data for a request.

A **mock** is a broader concept. It simulates the behavior of a real dependency. A mock may contain one or more stubs to return different responses.

In simple terms:

- **Stub = Fake response.**
- **Mock = Fake implementation of a service or dependency.**

In API testing and automation, people often use the terms **mock** and **stub** interchangeably.

---

## 10. What is Intercepting?

**Answer:**

Intercepting means catching a request before it reaches the actual server.

Once intercepted, we can:

- Continue the request unchanged.
- Modify the request.
- Return a mocked response.
- Block the request.

---

## 11. What can you mock?

**Answer:**

Using Playwright's `page.route()`, we can return any response that we want to test.

For example, we can mock:

- Success responses (2xx)
- Client errors (4xx)
- Server errors (5xx)
- Custom response body
- Custom headers
- Different response delays (to simulate slow APIs)

This allows us to test both **positive** and **negative** scenarios without calling the real backend.
