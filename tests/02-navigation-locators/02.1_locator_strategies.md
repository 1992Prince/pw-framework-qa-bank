
# Playwright Locators — Complete Conceptual Notes

---

## 1️⃣ `page.locator()` — The Foundation

**Definition:** `page.locator()` is the primary method to find and interact with elements in Playwright.

**Return Type:** It returns a **Locator object** — not the actual element, not a Promise. It's a "smart pointer" / query description.

**Lazy Initialization:** Calling `page.locator()` does **not** search the browser. It just stores *instructions* (the selector logic) on how to find the element later.

```js
const emailInput = page.locator('#inputEmail'); // No await — nothing searched yet
await emailInput.fill('hello@example.com');     // NOW the browser is queried
```

**Sync vs Async — very important interview point:**

- `page.locator()` itself is **synchronous** → no `await` needed to define it (it's a local, in-memory operation, no browser round-trip).
- **Action methods** (`.click()`, `.fill()`, `.check()`) and **assertions** (`expect(locator).toBeVisible()`) are **asynchronous** → they need `await` because they involve a round-trip to the browser and must perform **actionability checks** (element attached, visible, stable, enabled, receiving events) before acting.

**How this avoids Selenium's `StaleElementReferenceException`:**

- In Selenium, `findElement()` fetches a **reference to a specific DOM node** immediately. If the DOM re-renders (React/Angular replaces that node), the old reference becomes "stale" and throws an error on next use.
- In Playwright, a Locator **never holds a reference to an actual DOM node**. It only holds the *query* (selector string + options). Every time you call an action, Playwright **re-resolves the query against the current DOM** from scratch.
- So even if the page re-renders 10 times, the same Locator variable will always find the *current* matching element — there's nothing to go "stale."

> **Interview one-liner:** *"Playwright locators don't store elements, they store the logic to find elements — so they auto-re-query on every action, which is why Playwright never throws stale element exceptions like Selenium does."*

## 2️⃣ Accessibility Locators — What & Why

**Also known as:** User-Facing Locators / User-Centric Locators / Accessibility-Based Locators (`getByRole`, `getByLabel`, `getByText`, `getByPlaceholder`, `getByAltText`, `getByTitle`, `getByTestId`).

**Why Playwright recommends them:**
They locate elements the way a **real user or a screen reader** perceives the page — by role and visible name — instead of by internal implementation details like IDs, classes, or DOM position.

**Internal working (how it's different from CSS/XPath):**

- CSS/XPath query the **DOM Tree** — the raw HTML structure (tags, classes, IDs, hierarchy).
- Accessibility locators query the **Accessibility Tree** — a separate, simplified tree the browser builds from the DOM specifically for assistive technology. It only contains **Name, Role, and State** of meaningful elements (noise like layout `<div>`s and styling classes is stripped out).
- `getByRole('button', {name:'Login'})` doesn't search for a `<button>` tag — it asks the Accessibility Tree: *"give me the element whose computed Role is 'button' and Accessible Name is 'Login'"*. It doesn't matter if it's a real `<button>`, an `<a>`, or a `<div role="button">`.

**Why they are resilient (not fragile) — the core reasoning:**

- The DOM Tree and Accessibility Tree are **two separate trees** derived from the same HTML but serving different purposes.
- CSS classes, IDs, and DOM structure can change freely (a dev renames a class, wraps an element in a new `<div>`, restructures layout) — this changes the **DOM Tree**, but the element's **Role and Accessible Name stay the same** in the Accessibility Tree, *as long as the element still functions the same way to a user*.
- So a test using `getByRole('button', {name:'Submit'})` keeps passing even if the underlying tag/class/structure is refactored — because Playwright isn't reading the DOM tree that changed, it's reading the Accessibility Tree that didn't.
- This also works across **Shadow DOM** — normal CSS/XPath cannot cross a shadow boundary, but the Accessibility Tree flattens shadow DOM content, so `getByRole`/`getByLabel` etc. pierce it automatically.

> **Interview one-liner:** *"page.locator() with CSS/XPath reads the DOM Tree, so it breaks when DOM structure changes. Accessibility locators read a separate Accessibility Tree built from Role + Name, which only changes when the element's actual function/label changes — not when its markup does. That's why they're resilient."*

## 3️⃣ List of Accessibility (Built-in) Locators

| Locator                      | Definition                                                                                                      |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------- |
| `getByRole(role, options)` | Finds by ARIA role + accessible name + state. The most powerful/recommended.                                    |
| `getByLabel(text)`         | Finds form fields by the text of their associated`<label>`.                                                   |
| `getByPlaceholder(text)`   | Finds inputs by their placeholder hint text.                                                                    |
| `getByText(text)`          | Finds elements by visible text content (substring, exact, or regex).                                            |
| `getByAltText(text)`       | Finds images/elements by their`alt` attribute.                                                                |
| `getByTitle(text)`         | Finds elements by their`title` attribute.                                                                     |
| `getByTestId(id)`          | Finds elements by`data-testid` attribute (explicit test hook, not accessibility-based but equally resilient). |

All return type: **Locator object** (same as `page.locator()`), so `.click()`, `.fill()`, chaining, `.filter()` all work identically.

## 4️⃣ Deep Dive: `getByRole()` (highest interview weightage)

### Syntax

```js
page.getByRole(role, { options });
```

- `role`: string — one of 60+ ARIA roles (`button`, `link`, `heading`, `checkbox`, `textbox`, `tab`, `img`, etc.)
- `options`: object to narrow down the match

### What is a "Role"?

The Role defines *what the element functions as* to a user — not what tag it is.

### Implicit vs Explicit Roles

- **Implicit Role:** the default role the browser auto-assigns based on the HTML tag.
  - Example: `<button>` → implicit role `"button"`. No extra code needed.
- **Explicit Role:** a role manually added by a developer via the `role` attribute, usually on a non-semantic tag.
  - Example: `<div role="button">Click Me</div>` → this div is now findable via `getByRole('button', ...)`.
- **Rule:** Explicit role always takes preference over the native HTML tag's implicit role.
- **Note:** A generic `<div>`/`<span>` with **no role attribute at all** has no role in the Accessibility Tree, and `getByRole` won't find it.

### Filtering with `name` (Accessible Name)

The `name` option matches the **accessible name** — computed from: inner text, `aria-label`, or a linked `<label>`.

**Substring match (default):**

```js
await page.getByRole('button', { name: 'Submit' });
// matches "Submit Now", "Submit Form", etc.
```

**Exact match:**

```js
await page.getByRole('button', { name: 'Submit', exact: true });
// ONLY matches "Submit" — not "submit" or "Submit Form"
```

**Regex match (case-insensitive):**

```js
page.getByRole('button', { name: /log in/i });
// matches "Log in", "LOG IN", "Login"
```

### Filtering with Element State

In Playwright, `checked`, `disabled`, `expanded`, `pressed`, `selected` are called **Element States**. They let you filter/locate elements based on their **current live status**, not just role+name — this lets your locator itself encode business-logic assertions.

**Why this is powerful:** Instead of finding a generic "button" and then separately checking if it's disabled, you find *only the button that is currently in that state* — reducing strict-mode violations and making tests read like requirements ("verify Submit is disabled until Terms is checked").

**Pattern for waiting on state without hardcoded sleeps:**

```js
const submitBtn = page.getByRole('button', { name: 'Submit', disabled: true });
await expect(submitBtn).toBeVisible();
```

`expect(locator).toBeVisible()` on a state-filtered locator auto-retries until that condition becomes true — a clean way to wait for UI logic to complete instead of `page.waitForTimeout()`.

---

### The 5 Essential State Filters

**A. `checked`** — Target roles: `checkbox`, `radio`
Real scenario: verifying a "Terms & Conditions" checkbox is ticked, or which payment-method radio is active.

```js
const checkedBox = page.getByRole('checkbox', { name: 'I agree', checked: true });
await expect(checkedBox).toBeVisible();
```

**B. `disabled`** — Target roles: `button`, `textbox`, `select`, `checkbox`
Real scenario: Submit button should stay disabled until email is valid.

```js
const submitBtn = page.getByRole('button', { name: 'Submit', disabled: true });
await expect(submitBtn).toBeVisible();
// Wait until it becomes enabled
await expect(page.getByRole('button', { name: 'Submit', disabled: false })).toBeVisible();
```

**C. `expanded`** — Target roles: `button`, `link`, `menuitem`
Real scenario: FAQ accordion or side-nav open/closed state.

```js
await page.getByRole('button', { name: 'What is DOM?', expanded: false }).click();
await expect(page.getByRole('button', { name: 'What is DOM?', expanded: true })).toBeVisible();
```

**D. `pressed`** — Target roles: toggle-style `button`
Real scenario: Mute button in a call, Bold icon in editor — stays "pushed" state.

```js
const muteBtn = page.getByRole('button', { name: 'Mute', pressed: true });
await expect(muteBtn).toBeVisible();
```

**E. `selected`** — Target roles: `tab`, `option`, `treeitem`
Real scenario: verifying which tab (Home/Profile/Settings) is currently active.

```js
const settingsTab = page.getByRole('tab', { name: 'Settings', selected: true });
await expect(settingsTab).toBeVisible();
```

**Why state filters beat plain assertions:**

```js
await expect(page.getByRole('checkbox')).toBeChecked(); // ❌ fails if 2+ checkboxes exist (strict mode violation)
await page.getByRole('checkbox', { checked: true }).click(); // ✅ precisely targets the one that's checked
```

## 5️⃣ `.filter()` — Locating a Parent by Validating Its Children

### Purpose (say this in the interview)

> *"`.filter()` doesn't locate the child — it locates the **parent/container**, but the decision of *which* parent to pick is made by looking **inside** that parent. So the target of the locator stays the parent card; the filter condition is based on what's living inside it."*

**Concrete example to say out loud:**

> "Suppose there are 4 cards on the page, all with the same class `.user-card`. I don't want *a* card, I want **the specific card that contains the child text 'Admin User'**. So I first locate all 4 cards, then `.filter({ hasText: 'Admin User' })` keeps only the one card whose subtree contains that text — the other 3 are dropped. My final locator still points to the *card* (the parent), not to the text itself."

### What does "subtree" mean?

**Subtree = the element itself + every descendant inside it, at any level of nesting** — not just direct/immediate children.

So if the text or the target child element is nested 3 or 4 levels deep inside the card (e.g., `.user-card > div > span > h3`), `.filter({ hasText })` or `.filter({ has })` will still find it, because it searches the **entire subtree**, not just one level down.

```html
<div class="user-card">
  <div class="header">
    <h3>Admin User</h3>       <!-- 2 levels deep, still matched -->
  </div>
  <button>Edit</button>
</div>
```

```js
page.locator('.user-card').filter({ hasText: 'Admin User' })
// ✅ still matches — "Admin User" is anywhere in the subtree, regardless of nesting depth
```

### Mental Model (say this first, always)

1. Locate the **common parent/container** first (this is the base set — e.g., all 4 cards).
2. **Then** `.filter()` on that set, using a condition that looks **inside** each element (text present, or a matching child element present/absent).
3. `.filter()` **never re-searches the whole page** — it only operates within the elements already found in step 1.
4. The **result is still a set of parent elements** (narrowed down) — not the child you filtered on.

### Syntax Variants

**`.filter({ hasText: 'X' })`** — keep only the container(s) whose subtree contains text X

```js
await page.locator('.user-card')
  .filter({ hasText: 'Admin User' })     // keeps only the card containing "Admin User" anywhere inside it
  .getByRole('button', { name: 'Edit' }) // now act on THAT card's Edit button
  .click();
```

**`.filter({ hasNotText: 'X' })`** — exclude container(s) whose subtree contains text X

```js
await page.locator('.user-card')
  .filter({ hasNotText: 'Guest User' })  // drop the Guest User card, keep the rest
  .getByRole('button', { name: 'Edit' })
  .click();
```

**`.filter({ has: locator })`** — keep container(s) that have a **specific child element** matching another locator (structure/state-based, not text-based)

```js
await page.locator('.user-card')
  .filter({ has: page.getByRole('button', { name: 'Delete', disabled: false }) })
  // keeps only the card that has an ENABLED Delete button inside it, at any nesting level
  .getByRole('button', { name: 'Delete' })
  .click();
```

**`.filter({ hasNot: locator })`** — exclude container(s) that have a matching child element

```js
await page.locator('.user-card')
  .filter({ hasNot: page.getByRole('button', { disabled: true }) })
  // keeps only cards that do NOT contain any disabled button
  .getByRole('button')
  .click();
```

### `hasText`/`hasNotText` vs `has`/`hasNot` — the difference to articulate

|                              | Checks                                                                                                                      | Based on                             |
| ---------------------------- | --------------------------------------------------------------------------------------------------------------------------- | ------------------------------------ |
| `hasText` / `hasNotText` | Is this**text string** present/absent anywhere in the subtree?                                                        | Text content                         |
| `has` / `hasNot`         | Is there a**specific element matching another locator** (with its own role/name/state) present/absent in the subtree? | Structure + state of a child locator |

`has`/`hasNot` is more powerful because the "child condition" can itself carry role, name, and state filters (e.g., "has an enabled Delete button" — not just "has the word Delete").

### Ultra-Short Summary (Revision)

- `locator()` → finds **all** matching elements (the base set)
- `.filter()` → narrows that set down, based on what's inside each element
- `hasText` → keep parent if text is present anywhere in its subtree
- `hasNotText` → keep parent if text is absent from its subtree
- `has` → keep parent if a specific child locator matches anywhere in its subtree
- `hasNot` → keep parent if a specific child locator does NOT match in its subtree
- **subtree = element + all descendants, any depth of nesting** — not just direct children
- Result of `.filter()` is always the **parent/container**, never the child you filtered by

## 6️⃣ `.all()` vs `count()` + `nth()` — Do We Still Need `.all()`?

**Yes — different purpose, different risk profile.**

| Approach                      | Behavior                                                                      | Risk                                                                   |
| ----------------------------- | ----------------------------------------------------------------------------- | ---------------------------------------------------------------------- |
| `count()` + `nth(i)` loop | Re-queries browser**on every iteration**                                | Safe for dynamic/re-rendering pages (React/Angular)                    |
| `.all()`                    | Queries browser**once**, returns a **snapshot array** of Locators | Fast, but list can go stale if DOM changes after the snapshot is taken |

```js
// nth-based (re-queries each time — safe for dynamic UI)
const count = await elements.count();
for (let i = 0; i < count; i++) {
  const text = await elements.nth(i).textContent();
}

// .all()-based (one snapshot — fast, use on stable UI)
const items = await page.getByRole('checkbox').all();
for (const item of items) {
  await item.check();
}
```

**Key distinction:** `.all()` doesn't auto-wait for elements to appear — if called before elements render, it returns `[]`. Each item in the returned array is still its own `Locator` object, so Playwright's auto-waiting logic still applies to it when you perform an action — just the *array construction itself* isn't auto-waited.

> **Interview one-liner:** *"`.all()` is still needed for clean `for...of` iteration over a fixed snapshot of elements — use it when the list is stable. Prefer `count()` + `nth()` when the UI re-renders dynamically, since it re-queries the DOM on every iteration and won't go stale."*
