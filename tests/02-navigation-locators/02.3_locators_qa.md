
# Playwright Locators — Interview Q&A

---

### Q1. What does `page.locator()` return, and why don't you use `await` when defining it?

**Answer:** It returns a **Locator object**, which is just a stored query (selector + options) — no browser call happens yet. It's synchronous because storing a string locally doesn't need a browser round-trip. `await` is only needed on **actions** (`.click()`, `.fill()`) because those actually talk to the browser and run actionability checks, which takes time and returns a Promise.

```js
const btn = page.locator('#submit');   // no await — just defines "how to find"
await btn.click();                     // await — this is where the search + click happens
```

---

### Q2. How does Playwright avoid Selenium's `StaleElementReferenceException`?

**Answer:** Selenium's `findElement()` grabs a reference to one specific DOM node at that instant. If the page re-renders and that node is replaced, the reference becomes stale. Playwright's Locator never holds a node reference — it holds the *query itself* and re-resolves it against the live DOM every time an action runs. So even after 10 re-renders, the same Locator variable always finds the current matching element.

---

### Q3. What is the difference between a Selector and a Locator?

**Answer:**

- **Selector** = just a string (e.g., `'.save-btn'`) — the "address," no intelligence, can't be clicked.
- **Locator** = a Playwright object built from a selector, with auto-waiting, retry logic, and action methods (`.click()`, `.fill()`).

> "The Selector is the input, the Locator is the processor that uses that input to find and act on the element."

---

### Q4. Why does Playwright recommend Accessibility-based locators (`getByRole`, `getByLabel`, etc.) over CSS/XPath?

**Answer:** CSS/XPath read the raw **DOM Tree** — tags, classes, IDs, structure. Accessibility locators read a separate **Accessibility Tree** built by the browser, containing only Role, Name, and State — the same information a screen reader uses. Since dev-side refactors (renaming a class, changing a `<div>` to a `<span>`, restructuring markup) change the DOM Tree but not the element's Role/Name (as long as it still functions the same way), Accessibility locators don't break. They're also more readable and push developers toward writing more accessible HTML.

---

### Q5. Explain implicit vs explicit roles with an example.

**Answer:**

- **Implicit role** — auto-assigned by the browser from the HTML tag. `<button>` automatically has role `"button"`, no extra markup needed.
- **Explicit role** — manually added via the `role` attribute, typically on a non-semantic tag like `<div>`. Example: `<div role="button">Click Me</div>` — the Accessibility Tree now treats it as a button.
- **Rule:** if both exist, the explicit `role` attribute wins over the tag's implicit role.
- A plain `<div>`/`<span>` with no `role` at all has **no role** in the Accessibility Tree and `getByRole` cannot find it.

---

### Q6. How do you locate an element using `getByRole` with an exact name match vs a substring match vs a regex?

```js
// Substring (default) — matches "Submit", "Submit Now", "Submit Form"
await page.getByRole('button', { name: 'Submit' });

// Exact — only matches "Submit" exactly (case-sensitive)
await page.getByRole('button', { name: 'Submit', exact: true });

// Regex, case-insensitive — matches "Log in", "LOG IN", "Login"
await page.getByRole('button', { name: /log in/i });
```

---

### Q7. What are "Element States" in `getByRole`, and why are they useful?

**Answer:** States (`checked`, `disabled`, `expanded`, `pressed`, `selected`) let you filter elements by their **current live condition**, not just role/name. This is useful because it lets the locator itself encode business logic — e.g., "give me the Submit button only while it's disabled" — instead of finding a generic button and asserting separately. It also avoids strict-mode violations when multiple elements share the same role.

### Q8. Write a locator + assertion to verify a Submit button starts disabled and later becomes enabled, without using a hardcoded wait.

```js
const disabledBtn = page.getByRole('button', { name: 'Submit', disabled: true });
await expect(disabledBtn).toBeVisible();

// fill required field...
await page.getByLabel('Email').fill('user@example.com');

const enabledBtn = page.getByRole('button', { name: 'Submit', disabled: false });
await expect(enabledBtn).toBeVisible(); // auto-retries until state changes — no page.waitForTimeout()
```

### Q10. Why is `page.getByRole('checkbox', { checked: true }).click()` better than `expect(page.getByRole('checkbox')).toBeChecked()` when there are multiple checkboxes?

**Answer:** The second form throws a **strict mode violation** because `getByRole('checkbox')` matches all checkboxes and Playwright refuses to guess which one you mean. The first form uses the `checked: true` filter *inside the locator itself* to precisely target only the checkbox that is already checked — no ambiguity, no strict-mode error.

---

### Q11. What is `.filter()` used for, and when do you reach for it?

**Answer:** `.filter()` narrows down a set of already-located elements (a locator that matches multiple elements — repeated cards, table rows, list items) based on a condition, without using brittle `nth()` indexing. You use it whenever a base locator returns multiple matches and you need only the ones meeting a specific criterion (contains certain text, or contains/lacks a certain child element).

---

### Q12. I have 4 cards on a page with the same class. I want to click "Edit" only on the card that has "Admin User" in it. How do you approach this, and what does the final locator actually point to?

**Answer & code:**

```js
await page.locator('.user-card')            // Step 1: locate the common parent — all 4 cards
  .filter({ hasText: 'Admin User' })         // Step 2: keep only the card whose subtree contains this text
  .getByRole('button', { name: 'Edit' })     // Step 3: within THAT card, find its Edit button
  .click();
```

**Key point to say out loud:** *".filter() still returns the parent/container — the card — not the text I filtered on. I'm validating the child content to decide which parent to keep, but the object I act on afterward is scoped inside that chosen parent."*

### Q13. What does "subtree" mean in the context of `hasText`/`has`? Does nesting depth matter?

**Answer:** Subtree = the element itself **plus every descendant inside it, at any level of nesting** — not just direct/immediate children. So even if the matching text or child element is nested 3–4 levels deep (`.card > div > span > h3`), `.filter({ hasText })` / `.filter({ has })` will still find it, because the search isn't limited to one level down — it scans the whole subtree.

```html
<div class="user-card">
  <div class="header"><h3>Admin User</h3></div>  <!-- 2 levels deep -->
</div>
```

```js
page.locator('.user-card').filter({ hasText: 'Admin User' }) // ✅ still matches
```

### Q14. What's the difference between `.filter({ hasText })` and `.filter({ has: locator })`?

**Answer:**

- `hasText` / `hasNotText` — checks for a **plain text string** anywhere in the subtree. Simple presence/absence check.
- `has` / `hasNot` — checks for a **specific element matching another locator** in the subtree. That inner locator can itself carry role, name, and state filters — e.g., "has an *enabled* Delete button," not just "contains the word Delete." So `has`/`hasNot` is structurally more precise/powerful than plain text matching.

```js
// hasText — just checks text exists
.filter({ hasText: 'Delete' })

// has — checks a specific enabled button element exists
.filter({ has: page.getByRole('button', { name: 'Delete', disabled: false }) })
```

### Q15. Write a locator to select the card that does NOT contain any disabled button.

```js
await page.locator('.user-card')
  .filter({ hasNot: page.getByRole('button', { disabled: true }) })
  .getByRole('button')
  .click();
```

### Q16. What is the difference between `.all()` and `count()` + `nth()` for iterating multiple elements? Which is safer for dynamic pages?

**Answer:**

- `count()` + `nth(i)` — **re-queries the browser on every loop iteration**, so it always reflects the current DOM. Safer for dynamic/re-rendering UIs (React/Angular).
- `.all()` — resolves the locator **once**, immediately, and returns a **snapshot array** of Locators at that exact moment. Fast and clean for `for...of` loops, but if the DOM changes after the snapshot (elements added/removed/reordered), the array can become outdated.

**Rule of thumb:** use `.all()` on stable/static UI; use `count()+nth()` when the page is dynamically re-rendering.

```js
// Dynamic-safe
const count = await page.locator('.item').count();
for (let i = 0; i < count; i++) {
  await page.locator('.item').nth(i).click();
}

// Stable-UI, cleaner syntax
const items = await page.getByRole('checkbox').all();
for (const item of items) {
  await item.check();
}
```

---

### Q17. Does `.all()` auto-wait for elements to appear?

**Answer:** No. `.all()` does not auto-wait — if called before the elements exist in the DOM, it simply returns an empty array `[]`. Best practice: ensure elements are present first (e.g., `await expect(locator.first()).toBeVisible()`) before calling `.all()`. However, each individual Locator *inside* the returned array still has normal auto-waiting applied when you later call an action on it.

---

### Q18. How does Playwright locate elements inside a Shadow DOM, when normal CSS/XPath cannot cross a shadow boundary?

**Answer:** Standard CSS selectors and XPath only traverse the raw DOM Tree, and a Shadow DOM's shadow root is a separate encapsulated boundary that they can't pierce. Playwright's built-in locators (`getByRole`, `getByLabel`, `getByText`, etc.) instead query the **Accessibility Tree**, which is a flattened representation that already includes elements exposed inside open shadow roots. So these locators work through shadow DOM without needing any explicit JavaScript (`element.shadowRoot.querySelector(...)`).

---

### Q19. What is Playwright's "Strict Mode," and when does it throw an error?

**Answer:** Strict mode ensures Playwright never guesses between multiple matching elements.

- **Defining** a locator that matches multiple elements does **not** throw — it's just a stored query.
- **Performing an action** (`.click()`, `.fill()`) on a locator matching **more than one element** throws a *"strict mode violation"* error.
- **Resolve it** with `.first()`, `.last()`, `.nth(index)`, or `.filter()`.
- **Exception:** `.count()` never triggers strict mode since it's explicitly designed to work with multiple matches.

```js
await page.getByRole('button').first().click(); // ✅ resolves ambiguity
const total = await page.getByRole('button').count(); // ✅ fine with multiple matches
```

### Q20. Give a one-line summary you could say to close out a question on locators in general.

**Answer:** *"I prefer accessibility-based locators like `getByRole` for resilience since they query the Accessibility Tree rather than the DOM, I use `.filter()` with `hasText`/`has` to scope down to the right parent when multiple similar elements exist, and I choose between `.all()` and `count()+nth()` based on whether the UI is static or dynamically re-rendering."*
