
# 📝 Notes: Custom Fixture kaise banate hain (Login Fixture Example)

## 1) Requirement — Kya chahiye

- Humein aisa fixture chahiye jo **already logged-in state** provide kare
- Tests ko har baar login ka code nahi likhna chahiye — repetitive setup avoid karna hai
- Fixture ka kaam hoga:
  1. **Login karna** (setup)
  2. Logged-in `page` ko **test ko handover karna**
  3. Test apna kaam kare
  4. Test complete hone ke baad **cleanup/teardown code chalna chahiye**
  5. **Important:** yeh cleanup code chalna chahiye chahe test **pass ho ya fail** — guaranteed teardown

---

## 2) Fixture "kis type" ka hoga — concept

- Page Object ek web page ko represent karta hai (locators + actions)
- Yahan hum jo fixture bana rahe hain — `loggedInUser` — woh basically ek **`page` fixture hi hai**, bas already logged-in state mein
- Fixture `page` object ko hi test ko handover karega, bas login perform karne ke baad
- Test ke andar `loggedInUser` ko destructure karke use karte hain — aur woh normal `page` ki tarah hi behave karta hai (`.click()`, `.fill()`, `.goto()` sab available hai)
- **Key point:** test ko ab Playwright ka inbuilt `page` fixture directly maangne ki zaroorat nahi — `loggedInUser` hi kaafi hai, jo internally `page` ka hi extended/logged-in version hai

---

## 3) Custom Fixture banane ke Steps — Code ke saath

### Step 1: Fixtures folder banao

Framework mein `fixtures/` naam ka folder banate hain, file ka naam `auth.fixture.js` rakhte hain — naam se hi pata chale ki yeh authentication-related fixture hai.

### Step 2: Playwright ka `test` import karo, naam `base` rakho

```js
import { test as base } from "@playwright/test";
```

- Original `test` ko `base` naam se import kar rahe hain
- **Reason:** hum aage khud ka naya `test` banayenge aur export karenge — agar naam same `test` rakhte, toh conflict ho jata

### Step 3: `base.extend()` se naya `test` banao

```js
export const test = base.extend({
  // yahan custom fixtures define honge
});
```

- `base.extend({...})` naya `test` object return karta hai jisme Playwright ke saare built-in fixtures (`page`, `context`, etc.) already honge, **plus** humara custom fixture bhi add ho jayega

### Step 4: Fixture define karo — setup, handover, teardown

```js
loggedInUser: async ({ page }, use) => {
  // ── SETUP: yeh test se PEHLE chalega ──
  await page.goto("https://gauravkhurana.com/practise-api/ui/index.html#/practice");
  await page.getByTestId("form-email").fill("gaurav");
  await page.getByTestId("form-password").fill("gaurav");
  await page.getByTestId("form-submit").click();
  console.log("✅ [auth fixture] Logged in successfully");

  // ── HANDOVER: page ko test ko de rahe hain ──
  await use(page);

  // ── TEARDOWN: yeh test ke BAAD chalega — pass ho ya fail, dono case mein ──
  await page.getByTestId("form-reset").click();
  console.log("🔴 [auth fixture] Logged out successfully");
},
```

**Kahan kya likhna hai — important:**

- `use()` se **pehle** ka code = **SETUP** (login yahan hota hai)
- `await use(page)` line = **HANDOVER point** — yahan control test ko milta hai, page ka logged-in state test ko diya jaata hai
- `use()` ke **baad** ka code = **TEARDOWN** — yeh guaranteed chalega, chahe test pass ho ya fail, kyunki Playwright internally `try/finally` jaisa mechanism use karta hai fixture ke around

### Step 5: `expect` ko bhi re-export karo (Point 7 ka detail neeche hai)

```js
export { expect } from "@playwright/test";
```

### Step 6: Spec file mein fixture file se import karo

```js
import { test, expect } from "../../fixtures/auth.fixture.js";
```

- Ab spec file `@playwright/test` se directly import nahi karti — apni `auth.fixture.js` se karti hai, jisme extended `test` hai (jisme `loggedInUser` fixture included hai)

### Step 7: Test mein fixture destructure karke use karo

```js
test("TC001 - Verify logged in user can see practice page", async ({ loggedInUser }) => {
  console.log("▶ TC001 running — user is already logged in via fixture");
  await expect(loggedInUser).toHaveURL(/practice/);
  console.log("✅ TC001 — practice page URL confirmed");
});
```

- `{ loggedInUser }` destructure kiya — Playwright automatically fixture function call karta hai, login karta hai, aur logged-in `page` (naam se `loggedInUser`) test ko de deta hai
- Test ke andar login ka ek bhi line nahi — seedha assertion/actions likh sakte hain
- Test complete hone ke baad teardown (logout) automatically chal jaayega

## 4) Multiple fixtures same file mein — possible hai, lekin avoid karo

```js
export const test = base.extend({
  loggedInUser: async ({ page }, use) => { /* ... */ },
  adminUser: async ({ page }, use) => { /* ... */ },
  apiClient: async ({ request }, use) => { /* ... */ },
});
```

- Technically ek hi `base.extend({...})` call mein multiple fixtures define kiye ja sakte hain
- **Lekin avoid karna chahiye** — ek file mein bohot saare unrelated fixtures dalne se file ka purpose unclear ho jaata hai aur maintain karna mushkil hota hai
- **Better practice:** har concern ke liye alag fixture file banao — jaise `auth.fixture.js` (login related), `api.fixture.js` (API client related), `data.fixture.js` (test data related) — phir zaroorat padne pe inko **merge/combine** kar sakte hain ek common fixture file mein jo spec files import karengi

## 5) API Client Login Fixture — `request` type ka fixture

Agar login UI se nahi, **API se** karna hai (fast, aur UI dependency avoid karne ke liye), toh fixture `page` ki jagah `request` fixture use karega:

```js
export const test = base.extend({
  apiLoggedInContext: async ({ request }, use) => {
    // SETUP — API call se login
    const response = await request.post("/api/login", {
      data: { username: "gaurav", password: "gaurav" },
    });
    const { token } = await response.json();
    console.log("✅ [api fixture] Logged in via API, token received");

    // HANDOVER — token ya authenticated request context test ko de rahe hain
    await use(token);

    // TEARDOWN
    console.log("🔴 [api fixture] Cleanup done");
  },
});
```

- Yahan `page` ki jagah `request` fixture use hua — kyunki humein browser UI ki zaroorat nahi, seedha API call se authentication token chahiye
- Yeh approach UI login se **kaafi fast** hota hai — real frameworks mein isko combine karte hain `storageState` ke saath taaki UI tests bhi bina UI-login ke chal sakein

## 6) Page Objects ko fixture ke through inject karna

Login jaisa hi pattern Page Objects ke liye bhi use karte hain — taaki spec files mein `new LoginPage(page)` baar baar likhne ki zaroorat na ho.

```js
import { test as base } from "@playwright/test";
import { LoginPage } from "../pages/LoginPage";

export const test = base.extend({
  loginPage: async ({ page }, use) => {
    // SETUP — Page Object create karo
    const loginPage = new LoginPage(page);
    await loginPage.goto();

    // HANDOVER
    await use(loginPage);

    // TEARDOWN (agar zaroorat ho)
  },
});

export { expect } from "@playwright/test";
```

**Spec file mein:**

```js
import { test, expect } from "../../fixtures/pageObjects.fixture.js";

test("login with valid credentials", async ({ loginPage }) => {
  // loginPage already ready hai — Page Object instance directly mil gaya
  await loginPage.fillCredentials("sarika@example.com", "password123");
  await loginPage.submit();
});
```

- Fixture ne `LoginPage` ka instance banaya aur seedha test ko de diya
- Test ko `new LoginPage(page)` likhne ki zaroorat nahi — fixture ne already object bana ke handover kar diya
- Yeh pattern scale hota hai — jitne Page Objects utne fixtures, aur spec files sirf destructure karke use karti hain

## 7) `expect` ko bhi same fixture file se export karna — kyun zaroori hai

```js
export { expect } from "@playwright/test";
```

- Agar hum sirf `test` export karte aur `expect` nahi, toh spec file ko do jagah se import karna padta:

```js
import { test } from "../../fixtures/auth.fixture.js";
import { expect } from "@playwright/test"; // extra import, confusing
```

- Isse import statements inconsistent ho jaate — kabhi fixture file se, kabhi Playwright se directly
- **`expect` ko bhi re-export karne se** spec file ko sirf **ek hi jagah se** dono milte hain:

```js
import { test, expect } from "../../fixtures/auth.fixture.js";
```

- Yeh cleaner hai, aur agar kal fixture file mein `expect` ko bhi customize/extend karna pade (jaise custom matchers add karna), toh sirf ek jagah change karna padega — spec files ka import statement same rahega
