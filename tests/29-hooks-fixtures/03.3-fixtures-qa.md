
# 📝 Notes: Fixtures — Interview Q&A (Speakable Answers)

## 1) When we have hooks already, why do we need fixtures?

**How to speak in interview:**

"Hooks like `beforeEach` and `afterEach` are scoped to a file or a `describe` block — so if I need the same setup and teardown logic across multiple spec files, I end up writing that same code again and again in every file. Fixtures solve this problem — they're written once, in a separate file, and can be injected into any number of tests across any number of spec files, just by importing them.

Another advantage is that fixtures are composable — I can create multiple custom fixtures, and each test only asks for the ones it actually needs. A spec file can import as many fixtures as required, but if I try to write all that setup and teardown logic directly using hooks inside the spec file itself, the file becomes long and hard to read.

So essentially, fixtures make the code more readable by pulling boilerplate setup/teardown logic out into a separate, reusable file — the spec file stays clean and only contains actual test logic."

---

## 2) What are custom fixtures? Have you created any?

**How to speak in interview:**

"Custom fixtures are fixtures that we define ourselves, to inject our own reusable objects into tests — things Playwright doesn't provide out of the box.

Yes, I've created custom fixtures in my framework. A few examples:

- **Login fixture** — instead of every test manually performing the login steps, I created a fixture that logs in once and hands over an already logged-in `page` to the test.
- **Page Object fixtures** — instead of writing `new LoginPage(page)` in every single test, I inject a ready-made `loginPage` instance directly through the fixture.
- **API client fixture** — I inject a pre-configured request client that already has auth headers set, so tests can make authenticated API calls without repeating that setup."

---

## 3) How do you create a custom fixture? Steps.

**How to speak in interview, step by step:**

"To create a custom fixture, I follow these steps:

**Step 1** — I create a dedicated fixtures folder in my framework, and name the file based on what it does — for example, `auth.fixture.js` for a login-related fixture.

**Step 2** — I import Playwright's `test` but rename it to `base`, like `import { test as base } from '@playwright/test'`. I rename it because I'm going to create my own extended `test`, and I can't reuse the same name — that would cause a naming conflict.

**Step 3** — I call `base.extend({...})` to create a new `test` object. This new `test` has all of Playwright's built-in fixtures like `page` and `context`, plus whatever custom fixture I add inside it.

**Step 4** — Inside `extend`, I define my fixture as a function. It receives Playwright's existing fixtures I depend on — like `{ page }` — and a special `use` function.

**Step 5** — Any code I write *before* calling `use()` is my **setup** — for a login fixture, that's where I navigate to the page and perform the login steps.

**Step 6** — I call `await use(page)` — this is the handover point, where control passes to the test and the test body actually runs.

**Step 7** — Any code I write *after* `use()` is my **teardown** — this runs automatically once the test finishes, whether it passes or fails.

**Step 8** — I also re-export `expect` from the same fixture file, so that spec files only need a single import statement for both `test` and `expect`, instead of importing from two different places.

**Step 9** — Finally, in the spec file, I import `test` and `expect` from my fixture file instead of `@playwright/test`, and I destructure my custom fixture — like `{ loggedInUser }` — directly in the test's parameters, the same way we normally destructure `{ page }`."

## 4) What is the scope of fixtures? Can we change scope?

**How to speak in interview:**

"Fixtures in Playwright have a scope, which decides how often the fixture is created — and there are two scopes: **test** scope and **worker** scope.

**Test scope** is the default scope. A test-scoped fixture is created fresh for every single test, and torn down after that test finishes. This is what gives us **test isolation** — no test's state leaks into another test. All of Playwright's built-in fixtures, like `page` and `context`, are always test-scoped by default, to guarantee isolation — we cannot change the scope of built-in fixtures. We can only change scope for **custom** fixtures.

**Worker scope** means the fixture is created only **once per worker**, and reused across all tests that run in that same worker — instead of being recreated for every test. We can set this by passing `{ scope: 'worker' }` when defining a custom fixture.

**Use case for worker scope** — for example, if I want to log in only once and reuse that same authenticated session across all UI tests in that worker, instead of logging in before every single test, I'd make that login fixture worker-scoped.

**But there's a risk with worker scope** — since the same instance is shared across multiple tests, if that instance gets into a bad state — for example, if one test fails in a way that corrupts the session or logs the user out — then all the other tests sharing that same worker-scoped fixture can start failing too, because they're depending on the same shared instance instead of getting a fresh one.

So the general rule I follow — use test scope by default for anything involving state that could change during a test, and only use worker scope for expensive, mostly read-only or stateless setups where sharing across tests is safe."

```javascript
export const test = base.extend({
	apiClient: [
		async ({}, use) => {
			// SETUP — sirf ek baar is worker ke liye
			const client = await createApiClient();
			console.log("✅ API client created for this worker");

			await use(client);

			// TEARDOWN — worker khatam hone pe
			await client.close();
			console.log("🔴 API client closed");
		},
		{ scope: 'worker' }, // yahan scope specify kiya
	],
});
```
