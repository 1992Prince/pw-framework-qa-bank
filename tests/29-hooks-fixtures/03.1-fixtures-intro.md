
# 📝 Notes: Fixtures in Playwright (Part 1 — Concepts & Inbuilt Fixtures)

## 1) What are Fixtures & their purpose

- Fixtures are Playwright's way of providing **objects/setup that tests need** (like `page`, `browser`, `context`) automatically, without you manually creating/tearing them down.
- Think of fixture as: **"give me a ready-to-use resource, and clean it up automatically when the test finishes."**
- Purpose: eliminate repetitive setup/teardown code, isolate tests from each other (fresh instance per test), and let you inject **your own custom objects** (like a logged-in page, API client, test data) into tests the same clean way Playwright injects `page`.

```js
test("login test", async ({ page }) => {
  // "page" here IS a fixture — Playwright creates a fresh browser page for you automatically
});
```

---

## 2) If we have hooks, why do we need fixtures?

This is the key conceptual difference:

| Hooks                                                        | Fixtures                                                                                   |
| ------------------------------------------------------------ | ------------------------------------------------------------------------------------------ |
| Run**imperative code** before/after tests              | **Provide/inject objects** that tests use                                            |
| Scoped to file/describe block                                | Scoped to test/worker, reusable across files                                               |
| You still access shared state via closures/variables (messy) | Dependency-injected directly into test's parameter list — clean                           |
| Manual teardown logic (you write`afterEach` yourself)      | **Automatic teardown built-in** — code after `use()` runs as cleanup              |
| Can't be composed/reused easily across projects              | Fixtures can depend on other fixtures, and are easily reusable across your whole framework |

**Simplest way to say it:** hooks run code at certain times; fixtures provide **objects with built-in setup + teardown**, and get injected wherever needed — like a mini dependency injection system. Hooks answer "when should this run?" Fixtures answer "what does my test need, and how do I get it ready + clean it up?"

## 3) Inbuilt fixtures in Playwright — and boilerplate they save

Playwright provides several fixtures out of the box: `page`, `context`, `browser`, `browserName`, `request`.

### Example: `page` fixture

**Without fixtures (manual way you'd have to write without Playwright's Test runner):**

```js
const { chromium } = require('playwright');

let browser, context, page;

// setup — you write this manually, every single test file
browser = await chromium.launch();
context = await browser.newContext();
page = await context.newPage();

// your actual test logic
await page.goto("https://example.com");
await page.getByTestId("form-email").fill("test@example.com");

// teardown — you must remember to write this too, every time
await page.close();
await context.close();
await browser.close();
```

That's **~8 lines of setup/teardown boilerplate**, repeated in every single test, and easy to forget cleanup (causing memory leaks / hanging processes in CI).

**With Playwright's `page` fixture:**

```js
test("fill email", async ({ page }) => {
  await page.goto("https://example.com");
  await page.getByTestId("form-email").fill("test@example.com");
});
```

- Setup (`launch → newContext → newPage`) happens automatically before the test
- Teardown (closing page/context) happens automatically after the test — even if the test fails
- Every test gets a **fresh, isolated `page`** — no state leaking between tests

**This is the core value of fixtures: 8+ lines of boilerplate reduced to zero, and it's automatically consistent across your entire framework.**

### Other inbuilt fixtures

**`context`** — useful when you want to open multiple pages/tabs with different URLs and test interactions between them (e.g., testing an invite flow across two tabs).

```js
test("two tabs", async ({ context }) => {
  const page1 = await context.newPage();
  const page2 = await context.newPage();
  await page1.goto("https://example.com/page1");
  await page2.goto("https://example.com/page2");
});
```

**`browser`** — the browser instance itself; needed when you want multiple **independent** browser contexts (e.g., simulating two different logged-in users in one test). Rarely used directly.

```js
test("two independent users", async ({ browser }) => {
  const contextA = await browser.newContext(); // separate cookies/session
  const contextB = await browser.newContext(); // separate cookies/session
  const pageA = await contextA.newPage();
  const pageB = await contextB.newPage();
});
```

**`browserName`** — a string (`"chromium"`, `"firefox"`, `"webkit"`) useful for conditional logic based on which browser the test is currently running in.

```js
test("skip on webkit", async ({ page, browserName }) => {
  test.skip(browserName === "webkit", "Feature not supported on WebKit yet");
  await page.goto("https://example.com");
});
```

**`request`** — for making API calls directly, without going through the UI; isolated per test, useful for setup (creating test data via API) or pure API testing.

```js
test("create user via API", async ({ request }) => {
  const response = await request.post("/api/users", { data: { name: "Sarika" } });
  expect(response.ok()).toBeTruthy();
});
```
